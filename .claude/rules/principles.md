# Принципы разработки

Универсальные принципы для всего кода в проекте (backend + frontend).

## Архитектурные принципы

### Separation of Concerns
Каждый слой/модуль отвечает за одну область знания.
- **Controller/Page** — HTTP/UI ввод-вывод, валидация, маршрутизация
- **Service/Hook** — бизнес-логика
- **Repository/API client** — персистентность или внешние вызовы
Не смешивать слои. Бизнес-логика не знает про БД или HTTP.

### Dependency Rule
Зависимости идут ТОЛЬКО внутрь, к стабильным абстракциям.
```
Controller → Service → Repository
Page → Hook → API client
```
Никогда наоборот. Repository не знает про Service. Service не знает про Controller.

### Boundary Translation
Типы не "протекают" через слои. На границе — маппинг.
- jOOQ `Record` не появляется в Controller
- DTO не появляется в Repository
- API response типы не появляются в UI-компонентах напрямую

## SOLID

- **Single Responsibility** — один класс = одна причина для изменения. Если класс считает репутацию И пишет в БД — разделяй.
- **Open/Closed** — расширяй через добавление нового, не через правку работающего кода.
- **Liskov Substitution** — наследник работает везде где родитель, без сюрпризов.
- **Interface Segregation** — много узких интерфейсов лучше одного "на всё".
- **Dependency Inversion** — зависим от абстракций, не от реализаций. Service зависит от `Repository` интерфейса, не от `JooqRepository`.

## Принципы простоты

### DRY (Don't Repeat Yourself)
Одна логика живёт в одном месте.
**ВАЖНО:** не путать с случайным дублированием. Если два куска кода выглядят похоже, но меняются по разным причинам — это не дубликат, это совпадение. Не склеивать.

### KISS (Keep It Simple, Stupid)
Самое простое решение, которое работает. Абстракция оправдана только когда есть **3+ конкретных случая**. До того — inline.

### YAGNI (You Ain't Gonna Need It)
Не пишем код "на будущее". Добавляем когда реально нужно.
- Интерфейс с одной реализацией "на случай расширения" — нарушение YAGNI
- Параметризация функции "вдруг пригодится" — нарушение YAGNI
- Feature flags "на случай отката" для простых изменений — нарушение YAGNI

## OOP принципы

### Law of Demeter
"Не разговаривай с незнакомцами". Правило: не цепляй больше одной точки.
```kotlin
// ❌ плохо
user.address.city.name

// ✅ хорошо
user.getCityName()
```
Объект знает только своих прямых соседей.

### Tell, Don't Ask
Не спрашивай объект о его состоянии чтобы принять решение — скажи ему что делать.
```kotlin
// ❌ плохо
if (user.isActive()) user.sendNotification()

// ✅ лучше
user.sendNotificationIfActive()
```
Логика живёт рядом с данными.

### Composition over Inheritance
Наследование создаёт жёсткую связь. Композиция (через конструктор) гибче.
Наследуемся ТОЛЬКО когда есть реальная "is-a" связь (`Admin is a User`), не "has-a" и не "uses".

## Иммутабельность

### По умолчанию — immutable
- Kotlin: `val` всегда, `var` только с обоснованием
- TypeScript: `const` всегда, `let` только в циклах/мутации
- Data classes для моделей (оба языка)

### Чистые функции где можно
Функция чистая если:
1. Всегда возвращает одно и то же для одного ввода
2. Не имеет побочных эффектов

Чистые функции легко тестировать, кешировать, переиспользовать.

### Побочные эффекты — на краях
- БД, HTTP, файлы, DOM — в Repository/API-client/useEffect
- Бизнес-логика — чистая: принимает данные → возвращает новые данные
