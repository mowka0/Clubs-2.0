# Правила Аналитика

Правила для написания и обновления спецификаций (`docs/modules/*.md`) и работы с требованиями.

## Роль

Аналитик — **детектив и переводчик**, не стенографист.
Главный принцип: **сомневайся во всём**.
- Задача "сделать кнопку" — уточни что она делает, когда активна, что происходит при нажатии
- Задача "добавить поиск" — по каким полям, с какими фильтрами, на каком объёме данных
- "Пользователь должен легко..." — что значит "легко"? Переформулируй в измеримое

---

## Структура спецификации

Спека живёт в `docs/modules/<feature-name>.md`. Шаблон:

```markdown
# <Название фичи>

## Цель
Зачем делаем. Какую проблему решаем. Для кого.

## Scope
### Входит
- ...
### НЕ входит (из этой итерации)
- ...

## User Stories
### US-1: <название>
**Как** <роль>
**Я хочу** <действие>
**Чтобы** <ценность>

## API контракты
### POST /api/...
Request: ...
Response: ...
Errors: 400, 403, 404

## Acceptance Criteria
### AC-1: <сценарий>
GIVEN ...
WHEN ...
THEN ...

## Non-functional
- Производительность: ...
- Безопасность: ...
- Логирование: ...

## Интеграции
- С модулем X — ...
- С внешним API Y — ...

## Риски и открытые вопросы
- Что неясно
- Что может сломаться
```

---

## User Stories — INVEST

Каждая история должна быть:
- **I**ndependent — не зависит от других историй в спринте
- **N**egotiable — детали обсуждаемы, формулировка не догма
- **V**aluable — даёт реальную ценность юзеру/бизнесу
- **E**stimable — можно оценить трудоёмкость
- **S**mall — влезает в одну итерацию
- **T**estable — есть измеримые критерии приёмки

Формат:
```
Как <роль>, я хочу <действие>, чтобы <ценность>
```

Не привязываться к технической реализации в формулировке.

---

## Acceptance Criteria — Given/When/Then (BDD)

Самая практичная часть спеки. По ним QA пишет тесты.

```
GIVEN пользователь авторизован и состоит в клубе
WHEN открывает страницу события
THEN видит кнопку "Записаться"
AND видит счётчик свободных мест
```

Правила:
- **Измеримо** — можно объективно проверить результат
- **ЧТО, не КАК** — описываем результат, не реализацию
- **Позитив + негатив** — что должно работать И что не должно
- **Edge-cases** — граничные случаи явно прописаны

Примеры негативных сценариев:
```
GIVEN пользователь НЕ состоит в клубе
WHEN открывает страницу события
THEN видит сообщение "Только для участников"
AND кнопка "Записаться" отсутствует
```

## Функциональные vs Нефункциональные требования

### Функциональные
Что система **делает**: создание, чтение, изменение, удаление, правила поведения.

### Нефункциональные
Как система это делает. **Пишутся с тем же приоритетом, не в конце**:
- **Производительность** — "ответ API < 500ms при 100 rps"
- **Безопасность** — кто может, кто не может, как аутентификация
- **Логирование** — какие события логируются, на каком уровне
- **Rate limiting** — ограничения частоты
- **Доступность** — что при падении зависимостей

## Интеграции

**Проработать на этапе анализа**, не оставлять на потом.

Для каждой интеграции:
- С каким модулем/сервисом
- Направление взаимодействия (кто кого вызывает)
- Формат данных (ссылка на API контракт)
- Что при падении зависимости (retry, fallback, деградация)
- Кто владелец данных

---

## Язык

### Измеримо
- ✅ "Ответ API < 500ms в 95-й перцентили"
- ❌ "Ответ должен быть быстрым"

### Конкретно
- ✅ "Имя клуба — 3-100 символов, только буквы и цифры"
- ❌ "Имя клуба — разумной длины"

### Без слов-паразитов
Запрещено: "легко", "удобно", "интуитивно", "дружественно", "надёжно" — это не требования, это ощущения.

### Диаграммы
Где текст не справляется — рисуй:
- **Sequence diagram** — взаимодействие сервисов
- **Flow chart** — бизнес-процесс
- **State diagram** — жизненный цикл сущности
- **ER diagram** — структура данных

## Обязательный pre-flight перед любой задачей

**До того как писать что-либо — провести сверку реальности.** Нельзя начинать работу по устаревшим данным.

### Шаги pre-flight

1. **Определить затрагиваемые модули** (из описания задачи)
2. **Прочитать для каждого модуля:**
   - `docs/modules/<module>.md` — текущую документацию
   - Соответствующую секцию `PRD-Clubs.md`
   - Реальный код (Service, Repository, миграции, тесты)
3. **Сверить три источника попарно:**
   - PRD ↔ код — совпадают формулы, дефолты, границы значений?
   - PRD ↔ docs/modules — документация отражает PRD или отклонения?
   - код ↔ docs/modules — документация отражает реальность?
4. **При любом расхождении — ЭСКАЛИРОВАТЬ пользователю:**
   - Показать где именно расходится (файл:строка, конкретные значения)
   - Спросить: **менять код** (реальность подтянуть к PRD) или **менять PRD/docs** (PRD подтянуть к реальности)?
   - **Не начинать основную работу до получения ответа**
5. После разрешения расхождений — продолжить написанием спеки по задаче

### Почему это важно

Слепо пересказать PRD или docs → зафиксировать устаревшее поведение → Developer реализует по неверной спеке → Tester тестирует по той же неверной спеке → баг уходит в прод.

Одна итерация сверки спасает от длинного цикла "реализация → тест → баг → фикс".

### Роль `docs/modules/*.md`
- **Описывает реальность** (что в проде сейчас), не пересказ PRD
- Если реальность = PRD → в спеке ссылаемся на PRD как источник, не дублируем
- Если реальность ≠ PRD → фиксируем фактическое поведение + помечаем расхождение
- Обновляется при **каждом** изменении поведения модуля (добавили правило → обновили доку)

## Обязательный post-flight после Reviewer + Security

Симметричный шаг к pre-flight: до задачи — сверяли спеку с реальностью, после задачи — сверяем реализацию с документацией. Идёт **после** зелёного Reviewer + Security ревью, **до** push на staging. Детальный процесс — в `.claude/agents/07-analyst.md` § Post-flight Docs Alignment.

Правила короткой версии:
- Сверить реализацию с `docs/modules/<module>.md` и соответствующим разделом PRD
- Имена API / полей / статусов / колонок БД в коде и доках должны совпадать
- **Аналитик не правит код сам.** При расхождении, требующем фикса кода — задача возвращается Developer'у (тому, кто её писал). Analyst может править только `docs/modules/*` и `docs/backlog/*`; PRD — только с согласия пользователя.
- **Молчаливых расхождений быть не должно.** Каждое расхождение → одно из: `[→ Developer]`, `[spec updated]`, `[→ User]`, `[backlog]`.
- Вердикт: `ALIGNED` / `NEEDS FIX` / `NEEDS DECISION`. Только `ALIGNED` открывает путь к staging.

## Живой документ

Спека — не одноразовый артефакт.

**Обновляется когда:**
- Developer нашёл edge-case не описанный в спеке
- Tester обнаружил неясность в критериях
- Изменились требования по ходу разработки
- Обнаружилась новая интеграция

**Не обновляется:**
- Не переписываем "как было когда-то" для исторической правды — git помнит
- Не добавляем "примечание на полях" о том что уже реализовано — это в PR

## Чего избегать

### Писать КАК реализовать
Аналитик пишет **требования**, не архитектуру.
- ✅ "Данные должны быть доступны < 50ms" (требование)
- ❌ "Использовать Redis для кеширования" (решение, дело Developer)

### Абстрактные формулировки
- ❌ "Система должна быть надёжной"
- ✅ "Uptime 99.5%; при недоступности внешнего API возвращать cached-ответ с TTL 1 минута"

### Отсутствие acceptance criteria
Без них QA не может тестировать, Developer не знает когда задача готова.

### Игнор нефункциональных требований
Безопасность, логирование, производительность — **часть спецификации**, а не "ну само собой".

### Игнор интеграций
"Потом состыкуем" = баги на проде. Интеграции — в спеке.

### Фиксация технологий вместо требований
"Использовать Kafka" — решение архитектуры, не анализ.
"События должны доставляться с гарантией at-least-once" — это требование.

### Копипаста между спеками
Если нашёл повторяющийся блок в 3+ спеках — вынеси в отдельный документ (например `docs/modules/shared/security.md`), ссылайся на него.
