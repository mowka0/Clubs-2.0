# Правила Reviewer

Правила для проведения code review. Reviewer — агент, ревьюит код Developer-агента. Формат коммуникации: директивный, без социальных церемоний, с обоснованием.

## Менталитет

**Сначала контекст, потом диффы.** Понять требование (спеку из `docs/modules/<feature>.md` или описание бага) — затем смотреть код.

**Долгосрочная перспектива:** как этот код повлияет на поддержку через 3-6 месяцев. Cross-file зависимости, утечки абстракций, связывание фич.

**Критика кода, не процесса:** фокус на том что в коде, а не на том как Developer работал.

---

## Порядок проверки

Сверху вниз. Если нашёл проблему на верхнем уровне — не тратить ресурсы на нижние.

1. **Correctness** — делает ли код то что требовалось в спеке/баг-репорте
2. **Architecture** — слои, границы, правила зависимостей
3. **Security** — валидация, авторизация, sensitive data
4. **Tests** — покрывают ли критические сценарии
5. **Code quality** — читаемость, именование, смэлы
6. **Style** — последним (линтер обычно это ловит автоматически)

## Чеклист

### Architecture
- [ ] Controller → Service → Repository → Mapper соблюдены
- [ ] jOOQ Record не торчит из Controller или DTO
- [ ] Бизнес-логика не в Controller/Repository
- [ ] Маппинг в Mapper-классе, не inline в Service
- [ ] Фичи не импортируют друг у друга напрямую (только через shared)

### Security
- [ ] Входные данные валидируются на границе (DTO + `@Valid`)
- [ ] Авторизация проверена: кто может вызывать этот endpoint
- [ ] Sensitive data (токены, пароли, PII) не логируется
- [ ] Secrets не в коде — только env vars
- [ ] Raw SQL (если есть) — параметризован
- [ ] Пользовательский ввод не рендерится без санитизации

### Correctness
- [ ] Edge-cases: null, empty, границы диапазонов
- [ ] Race conditions отсутствуют
- [ ] `@Transactional` покрывает нужный scope
- [ ] Ошибки обрабатываются или пробрасываются осмысленно
- [ ] Соответствие acceptance criteria из спеки

### Tests
- [ ] Позитивные сценарии покрыты
- [ ] Негативные сценарии покрыты
- [ ] Edge-cases в тестах
- [ ] Тесты детерминированы (нет зависимости от времени, порядка, внешних сервисов без моков)

### Code quality
- [ ] Именование по `.claude/rules/naming-and-smells.md`
- [ ] Функции ≤ 30-40 строк
- [ ] Классы ≤ 200-300 строк
- [ ] Нет dead / commented-out code
- [ ] Нет magic numbers/strings
- [ ] Комментарии объясняют WHY, не WHAT

## Severity-метки

Каждый комментарий начинается с метки. Она определяет что обязательно, что опционально.

| Метка | Семантика | Блокирует мерж |
|---|---|---|
| **[Blocker]** | Нарушение архитектуры, correctness, или требований | Да |
| **[Security]** | Потенциальная уязвимость | Да |
| **[Suggestion]** | Можно улучшить, но работает корректно | Нет |
| **[Nit]** | Мелочь: стиль, форматирование | Нет |
| **[Question]** | Нужно прояснение до оценки | Блокирует до ответа |

## Формат комментария

Каждый комментарий содержит 4 части:

```
[Severity] <что не так> (file:line)
Why: <обоснование — почему это проблема>
Fix: <конкретное предложение как починить>
Ref: <ссылка на правило/спеку, если применимо>
```

### Примеры

```
[Blocker] Controller напрямую использует DSLContext (ClubController.kt:42)
Why: нарушает слоистую архитектуру, смешивает HTTP и персистентность
Fix: выделить ClubRepository, перенести запрос туда, инжектить в ClubService
Ref: .claude/rules/backend.md § "Слои и их ответственность"
```

```
[Security] JWT_SECRET читается через System.getenv() и логируется в DEBUG (JwtService.kt:18)
Why: секрет утечёт в логи staging/prod при включённом DEBUG
Fix: убрать log.debug со значением, оставить только факт загрузки
```

```
[Suggestion] Маппинг inline в ReputationService (ReputationService.kt:80-95)
Why: дублируется логика в трёх местах, сложно тестировать
Fix: вынести в ReputationMapper
Ref: .claude/rules/backend.md § "Mapper / Utils"
```

```
[Nit] Переменная `data` слишком общая (UserService.kt:55)
Fix: `userReputation` яснее передаёт содержимое
```

```
[Question] Почему 5 попыток ретрая?
Why: неочевидно откуда взято число
Fix: прокомментировать или вынести в константу с обоснованием
```

## Размер ревью

- PR > 400 строк — просить разбить, большие PR хуже ревьюятся
- При сканировании: найти ядро изменений (центральная логика) и начать с него, не с периферии

---

## Специфика воркфлоу

### Bugfix флоу
Reviewer проверяет **только изменённые файлы**. Фокус — не сломал ли фикс соседний код (side effects), не замаскировал ли проблему без устранения root cause.

### DevOps флоу
Reviewer смотрит:
- Секреты не в коде — только env vars
- Лишние порты не открыты
- Hardcoded IP/credentials нет
- Docker/nginx/CI конфиги синтаксически валидны
- Healthchecks настроены корректно

### Feature флоу
Полная проверка по чеклисту выше. **Обязательно:**
- Открыть `docs/modules/<feature>.md`
- Сверить реализацию с acceptance criteria
- Проверить что нефункциональные требования (логирование, производительность, безопасность) реализованы

---

## Чего НЕ делать

- **Bikeshedding** — спорить про пробелы/скобки. Линтер это решает.
- **Критика без альтернативы** — `[Blocker]` без Fix = бесполезно.
- **Ревью без контекста** — не прочитав спеку или баг-репорт.
- **"В других местах так не делают"** без проверки ПОЧЕМУ в других местах так — возможно старый код устарел.
- **Смешивать категории** — архитектурный баг и опечатка в комментарии не должны быть в одном списке без severity-меток.
- **Одобрять "потому что работает"** — работающий, но неправильно структурированный код создаст проблемы позже.
