# Репутация — ручной тест-план (staging)

Пошаговый план для ручной проверки блока репутации (PRD §4.4.4) вместе с **Блоком 1 живого пути**
(EXP-2 / ATT-2 / ATT-3 + UI оспаривания, ветка `feature/two-stage-confirmation-gaps`). Это один из
самых важных модулей — проверяем **все** исходы таблицы начисления + краевые случаи + новый
поток оспаривания.

> **Контракт начислений** — `docs/modules/reputation-v2.md`. **Поведение Блока 1** —
> `docs/modules/events.md` § «Репутация — Блок 1». **Реестр багов** —
> `docs/backlog/two-stage-reputation-bug-register.md` (читать ПЕРЕД тестом).

---

## 0. Перед началом — обязательно прочитать

Репутация считается **только** после полной цепочки:
```
событие создано → голос «Пойду/Возможно» → ПОДТВЕРЖДЕНИЕ (Этап 2)
→ событие прошло → организатор ОТМЕТИЛ явку → окно оспаривания истекло → finalize → репутация
```

### Что важно знать про текущее поведение (что починили, что ещё нет)

- **S2T-2 (ещё НЕ починено):** DM «Подтверди участие» при старте Этапа 2 **не отправляется**.
  Поэтому подтверждать нужно **вручную** на странице события (кнопка «Подтвердить участие»),
  пока событие не началось. Это не ваша ошибка тестирования — просто рассылка пока не подключена.
- **EXP-2 (ПОЧИНЕНО в Блоке 1):** раньше, если организатор не отмечал явку, событие **никогда** не
  финализировалось и репутация не считалась вообще. Теперь два варианта:
  - организатор **отметил** явку → через окно оспаривания событие финализируется и репутация
    начисляется (обычный путь);
  - организатор **не отметил** до дедлайна (`ATTENDANCE_AUTO_FINALIZE_UNMARKED_MINUTES`, дефолт 48ч) →
    событие **закрывается нейтрально**: всем confirmed-участникам **0** (ни +100, ни −50), ledger не
    пишется. Никого не наказывают и не награждают за бездействие организатора.
- **ATT-2 (ПОЧИНЕНО в Блоке 1):** раньше нерешённый спор обнулял штраф (давал 0 вместо −50). Теперь,
  если организатор **не отреагировал** на спор до финализации, действует исходная отметка «не пришёл»
  → начисляется штраф (**−50** для «Пойду», **−20** для «Возможно»).
- **ATT-3 (ПОЧИНЕНО в Блоке 1):** появился полноценный поток оспаривания. После отметки явки
  отсутствующим участникам приходит DM «оспорьте», и в приложении на странице события у них есть
  кнопка **«Оспорить»**. Организатор видит блок **«Оспоренные отметки»** и решает «Пришёл / Не пришёл».

### 0.1 Конфиг staging (ускорить цикл) — выставить в Coolify (staging app) env:
- `ATTENDANCE_DISPUTE_WINDOW_MINUTES=5` — окно оспаривания 5 мин вместо 48ч.
- `ATTENDANCE_FINALIZE_POLL_MS=60000` — финализатор раз в минуту вместо часа.
- `STAGE2_EXPIRE_POLL_MS=60000` — авто-истечение брони раз в минуту.
- `ATTENDANCE_AUTO_FINALIZE_UNMARKED_MINUTES` — **оставить дефолт (2880)** для всех сценариев, КРОМЕ
  специального теста EXP-2 (сценарий E9). Если выставить его маленьким (например 5) глобально, то любое
  событие, где вы не успели отметить явку за 5 минут, закроется нейтрально, и отметить его будет уже
  нельзя — это помешает позитивным/негативным сценариям. Поэтому ставьте `=5` **только** на время теста E9,
  потом верните дефолт.
- после всего теста — **удалить** добавленные переменные (prod-дефолты: 48ч / 1ч / 5мин / 48ч).

### 0.2 Аккаунты
- **Орг** — владелец клуба (NB: в своём клубе репутация НЕ копится — анти-фарм правило 1).
- **U1, U2, U3** — обычные участники (активная подписка). Репутацию проверяем на них.
- Минимум 3 исхода на участника, иначе UI покажет **«Новичок»** (порог `MIN_OUTCOMES_FOR_DISPLAY=3`),
  даже если индекс уже посчитан. Проверять число — после ≥3 финализированных исходов.

### 0.3 Где смотреть результат
- Профиль участника (таб «Участники» клуба → тап по юзеру → `MemberProfileModal`): индекс
  надёжности, выполнение обещаний, подтверждений участия, посещений событий, **«Спонтанные визиты»**
  (счётчик спонтанности).
- Глобальный таб «Профиль» → «Моя репутация» (карточка на клуб): надёжность + строка
  «обещания % · N подтв. · M посещ. · K спонт.» (спонтанность показывается только при K>0).

## 1. Цикл одного события (базовый сценарий)
1. Орг создаёт событие с `event_datetime` через **~10 минут**, `participantLimit` ≥ числа тестеров.
2. Дождаться, пока сработает триггер Этапа 2 (≤5 мин) — статус события `stage_2`.
   (DM не придёт — это S2T-2. Просто открыть событие в приложении.)
3. Каждый тестер на странице события жмёт нужный голос Этапа 1, затем **«Подтвердить участие»**
   (для сценариев, где нужен `confirmed`).
4. Дождаться `event_datetime` (событие началось) → кнопки подтверждения исчезают (Bug B).
5. Орг открывает прошедшее событие → «Отметить посещаемость» → проставляет Пришёл/Не пришёл
   **только по confirmed-участникам** (forgot-to-confirm в списке НЕ показываются) → «Сохранить».
6. Подождать `ATTENDANCE_DISPUTE_WINDOW_MINUTES` (5 мин) + 1 цикл финализатора (1 мин).
7. Проверить репутацию участников.

## 2. Позитивные сценарии (таблица §4.4.4)

| # | Поведение | Этап 1 | Этап 2 | Явка | Ожидаемо |
|---|---|---|---|---|---|
| P1 | **Железобетонный** | Пойду | Подтвердил | Пришёл | надёжность **+100**, обещания +1/+1 (100%), посещение +1 |
| P2 | **Спонтанный** | Возможно | Подтвердил | Пришёл | надёжность **+30**, спонтанность **+1**, посещение +1 |

**P1:** U1 голосует «Пойду» → «Подтвердить» → орг отмечает «Пришёл» → finalize → **+100**.
**P2:** U2 голосует «Возможно» → «Подтвердить» → «Пришёл» → **+30** и счётчик спонтанности +1.

## 3. Негативные сценарии

| # | Поведение | Этап 1 | Этап 2 | Явка | Ожидаемо |
|---|---|---|---|---|---|
| N1 | **Пустозвон** | Пойду | Подтвердил | **Не пришёл** | надёжность **−50**, обещание не выполнено (0%) |
| N2 | **Зритель** | Возможно | Подтвердил | **Не пришёл** | надёжность **−20** |
| N3 | **Передумавший** | Пойду | **Отказался** | — | **0** (не в ledger) |
| N4 | **Вечный сомневающийся** | Возможно | не подтвердил | — | **0** (→ `expired_no_confirm`) |
| N5 | **Молчун** | Не пойду | — | — | **0** |

**N1:** U1 «Пойду» → «Подтвердить» → орг «Не пришёл» → finalize → **−50**.
**N2:** U2 «Возможно» → «Подтвердить» → «Не пришёл» → **−20**.
**N3:** U1 «Пойду» → «Подтвердить» → затем «Отказаться» (до старта) → 0, место уходит листу ожидания.
**N4:** U2 «Возможно», НЕ жмёт подтвердить → после старта статус `expired_no_confirm`, репутация 0,
   в списке отметки явки НЕ появляется.
**N5:** U3 «Не пойду» → 0, в Этапе 2 не участвует.

## 4. Краевые сценарии

- **E1 (порог «Новичок»):** у нового участника 1–2 финализированных исхода → профиль показывает
  **«Новичок»** (не число), хотя индекс посчитан. После 3-го исхода — показывается число.
- **E2 (анти-фарм владельца):** орг сам участвует+подтверждает+приходит на СВОём событии → ledger-строки
  **нет**, репутация в своём клубе не растёт.

### E3 — оспаривание явки (ПОЛНЫЙ поток, ATT-2 + ATT-3)

Предусловия: U1 «Пойду» → «Подтвердить» → событие прошло → орг **отметил** U1 как «Не пришёл» →
«Сохранить». Окно оспаривания открыто (event уже `attendance_marked`, ещё не finalized).

- **E3.0 (DM участнику):** сразу после сохранения отметки U1 должен получить DM «Организатор отметил
  присутствие… вас отметили как отсутствующего… оспорьте на странице события» (ATT-3) с
  inline-кнопкой **«Оспорить явку»**, которая открывает страницу события.
- **E3.1 (участник видит «Оспорить»):** U1 открывает страницу события → видит блок **«Ваша явка»** с
  текстом «Организатор отметил вас как отсутствующего…», полем **«Комментарий организатору
  (необязательно)»** и кнопкой **«Оспорить»**.
- **E3.2 (участник оспаривает):** U1 (опционально вписывает комментарий) жмёт «Оспорить» → тост
  «Отметка оспорена», блок меняется на «Вы оспорили… ждёт решения организатора». (Статус явки = `disputed`.)
- **E3.3 (организатор видит спор):** орг открывает событие → под «✓ Посещаемость отмечена» виден блок
  **«Оспоренные отметки»** с именем U1, **его комментарием** (если оставил) и кнопками
  **«Пришёл» / «Не пришёл»**.
- **E3.4 (орг исправляет → +100):** орг жмёт **«Пришёл»** → дождаться finalize → у U1 **+100** (как P1).
- **E3.5 (орг НЕ реагирует → −50, ATT-2):** повторить E3 на другом событии, но орг спор **не трогает** →
  дождаться истечения окна (5 мин) + finalize → у U1 **−50** (не 0!). Это и есть фикс ATT-2: нерешённый
  спор = действует исходная отметка «не пришёл».
- **E3.6 (окно закрылось):** после finalize кнопки «Оспорить» (у участника) и «Пришёл/Не пришёл»
  (у организатора) **исчезают** — оспаривать уже нечего.

### E4–E8 (без изменений)
- **E4 (лист ожидания / FIFO):** `participantLimit=1`. U1 и U2 «Пойду». U1 подтверждает (1/1 confirmed),
  U2 подтверждает → **waitlisted**. U1 «Отказаться» → U2 **авто-промоутится** в confirmed (FIFO).
- **E5 (идемпотентность):** повторно нажать «Подтвердить» — статус не дублируется, счётчики не растут.
- **E6 (двойная финализация):** дождаться 2 циклов финализатора — индекс **не удваивается**
  (ledger append-only, `claimEvent` одноразовый).
- **E7 (overflow → waitlist):** `participantLimit=2`, 4×«Пойду». При триггере первые 2 (по времени
  голоса) могут подтвердить, 3–4 → `waitlisted`. ⚠️ NB баг **S2-04**: maybe-голосующий может занять
  слот раньше going в overflow — если тестируете смешанный пул, проверьте порядок.
- **E8 (микс осей):** участник получил +100 за событие И участвовал в складчине (±) → итоговый
  индекс = сумма обеих осей; счётчики посещений считают только событийную ось.

### E9 — нейтральная авто-финализация (EXP-2)

Проверяет, что событие без отметки явки закрывается нейтрально (0 всем), а не висит вечно.

1. **Только для этого теста** выставить `ATTENDANCE_AUTO_FINALIZE_UNMARKED_MINUTES=5` (см. § 0.1).
2. Орг создаёт событие через ~10 мин. U1 «Пойду» → «Подтвердить». Дождаться старта события.
3. Организатор **намеренно НЕ открывает отметку явки**.
4. Подождать 5 мин (дедлайн) + 1 цикл финализатора (1 мин).
5. **Ожидаемо:**
   - у U1 репутация за это событие **не изменилась** (нет +100 и нет −50; ledger-строки нет);
   - организатор на странице события видит «**Окно отметки явки истекло. Событие закрыто без отметки —
     репутация участникам за него не начислена.**», блока «Отметить посещаемость» **нет**;
   - попытка отметить явку (если успеть зайти) отклоняется — окно закрыто.
6. После теста вернуть `ATTENDANCE_AUTO_FINALIZE_UNMARKED_MINUTES` к дефолту (2880).

## 5. Что проверять в каждом сценарии
1. **Индекс надёжности** = Σ всех начислений (может быть отрицательным).
2. **% выполнения обещаний** = приходы-после-подтверждения / подтверждения × 100.
3. **Счётчик спонтанности** = число «Возможно→Подтвердил→Пришёл».
4. **outcome_count ≥ 3** прежде чем доверять отображаемому числу (иначе «Новичок»).
5. Репутация **per-club** (один юзер — разные числа в разных клубах).
