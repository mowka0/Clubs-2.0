# Репутация — ручной тест-план (staging)

Пошаговый план для ручной проверки блока репутации (PRD §4.4.4). Один из самых важных
модулей — проверяем **все** исходы таблицы начисления + краевые случаи.

> **Контракт начислений** — `docs/modules/reputation-v2.md`. **Известные баги, влияющие на
> тест** — `docs/backlog/two-stage-reputation-bug-register.md` (читать ПЕРЕД тестом).

---

## 0. Перед началом — обязательно прочитать

Репутация считается **только** после полной цепочки:
```
событие создано → голос «Пойду/Возможно» → ПОДТВЕРЖДЕНИЕ (Этап 2)
→ событие прошло → организатор ОТМЕТИЛ явку → окно оспаривания истекло → finalize → репутация
```
Два известных бага делают «наивный» тест бесполезным (это НЕ ваша ошибка тестирования):
- **S2T-2:** DM «Подтверди участие» при старте Этапа 2 **не отправляется**. Подтверждать нужно
  **вручную** на странице события (кнопка «Подтвердить участие»), пока оно не началось.
- **EXP-2:** если организатор **не отметил явку** — finalize/репутация **никогда** не запустятся.
  Отметка явки обязательна. Это объясняет «5 событий подтвердил — всё ещё Новичок».

### 0.1 Конфиг staging (ускорить цикл) — выставить в Coolify (staging app) env:
- `ATTENDANCE_DISPUTE_WINDOW_MINUTES=5` (окно оспаривания 5 мин вместо 48ч)
- `ATTENDANCE_FINALIZE_POLL_MS=60000` (финализатор раз в минуту вместо часа)
- `STAGE2_EXPIRE_POLL_MS=60000` (авто-истечение раз в минуту)
- после теста — **удалить** эти переменные (prod-дефолты: 48ч / 1ч / 5мин).

### 0.2 Аккаунты
- **Орг** — владелец клуба (NB: в своём клубе репутация НЕ копится — анти-фарм правило 1).
- **U1, U2, U3** — обычные участники (активная подписка). Репутацию проверяем на них.
- Минимум 3 исхода на участника, иначе UI покажет **«Новичок»** (порог `MIN_OUTCOMES_FOR_DISPLAY=3`),
  даже если индекс уже посчитан. Проверять число — после ≥3 финализированных исходов.

### 0.3 Где смотреть результат
- Профиль участника (таб «Участники» клуба → тап по юзеру → `MemberProfileModal`): индекс
  надёжности, % обещаний, посещения, спонтанность.
- Глобальный таб «Профиль» → «Моя репутация» (карточка на клуб).

## 1. Цикл одного события (базовый сценарий)
1. Орг создаёт событие с `event_datetime` через **~10 минут**, `participantLimit` ≥ числа тестеров.
2. Дождаться, пока сработает триггер Этапа 2 (≤5 мин) — статус события `stage_2`.
   (DM не придёт — это S2T-2. Просто открыть событие в приложении.)
3. Каждый тестер на странице события жмёт нужный голос Этапа 1, затем **«Подтвердить участие»**
   (для сценариев, где нужен `confirmed`).
4. Дождаться `event_datetime` (событие началось) → кнопки подтверждения исчезают (Bug B).
5. Орг открывает прошедшее событие → «Отметить посещаемость» → проставляет Пришёл/Не пришёл
   **только по confirmed-участникам** (forgot-to-confirm в списке НЕ показываются) → «Сохранить».
6. Подождать `ATTENDANCE_DISPUTE_WINDOW_MINUTES` (5 мин) + 1 цикл финализатора (1 мин).
7. Проверить репутацию участников.

## 2. Позитивные сценарии (таблица §4.4.4)

| # | Поведение | Этап 1 | Этап 2 | Явка | Ожидаемо |
|---|---|---|---|---|---|
| P1 | **Железобетонный** | Пойду | Подтвердил | Пришёл | надёжность **+100**, обещания +1/+1 (100%), посещение +1 |
| P2 | **Спонтанный** | Возможно | Подтвердил | Пришёл | надёжность **+30**, спонтанность **+1**, посещение +1 |

**P1:** U1 голосует «Пойду» → «Подтвердить» → орг отмечает «Пришёл» → finalize → **+100**.
**P2:** U2 голосует «Возможно» → «Подтвердить» → «Пришёл» → **+30** и счётчик спонтанности +1.

## 3. Негативные сценарии

| # | Поведение | Этап 1 | Этап 2 | Явка | Ожидаемо |
|---|---|---|---|---|---|
| N1 | **Пустозвон** | Пойду | Подтвердил | **Не пришёл** | надёжность **−50**, обещание не выполнено (0%) |
| N2 | **Зритель** | Возможно | Подтвердил | **Не пришёл** | надёжность **−20** |
| N3 | **Передумавший** | Пойду | **Отказался** | — | **0** (не в ledger) |
| N4 | **Вечный сомневающийся** | Возможно | не подтвердил | — | **0** (→ `expired_no_confirm`) |
| N5 | **Молчун** | Не пойду | — | — | **0** |

**N1:** U1 «Пойду» → «Подтвердить» → орг «Не пришёл» → finalize → **−50**.
**N2:** U2 «Возможно» → «Подтвердить» → «Не пришёл» → **−20**.
**N3:** U1 «Пойду» → «Подтвердить» → затем «Отказаться» (до старта) → 0, место уходит листу ожидания.
**N4:** U2 «Возможно», НЕ жмёт подтвердить → после старта статус `expired_no_confirm`, репутация 0,
   в списке отметки явки НЕ появляется.
**N5:** U3 «Не пойду» → 0, в Этапе 2 не участвует.

## 4. Краевые сценарии

- **E1 (порог «Новичок»):** у нового участника 1–2 финализированных исхода → профиль показывает
  **«Новичок»** (не число), хотя индекс посчитан. После 3-го исхода — показывается число.
- **E2 (анти-фарм владельца):** орг сам участвует+подтверждает+приходит на СВОём событии → ledger-строки
  **нет**, репутация в своём клубе не растёт.
- **E3 (оспаривание):** орг отметил U1 «Не пришёл» → U1 жмёт «Оспорить» (статус `disputed`).
  - орг **исправляет** на «Пришёл» (в окне) → finalize → **+100**.
  - ⚠️ орг **НЕ реагирует** → finalize → **0** (НЕ −50). Это баг **ATT-2** — фиксируем как факт:
    нерешённый спор сейчас обнуляет штраф (должно быть −50). Записать в баг-репорт, если воспроизвели.
- **E4 (лист ожидания / FIFO):** `participantLimit=1`. U1 и U2 «Пойду». U1 подтверждает (1/1 confirmed),
  U2 подтверждает → **waitlisted**. U1 «Отказаться» → U2 **авто-промоутится** в confirmed (FIFO).
- **E5 (идемпотентность):** повторно нажать «Подтвердить» — статус не дублируется, счётчики не растут.
- **E6 (двойная финализация):** дождаться 2 циклов финализатора — индекс **не удваивается**
  (ledger append-only, `claimEvent` одноразовый).
- **E7 (overflow → waitlist):** `participantLimit=2`, 4×«Пойду». При триггере первые 2 (по времени
  голоса) могут подтвердить, 3–4 → `waitlisted`. ⚠️ NB баг **S2-04**: maybe-голосующий может занять
  слот раньше going в overflow — если тестируете смешанный пул, проверьте порядок.
- **E8 (микс осей):** участник получил +100 за событие И участвовал в складчине (±) → итоговый
  индекс = сумма обеих осей; счётчики посещений считают только событийную ось.

---

## 5. Что проверять в каждом сценарии
1. **Индекс надёжности** = Σ всех начислений (может быть отрицательным).
2. **% выполнения обещаний** = приходы-после-подтверждения / подтверждения × 100.
3. **Счётчик спонтанности** = число «Возможно→Подтвердил→Пришёл».
4. **outcome_count ≥ 3** прежде чем доверять отображаемому числу (иначе «Новичок»).
5. Репутация **per-club** (один юзер — разные числа в разных клубах).
