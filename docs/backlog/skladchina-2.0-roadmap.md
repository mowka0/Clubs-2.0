## Главный принцип

Деньги ходят мимо приложения (СБП/наличка оргу) → приложение **не платёжка, а трекер**:
«просьба + чеклист + напоминалка». Каждая попытка притворяться платёжкой (точные суммы,
автопересчёты, эскроу-валидации) рождает баги и трение (доказано staging-багом округления
2026-06-12).

**Шаблоны = маски поверх ОДНОГО движка**, не отдельные фичи. Под капотом — одна таблица
`skladchinas` + колонка `template` (`custom | split_bill | birthday | booking | gear`).
Шаблон определяет четыре вещи: **триггер** (что подсказало создать) · **пресет**
(предзаполненные поля) · **видимость** (кто видит) · **дефолт важности**.
Смысловая формула: *приложение замечает момент клуба → орг нажимает один раз → клуб теплее.*

## Фаза A — упрощение ядра (фундамент шаблонов)

1. **Один тап «Я оплатил {N} ₽»** в fixed-режимах — поле ввода суммы убрать
   (после фикса a2fb347 сервер всё равно записывает назначенную долю, поле декоративно).
2. **Орг отмечает оплату сам** («получил наличкой») — кнопка напротив участника.
   Орг — единственный, кто видит деньги; это его право поправить чеклист, не бюрократия.
3. **«Не хватает X ₽» вместо автопересчёта**: при отказе участника доли НЕ пересчитываются
   автоматически. Орг видит разрыв и кнопки: «Перераспределить на неоплативших»
   (доля = (цель − собрано) ÷ неоплатившие; **уже оплативших не трогаем никогда**) /
   «Оставить как есть». Математика наша, решение человеческое.
4. **Убрать автозакрытие по достижению цели** (источник сложности F5-02). Закрытие:
   дедлайн / вручную / все ответили. Цель и прогресс-бар в деньгах — декорация.
5. **Прогресс в людях** («скинулись 5 из 8») как главная метрика — деньги самодекларированы,
   люди честнее.

## Фаза B-1 — карточка сбора в чате клуба

У клубов есть привязка чата (`club_chat_links`, V47 — раньше `telegram_group_id`); **бот умеет DM и групповые сообщения чат-интеграции** (в NotificationService
нет group-методов) — нужна инфраструктура.

- Бот постит в группу одну карточку «🎾 {title} · скинулись 5/8 · до {дата} · [Открыть сбор]»
  (deep-link `startapp=skladchina_<id>`, паттерн PR #53) и **редактирует её же** при изменениях.
- **Решение PO-уровня (рекомендация):** поимённо показываем только ОПЛАТИВШИХ (позитив);
  молчунов в группе не палим — их пингует личный DM, не публичный позор.
- Шаблон «Подарок» — публикация в чат **выключена принудительно** (сюрприз).
- Fallback: нет группы → текущие DM.

PR-1: `sendGroupMessage`/`editGroupMessage` + обработка «бот удалён/нет прав» (403 → лог,
сброс message_id) + миграция `skladchinas.group_message_id BIGINT NULL` + пост при создании,
финальный edit при закрытии. PR-2: live-прогресс — TG лимитирует edit (~20 msg/min/группу),
поэтому флаг `card_dirty` при markPaid/decline + шедулер раз в ~20с редактирует грязные
карточки (дебаунс, переживает рестарты; паттерн `reminder_sent_at`).
Риск: onboarding привязки группы / права бота — проверить перед стартом.

## Фаза B-2 — «Разделить счёт» из события

Уникальное преимущество: **мы знаем, кто пришёл** (attendance). Бонус-выравнивание стимулов:
сплит работает только по отмеченной явке → орг получает шкурную причину отмечать явку →
лечит корень EXP-2-поведения.

- Кнопка «🧾 Разделить счёт» на прошедшем событии (орг, `attendance_marked`, attended ≥ 2,
  не старше 30 дней). Явка не отмечена → кнопка ведёт в отметку явки.
- Поле «Сумма чека» → предпросмотр «1 200 ₽ ÷ 4 = по 300 ₽», список attended с аватарками,
  исключение тапом → создать. Округление **вверх** до рубля с подписью «+N ₽ округление».
- Дефолты: fixed_equal, дедлайн +48ч, **важность выкл** (счёт не горит; вкл — работают гейты).
- Архитектура: вариант Б — nullable `skladchinas.event_id` FK (переиспользуется фазой D),
  валидация на бэке (событие клуба, прошло, явка отмечена), обратный блок на EventPage
  «Счёт: оплатили 2/4». Вход через `CreateActivityFlow` в режиме «из события»
  (пропуск шагов тип/клуб). Один PR.

## Фаза C — сбор на день рождения

- Миграция `users.birthday DATE NULL` + опциональный шаг онбординга. **Год не публикуем
  никогда** — только день/месяц. Поле строго opt-in.
- Шедулер: за ~10 дней до ДР участника → DM оргу «У {имя} скоро ДР — собрать на подарок?»
  с кнопкой → пресет «Подарок»: voluntary (вне репутации по locked-дизайну), **именинник
  не в участниках** → сбор ему не виден (feed показывает только свои сборы — уже так),
  карточка в чат не постится (см. B-1).

## Фаза D — сбор-к-событию (бронь корта) — самая глубокая

Продуктовый козырь: **платные события без эквайринга** (Stars запрещены для физических
услуг, ЮKassa требует юрлицо; у событий поля цены нет вообще — это единственный путь).

- **Оплата живёт на этапе-2, не на этапе-1**: голос «иду» остаётся лёгким сигналом;
  подтверждение участия открывает сбор и DM «скинься на корт». Орг видит две колонки:
  подтвердил / оплатил.
- **Жёсткий гейт «confirm требует оплаты» — только v2**, после данных по конверсии
  подтверждений (трение в confirm — самое дорогое место воронки).
- **Возвраты** (событие отменилось / отказ после оплаты): деньги офлайн — возврат
  ответственность орга, приложение показывает «кому вернуть». Честно писать в UI.
- Архитектурно: динамический состав сбора (участник добавляется при confirm) — сейчас
  состав фиксируется при создании; это самое глубокое изменение, поэтому фаза D последняя.
- Дизайн фазы D вести на повышенной глубине размышления (ultrathink) — клубок
  взаимодействующих механик (двухэтапка × сбор × репутация × возвраты).
