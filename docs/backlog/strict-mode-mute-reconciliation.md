# Строгий режим: мьют без самовосстановления (reconciliation gap)

> Зафиксировано 2026-07-11 при смоуке ветки `chore/telegrambots-9.5` на staging.
> Не регрессия апгрейда — архитектурное свойство слайса 5 с рождения.

## Суть

Мьют/анмьют должников — **чисто событийные** (`MembershipAccessClosedEvent` /
`MembershipAccessOpenedEvent` → `StrictModeService`). Значит:

- ручная правка статуса membership в БД мимо приложения события не рождает — чат не узнаёт;
- админ чата, снявший mute руками, создаёт рассинхрон навсегда;
- потерянное событие (рестарт в момент @Async, сбой Telegram) — рассинхрон навсегда.

Для сравнения: у тегов наград есть `TagSyncScheduler` (30 сек), который лечит любой дрейф.
У мьюта аналога нет.

## Как это проявилось

PO выставил `status=frozen` руками в staging-БД → мьют не случился → «строгий режим не
работает». Одновременно замьюченный вчера участник оказался размьючен (следы вчерашних
циклов тестирования) — приложение это не чинит.

## Кандидат на решение

Реконсиляция в том же проходе `TagSyncScheduler` (клубы со strict=on): участник
frozen/expired в приложении, но не restricted в чате → mute; активный, но restricted → unmute.
Осторожно: не трогать ручные ограничения, наложенные админом на активных участников,
отличить их от наших нельзя — значит unmute-ветку реконсиляции, возможно, не делать вовсе
(mute-only reconciliation, консервативно).

## Смежное

- Кнопка ручной заморозки удалена из UI (PO 2026-07-06) — путь в mute сегодня только через
  шедулер экспирации и вступление должником в платный клуб. Переосмысление freeze-флоу:
  `docs/backlog/freeze-flow-rethink.md`.
- Правило тестирования на staging: **статусные переходы — только через приложение**,
  руками в БД можно править только данные (даты, суммы) — переход мимо бэкенда = потерянное
  событие.
