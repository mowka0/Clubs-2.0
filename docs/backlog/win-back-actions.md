# Win-back: действия над ушедшими участниками (backlog)

**Создано:** 2026-06-21 · **Контекст:** owner-«Статистика» (`docs/modules/club-quality.md` §9).
**Память:** [[project_club_quality_track]].

## Что уже есть (в `feature/owner-statistics`)
Нудж «Верните N ушедших» в owner-«Статистике» **раскрывается** в ростер ушедших
(`GET /api/clubs/{clubId}/churned-members`, owner-only): строки участников + тап на **карточку профиля**
(переиспользуем `MemberProfileModal` из таба «Участники»). Ростер = distinct, кто `left`/`expired` за 30д и
сейчас не active/grace, newest-first. **Только просмотр** — действий пока нет.

## Что отложено (PO попросил зафиксировать на потом)
Над строкой/карточкой ушедшего добавить **действия**, например:
- **Пригласить на встречу** — отправить участнику DM-приглашение на ближайшее/конкретное событие клуба
  (deep-link на событие). Требует: выбор события, шаблон DM-уведомления, rate-limit, идемпотентность,
  трекинг «приглашён → вернулся» (конверсия win-back как метрика эффективности нуджа).
- Возможные смежные: «написать в DM», «предложить промо/скидку на возврат» (когда появится биллинг-логика),
  массовое «позвать всех ушедших».

## Открытые вопросы до реализации
- Канал доставки: бот-DM (нужен `telegram_id`, согласие на сообщения) vs deep-link, который владелец сам
  пересылает. Telegram Bot API не даёт писать первым, если юзер не начинал диалог с ботом — проверить
  ограничение перед «Пригласить на встречу».
- Анти-спам: сколько раз можно звать одного ушедшего; не превращать в назойливые рассылки.
- Приватность: ушедший участник — нужно ли его согласие на повторные приглашения (GDPR/анти-харассмент).
- Метрика возврата: засчитывать ли `rejoined` после приглашения как успешный win-back (атрибуция).

Не начинать без продуктового решения по каналу доставки и анти-спам-правилам.
