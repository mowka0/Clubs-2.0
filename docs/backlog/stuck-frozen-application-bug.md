# БАГ: заявка «одобрена» у участника, но пропала у организатора (frozen + повторная заявка)

**Статус:** open · **Severity: HIGH** (участник заперт в тупике, организатор не видит его нигде) ·
**Создано:** 2026-07-06, репорт PO со staging-теста · **Ветка фикса:** предложено `bugfix/stuck-frozen-application`.

## Репро по словам PO (точное репро НЕ подтверждено кодом — начать с воспроизведения!)

1. Участник X — в клубе, организатор закрыл ему доступ (frozen). X ждёт «Взнос получен».
2. Есть СТАРАЯ pending-заявка от другого участника Y. Организатор одобряет её.
3. Результат: X пропадает из списков организатора («Оплата вступления» / инбокс) — его нет НИГДЕ,
   а у самого X на странице клуба висит «Заявка одобрена — организатор откроет доступ» (см. скрин
   в чате сессии 2026-07-06: клуб «Американский клуб», плашка после блока «Активности клуба
   доступны участникам»).

## Что уже известно из кода (наводки, не выводы)

- `approveApplication` ([ApplicationService.kt:132]) метит только СВОЮ заявку; membership X напрямую
  не трогает → «пропадание» X, скорее всего, рассинхрон списков/статусов, а не прямое влияние.
- Плашка «Заявка одобрена — организатор откроет доступ» на ClubPage рендерится из состояния
  «approved-заявка + frozen/отсутствующий membership» — проверить ветку ClubPage.tsx ~строки 273-320
  (joinedStatus / application status / orphaned approval recovery).
- Бакет «Оплата вступления» строится по `accessStatus === 'frozen'` из members-списка; инбокс —
  по pending-заявкам; кросс-клубовое «Ждут оплаты» — по claimed-frozen (`findFrozenMembersByOwner`
  → `dues_claimed_at IS NOT NULL`?? — проверить: если фильтр по claim, то X без claim в «Мои клубы»
  и не должен показываться, но в клубном бакете «Оплата вступления» обязан быть).
- Гипотезы на проверку: (а) у X заявка осталась в статусе approved с прошлого цикла и «reactivate»
  ветка её не чистит; (б) members-список организатора теряет X из-за фильтра статусов;
  (в) approve заявки Y инвалидацией кэша перезаписывает какой-то общий список.
