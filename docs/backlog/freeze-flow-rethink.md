# Переосмысление freeze-флоу (пауза доступа)

**Статус:** open · **Создано:** 2026-07-06 (PO-фидбек по staging-тесту `feature/members-tab-unification`).

## Что уже сделано (2026-07-06, та же ветка)

- **Кнопка «Закрыть доступ» удалена из UI** (`MemberProfileModal`) по решению PO: просрочку окна
  ежедневно закрывает шедулер (`SubscriptionScheduler` 9:00 → `processExpiry`: active с истёкшим
  `subscription_expires_at` → frozen), ручная пауза дублировала автоматику и путала.
  Оставлен один радикальный рычаг — «Удалить из клуба».
- Удалены мёртвые фронт-клиенты `freezeMember`/`unfreezeMember` + мутации
  (`useFreezeMemberMutation`/`useUnfreezeMemberMutation`; unfreeze не использовался и до этого).
- Устаревший коммент «планировщик спит» в `SubscriptionScheduler` переписан — он уже один раз
  ввёл в заблуждение при анализе (автозаморозка ЖИВАЯ).

## Что осталось переосмыслить

1. **Бэкенд-эндпоинты `/freeze` и `/unfreeze` живы, но с UI не вызываются.**
   `AccessGateService.freezeAccess/unfreezeAccess` + DM «доступ закрыт». Решить: удалить целиком
   (dead code) или оставить под будущий «строгий режим» чата (бот мьютит должника — понадобится
   программная пауза). Решение связано с чат-интеграцией (docs/design/club-chat-link/notes.md).
2. **Семантика frozen перегружена** — сейчас это и «новичок ждёт первого взноса» (авто при approve
   платного клуба), и «просрочил продление» (авто-шедулер). UI-бакеты уже различают их косвенно
   (claim/не-claim, дата вступления), но статусная модель — нет.
3. **DM-коммуникация**: авто-freeze шлёт «доступ истёк» (шедулер, до заморозки); участник после
   заморозки видит ClubPage frozen-вид с «Оплатить взнос»/claim. Проверить сквозной путь возврата
   (claim → red-dot → «Взнос получен») после фикса точки (см. ниже) — ложных тупиков быть не должно.
4. Связанный баг: [[stuck-frozen-application-bug]] (docs/backlog/stuck-frozen-application-bug.md) —
   чинить ПЕРЕД или ВМЕСТЕ с переосмыслением.

## Контекст-факты (проверены 2026-07-06)

- Red-dot attention (`GET /member-attention`) с этой ветки считает только **claimed**-frozen
  (`countClaimedFrozenByClubs`) — просто frozen точку не зажигает (мяч у участника).
- «Своя дата» (`setAccessUntil`) принимает past-дату → active с истёкшим окном до ближайшего тика
  шедулера (тестовый способ получить «истёкшего active»).
