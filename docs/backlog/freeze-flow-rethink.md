**Статус:** open · **Создано:** 2026-07-06 (PO-фидбек по staging-тесту `feature/members-tab-unification`).

## Что уже сделано (2026-07-06, та же ветка)

- **Кнопка «Закрыть доступ» удалена из UI** (`MemberProfileModal`) по решению PO: просрочку окна
  ежедневно закрывает шедулер (`SubscriptionScheduler` 9:00 → `processExpiry`: active с истёкшим
  `subscription_expires_at` → frozen), ручная пауза дублировала автоматику и путала.
  Оставлен один радикальный рычаг — «Удалить из клуба».
- Удалены мёртвые фронт-клиенты `freezeMember`/`unfreezeMember` + мутации
  (`useFreezeMemberMutation`/`useUnfreezeMemberMutation`; unfreeze не использовался и до этого).
- Устаревший коммент «планировщик спит» в `SubscriptionScheduler` переписан — он уже один раз
  ввёл в заблуждение при анализе (автозаморозка ЖИВАЯ).

## Мёртвые статусы + враньё в именах (записано по PO-запросу 2026-07-06)

Проверено grep'ом всех записей `set(MEMBERSHIPS.STATUS, …)`: в de-Stars-модели пишутся только
**три живых статуса** — `active`, `frozen`, `cancelled`. Ещё два из enum (`membership_status`:
`active/grace_period/cancelled/expired/frozen`) — **мёртвые легаси эпохи Stars**:

- **`grace_period`** — не пишется НИКЕМ, но **читается** в ~6 местах как «живой» (занятость слотов,
  качество клуба: `JooqClubRepository`, `ClubService`, `JooqClubQualityRepository`,
  `JooqClubStatsRepository`). Строка в этом статусе — тупик: приложение её не создаёт, шедулер её
  не трогает (`expireOverdueAccess` ловит только `active`), фронт-бакетов для неё нет → падает в
  «Участники». (Именно так выглядел ручной тест PO на «Американском клубе».)
- **`expired`** (статус membership) — тоже не пишется прод-кодом. Не путать с `membership_event`-логом
  `expired` (V31) — то другое.

**Враньё в именах шедулера** (`SubscriptionScheduler`): переменная `enteringGrace` и коммент
«…перейдут в grace_period» — на деле `processExpiry`/`expireOverdueAccess` переводит в **`frozen`**.
Этот коммент уже вводил в заблуждение (2026-07-06 при анализе #3). Шапку класса переписали, но
внутренний `enteringGrace` + коммент на строке ~32 остались — **исправить** (→ `enteringFrozen`/по факту).

**Задача уборки (отдельная мелкая ветка):** задепрекейтить `grace_period`/`expired` в логике
(из enum не удалять — историю не трогаем, но чётко пометить «не использовать» + убрать из
READ-фильтров занятости, если они не должны считаться живыми), исправить имена шедулера.

## ✅ РЕШЕНИЕ PO (2026-07-06): вернуть к жизни статус `expired` для просрочки продления

Целевая модель (слайс 1 этого rethink'а, отдельная ветка после мержа members-tab-unification):

- Шедулер: `active` + просроченное окно → **`expired`** (не `frozen`). Enum-значение уже есть (V3),
  миграции не нужно. Заодно починить враньё имён (`enteringGrace`).
- DM «доступ закрыт — подписка истекла» **уже шлётся** шедулером (SubscriptionScheduler:43) —
  обогатить кнопкой-диплинком «Продлить взнос» (аналог sendAccessFrozenDM).
- `frozen` возвращает себе ЕДИНСТВЕННЫЙ смысл: новый платный участник ждёт первого взноса
  («💸 Оплата вступления», хинт перестаёт врать).
- Бакет «⛔ Доступ истёк» становится статусным (`accessStatus === 'expired'`) — стабильным и
  всегда населённым реальными должниками (снимает транзиентность из предыдущей секции).
- Гейт контента менять не надо: `MembershipAccess.hasAccess()` = только `active` — expired
  автоматически без доступа.

Карта правок (проверена по коду 2026-07-06): `expireOverdueAccess` (флип в expired);
гварды `claimDues`/`markDuesPaid` (+expired); ~4 запроса занятости слотов (+expired в «живые»);
`MembershipActivator.isAlive` (+expired — иначе повторное вступление сотрёт жизненный цикл);
red-dot + «Ждут оплаты» (claimed-frozen-OR-expired); фронт `bucketOf` (по статусу);
карточка участника (ветка expired — сейчас упала бы в «Подписка активна»); ClubPage self-view
expired (аналог frozen-вида с claim); `membership_history` — решить, логировать ли событие
`expired` (лог-enum имеет значение с V31); обновить матрицу status×surface в payment-v2.md.

## Контекст (историческая секция): бакет «⛔ Доступ истёк» почти недостижим без статуса (PO 2026-07-06)

Бакет из #2 (`feature/members-tab-unification`) ловит `active` + `subscription_expires_at` в прошлом.
Но это **транзиентное состояние ≤24ч**: ежедневный тик шедулера (9:00) переводит `active`+просрочка →
`frozen`, и участник уезжает в «💸 Оплата вступления». В устойчивом состоянии бакет «Доступ истёк»
ПУСТ; он честен только в окне между истечением и ближайшим тиком (или при задержке шедулера —
редеплой/простой). Решить в рамках этого рефактора: (а) оставить как честный транзиентный/safety-бакет,
либо (б) убрать и слить с frozen-веткой. Связанная более глубокая проблема — см. ниже.

**`frozen` перегружен ДВУМЯ смыслами, оба в «Оплата вступления»:** (1) новый платный участник, ждущий
ПЕРВОГО взноса (approve → `createFrozen`), и (2) старый участник, у которого просрочилось продление
(шедулер `active`→`frozen`). Хинт бакета «Вступили — подтвердите взнос» верен только для (1); для (2)
он врёт (человек не «вступает», у него ЛАПС продления). Правильный фикс — различать их (по
`joinedAt`/наличию прошлого доступа/`dues_marked_paid_at`) и, возможно, развести на два бакета
(«Новые — первый взнос» vs «Просрочка продления»). Это и есть настоящий дом для «истёкших», а не
транзиентный active-бакет.

## Что осталось переосмыслить

1. **Бэкенд-эндпоинты `/freeze` и `/unfreeze` живы, но с UI не вызываются.**
   `AccessGateService.freezeAccess/unfreezeAccess` + DM «доступ закрыт». Решить: удалить целиком
   (dead code) или оставить под будущий «строгий режим» чата (бот мьютит должника — понадобится
   программная пауза). Решение связано с чат-интеграцией (docs/design/club-chat-link/notes.md).
2. **Семантика frozen перегружена** — сейчас это и «новичок ждёт первого взноса» (авто при approve
   платного клуба), и «просрочил продление» (авто-шедулер). UI-бакеты уже различают их косвенно
   (claim/не-claim, дата вступления), но статусная модель — нет.
3. **DM-коммуникация**: авто-freeze шлёт «доступ истёк» (шедулер, до заморозки); участник после
   заморозки видит ClubPage frozen-вид с «Оплатить взнос»/claim. Проверить сквозной путь возврата
   (claim → red-dot → «Взнос получен») после фикса точки (см. ниже) — ложных тупиков быть не должно.
4. Связанный баг: [[stuck-frozen-application-bug]] (docs/backlog/stuck-frozen-application-bug.md) —
   чинить ПЕРЕД или ВМЕСТЕ с переосмыслением.

## Контекст-факты (проверены 2026-07-06)

- Red-dot attention (`GET /member-attention`) с этой ветки считает только **claimed**-frozen
  (`countClaimedFrozenByClubs`) — просто frozen точку не зажигает (мяч у участника).
- «Своя дата» (`setAccessUntil`) принимает past-дату → active с истёкшим окном до ближайшего тика
  шедулера (тестовый способ получить «истёкшего active»).
