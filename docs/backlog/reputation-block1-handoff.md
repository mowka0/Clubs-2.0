# Репутация — Блок 1 (живой путь): EXP-2 + ATT-2 + ATT-3 — handoff в новую сессию

**Статус:** зафиксировано, код — в НОВОЙ сессии · **Создано:** 2026-06-07 ·
**Origin:** решение делать репутацию по блокам (не «всё сразу»), группируя по одному сквозному
ручному e2e-сценарию. Реестр всех багов: [`two-stage-reputation-bug-register.md`](./two-stage-reputation-bug-register.md).
Тест-план: [`docs/qa/reputation-test-plan.md`](../qa/reputation-test-plan.md). Спека P1a: [`reputation-v2.md`](../modules/reputation-v2.md).

## Контекст / почему так
Весь блок репутации (P1b Trust/XP, P2 финансы+антифарм, P-org, P3 геймификация) **не делаем разом** —
там много незакрытых продуктовых решений ([redesign §11](./reputation-v2-redesign.md)) и кодить наугад = переделка.
Группируем по **единице ручного теста = сквозной сценарий**. Блок 1 = всё, что трогает **живой путь
репутации** «подтверждение → явка → финализация → начисление». Его тестируем ОДИН раз целиком,
а не прогоняем ручной e2e после каждого мелкого фикса.

## Состояние веток
- **`feature/two-stage-confirmation-gaps`** — на staging, **НЕ мержена** (Bug B, Feature A `expired_no_confirm`,
  roster confirmed-only, 4 фикса, reminders A/B). Всё зелёное.
- **План:** Блок 1 (EXP-2/ATT-2/ATT-3) **доложить в эту же ветку** в новой сессии → единый e2e-тест
  всего живого пути → «готово, запушь» → один PR. (Если ветку к тому моменту уже смержили —
  новая ветка `feature/reputation-block1`.)

## Scope Блока 1 (3 бага живого пути)

### EXP-2 (HIGH) — авто-финализация непомеченных событий
**Проблема:** `finalizeAttendance` гейтит на `attendance_marked=true`. Орг не отметил явку → событие
никогда не финализируется → репутация не считается. Главная причина «5 событий — всё ещё Новичок».

**РЕШЕНИЕ (продуктовое, зафиксировано 2026-06-07):** на дедлайне непомеченное прошедшее событие
**авто-финализируется с НЕЙТРАЛЬНЫМ исходом** — всем `confirmed`-участникам **0** (событие просто
не засчитываем: ни +100, ни −50). Никого не наказываем и не награждаем за бездействие организатора.

**Под-вопросы для спеки (новая сессия):**
- **Дедлайн:** предложение — **48ч после `event_datetime`** (через 24ч уходит напоминание оргу — Feature B,
  + ещё сутки на реакцию). Согласовать с C (штраф оргу) и dispute-window.
- **Как «нейтрально» технически:** рекомендация — **НЕ писать ledger-строку вообще** для авто-нейтральных
  (истинно «не засчитываем»: не растит `total_confirmations`, не портит `promise_fulfillment_pct`).
  Альтернатива — reuse `confirmed_unresolved` (0 очков, но conf+1 → процент проседает). **Выбрать в спеке.**
  Если новый путь — нужен флаг на events (напр. `attendance_auto_finalized`/новый), чтобы reputation-пайплайн
  отличал «нейтрально закрыто» от «помечено орг.».
- **Связь с C** (штраф оргу за неотметку, отложен в [register](./two-stage-reputation-bug-register.md)):
  нейтральная авто-финализация защищает УЧАСТНИКОВ; штраф оргу — отдельно, с фичей репутации организатора.

### ATT-2 (HIGH) — нерешённый спор обнуляет −50
**Проблема:** confirmed+going+absent (−50 «Пустозвон»). Участник жмёт «Оспорить» → `disputed`. Орг не
реагирует → на финализации остаётся `disputed` → `confirmed_unresolved` = 0. Штраф −50 обходится одним тапом.

**РЕШЕНИЕ (PRD-aligned, предложено):** на финализации остаточные `attendance='disputed'` → вернуть в
**исходную отметку `absent`** (окно оспаривания истекло без коррекции орг. → действует первичная отметка)
→ `attendanceKind(going, absent) = no_show (−50)`. Подтвердить продуктово в спеке.

### ATT-3 (MEDIUM) — спор недостижим (нет DM)
**Проблема:** `sendAttendanceMarked` (DM «вас отметили отсутствующим, оспорьте») — мёртвый код, ноль вызовов.
Без него absent-участник не знает, что надо спорить в 48ч окно.

**РЕШЕНИЕ:** вызвать `notificationService.sendAttendanceMarked(eventId)` после `markAttendanceMarked`
в `AttendanceService.markAttendance` (инжектить NotificationService; @Async, ошибки не роняют tx).

## Как стартовать новую сессию
Сказать: **«продолжаем по reputation-block1»** (или «фича: блок 1 репутации»). Флоу как обычно:
Analyst спека (закрыть под-вопросы EXP-2: дедлайн + нейтральный механизм) → код → ревью → Security →
Tester → staging → единый e2e по тест-плану. Тесты-first для `reputation/` (рекомендация по TDD из прошлой сессии).
