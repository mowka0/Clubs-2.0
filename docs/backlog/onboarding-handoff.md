# Handoff: онбординг (срез 1) — реализован, ждёт проверки на staging

> Ветка `feature/onboarding`. Обновлено 2026-07-13 (вторая сессия — код написан).
> **Спека:** `docs/modules/onboarding.md` — приведена в соответствие с реализацией.

## 1. Где мы

**Срез 1 написан целиком** (backend + frontend + тесты), прогнан ревью и security-аудит.
Осталось: ручная проверка на staging и мерж.

- **Backend:** `POST /api/users/me/onboarding` (тело `{"door":"MEMBER"|"ORGANIZER"}`) →
  `users.onboarded_at = now()`, возвращает `UserDto`. Повтор → 409. Миграция `V60`.
- **Frontend:** карусель из 3 слайдов (свайп + стрелки + точки) в `components/onboarding/`,
  гейт в `Layout.tsx` (рендерится **вместо** `<Outlet/>`), подсветка цели после выхода из двери.
- **Тесты:** 8 бэкенд (`OnboardingControllerTest`, Testcontainers) + 15 фронт
  (`OnboardingFlow.test.tsx` — 8, `OnboardingGate.test.tsx` — 7).

## 2. Что за фича (в двух абзацах)

Онбординг — **карусель из 3 слайдов, без единого вопроса**. Город/категории/интересы не спрашиваем:
клубов в системе нет, подбирать не из чего (PO отменил это по ходу — сначала спроектировали
таксономию, потом выбросили, см. §5).

Слайды: (1) что это и что даёт — 4 общих преимущества; (2) путь участника — 3 преимущества + кнопка
«Найти клубы в своём городе»; (3) путь организатора — 4 преимущества + кнопка «Создать клуб и пригласить
друзей». Кнопка живёт на своём слайде, а не сваливается в конце. Тексты — в `components/onboarding/slides.ts`,
дословно из спеки, утверждены построчно, **не переписывать**.

## 3. Решения, принятые при реализации (сверх спеки)

Спека их уже впитала, но чтобы не искать глазами:

1. **У запроса появилось тело `{door}`.** Спека говорила «тела нет», но нефункциональное требование
   велит логировать выбранную дверь — без неё сервер её просто не знает. Дверь в БД **не хранится**,
   только INFO-лог: это метрика намерения, колонка под неё была бы данными без потребителя.
2. **409 на фронте — не ошибка.** «Уже пройден» означает, что цель достигнута (прошли с другого
   устройства, эта сессия устарела). Фронт перечитывает профиль и идёт дальше, а не запирает человека
   в карусели. Иначе устаревший стор = ловушка до перезапуска мини-аппа.
3. **Гонку двойного тапа снимает сам UPDATE:** условие `onboarded_at IS NULL` стоит внутри него,
   а не отдельной проверкой перед. Read-then-write не бывает, второй запрос честно получает 409.
4. **`startParam` откладывает карусель шире, чем «только инвайт»** — любая deep-ссылка (клуб, событие,
   складчина). Пришедший по ссылке шёл к конкретной вещи, а не знакомиться с продуктом.
5. **Тип `UserDto` схлопнут в один** (`types/api.ts`); дубль в `useAuthStore` убран — два описания
   одного ответа разъехались бы на первом же новом поле.
6. **Порядок шагов двери НЕ переставлять** (баг с первого захода на staging): `ответ сервера →
   navigate → setUser`. Профиль в сторе — это гейт; запиши его раньше навигации, и карусель
   размонтируется, а TanStack не вызовет колбэки `mutate(...)` (у наблюдателя нет слушателей) —
   навигация теряется целиком, вместе с подсветкой. Поэтому мутация **не пишет в стор сама**.
   Ловится только тестом через ГЕЙТ (`OnboardingGate.test.tsx`), не через голую карусель.
7. **Название продукта на слайде 1** набрано шрифтом Allura (SIL OFL, самохостится).
   Файл — **сабсет ровно под слово «Clubs»** (`src/assets/fonts/Allura-Clubs.woff2`, 1.4 КБ вместо
   232 КБ). Любой другой текст этим шрифтом не отрисуется, пока сабсет не пересобран:
   `pyftsubset Allura-Regular.ttf --text="Clubs" --flavor=woff2`.

## 4. Ключевые решения PO (НЕ пересматривать)

- **Вопросов нет вообще.** Ни города, ни категорий, ни интересов.
- **Онбординг обязателен**, кнопки «Пропустить» нет. Пройденным считается **только тап главной кнопки слайда**.
- **Инвайт откладывает карусель, а не отменяет:** приглашённый идёт в клуб, `onboarded_at` остаётся NULL
  → при следующем обычном запуске карусель покажется.
- **Бэкфилла существующих пользователей нет** — реальных людей в проде нет.
- **Тон на «ты».** Слоган: «Найди своих для живых встреч!» / «Наполни свою жизнь активностями!».
- Кнопки ведут **на страницу с подсветкой**, а не открывают форму: человек должен запомнить, где эта кнопка живёт.

## 5. Что спроектировали и осознанно отложили

По ходу первой сессии была детально спроектирована **таксономия «категория → интерес»**: `interests.category`,
`user_categories`, `club_interests`, затравка словаря на ~100 интересов по 8 категориям, правила подбора клубов,
эндпоинт рекомендаций, разметка клубов интересами. **PO выбросил это из среза** («клубов нет — подбирать не из
чего»), но не насовсем: вернуть, когда в системе появятся клубы.

Всё лежит в истории ветки, коммиты `bda16be` → `947b3be`. Не изобретать заново — достать оттуда.

Остальное отложенное: чек-лист организатора (срез 2), приветствие новичка в клубе (срез 3), предложение
заполнить интересы при первом заходе в профиль, DM-напоминания (бот не может писать первым — нужен
`requestWriteAccess`).

## 6. Что проверить на staging

1. Новый пользователь (`onboarded_at = NULL`) видит карусель вместо приложения; листается свайпом и стрелками.
2. «Найти клубы в своём городе» → каталог, селектор города пульсирует пару секунд.
3. «Создать клуб и пригласить друзей» → «Мои клубы», кнопка «+ Клуб» пульсирует. Форма сама **не** открывается.
4. Перезапуск мини-аппа → карусели больше нет.
5. Закрыть на 2-м слайде, не нажав дверь → при следующем запуске карусель снова, с начала.
6. Заход по инвайт-ссылке новым пользователем → сразу в клуб, карусели нет; следующий обычный запуск → карусель.

Сброс для повторного теста: `UPDATE users SET onboarded_at = NULL WHERE telegram_id = <твой>;`
