# MyClubs — секция «Рекомендуем» внизу страницы

**Статус:** open · **Создано:** 2026-05-16 · **Origin:** при редизайне MyClubsPage обнаружилось что у пользователя с малым числом клубов остаётся ~250px пустого пространства между последней карточкой и bottom tab bar на больших экранах (iPhone 14 Pro 844px viewport).

## Идея

Когда у user'а 1-3 активных клуба, скролл-страница MyClubsPage оказывается короче viewport'а, остаётся пустота. Хороший способ её заполнить продуктивно — секция «Рекомендуем» внизу с 2-4 клубами по интересам.

```
[Header: «Мои клубы» + «+ Создать»]
[АКТИВНЫЕ · 1]
[ClubCard]
[ЗАЯВКИ · 0 или скрыто]
[Empty space  →  становится РЕКОМЕНДУЕМ секцией]
[РЕКОМЕНДУЕМ · 3]
[ClubCard (with «Вступить» CTA на карточке?)]
[ClubCard]
[ClubCard]
```

## Зачем

- **Дискавери внутри MyClubs.** Пользователь не должен переходить в Поиск чтобы найти ещё клубы — мы сами подсказываем релевантное.
- **Активация и retention.** Чем больше активных клубов у пользователя — тем выше engagement и LTV.
- **Заполняет пустое пространство.** На больших экранах MyClubs с 1-2 клубами не пустует.

## Что нужно сделать

### Backend
- Новый endpoint `GET /api/clubs/recommended` (или `GET /api/users/me/recommended-clubs`):
  - Возвращает 3-5 клубов, в которых user НЕ состоит и НЕ имеет pending application
  - Логика подбора (по убыванию приоритета):
    - **MVP:** клубы в том же городе что у user'а (geo-match)
    - **V2:** + по категориям клубов где user уже состоит (collaborative-filtering lite)
    - **V3:** + activity rating (предпочитать активные клубы)
  - Privacy: returns только public-видимые поля (то же что в search)

### Frontend
- В MyClubsPage добавить новую секцию `mc-section-label` «Рекомендуем» + `mc-list` с recommended-карточками
- Reuse `<ClubCard>` компонент из Discovery (когда он будет вынесен — сейчас он inline)
- Рендерится только если: (a) есть рекомендации, (b) у user'а < 10 активных клубов (если уже много — не загромождаем)

## Acceptance Criteria

- **AC-1:** User с 0 активных клубов видит «Рекомендуем · 3-5» секцию ниже empty-state карточки.
- **AC-2:** User с 1-9 активных клубов видит секцию после «Активные» / «Заявки» секций.
- **AC-3:** User с ≥ 10 активных клубов **не видит** секцию (опытный, ему не нужны подсказки).
- **AC-4:** Тап по карточке → переход на ClubPage этого клуба (как из Discovery).
- **AC-5:** Если backend возвращает пустой массив (нет подходящих) — секция не рендерится.
- **AC-6:** Endpoint имеет rate-limit на user (по аналогии с Discovery `/api/clubs/search`).

## Открытые вопросы

- **R-1:** Должна ли карточка рекомендации иметь inline «Вступить» CTA как на Discovery, или просто тап → ClubPage где CTA внутри? Для V1 предлагаю — просто навигация (single tap), чтобы карточка была компактнее.
- **R-2:** Сколько клубов показывать? 3 — клубный минимум, 5 — заполнит больше пустоты но может казаться спамом. Определимся когда увидим UX на staging.
- **R-3:** Должны ли recommended-клубы исключать private клубы? Скорее всего да — приватные не должны попадать в discovery flows (privacy invariant из PRD §4.1.3).
- **R-4:** Refresh strategy — каждое открытие MyClubs? Или один раз за сессию (cache в TanStack Query с stale-time 5 минут)?

## Не приоритет

Не блокер ни для чего. Лучше внедрять когда уже есть критическая масса клубов в системе (иначе рекомендовать особо нечего). Текущая база < 100 клубов — можно подождать.
