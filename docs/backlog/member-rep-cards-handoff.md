## 1. Где мы / что решили

- **PR-c** (XP/уровни/бейджи + калибровка) ✅ **в проде (#64)**. Self-панель уровня на профиле — нравится, **НЕ трогать**.
- **PR-e** (показ ГЛОБАЛЬНОГО уровня ДРУГИМ в мембер-листе/карточке) — на ветке `feature/reputation-p1b-pr-e`, на staging, **НЕ мержить**. PO отверг.
  **Причина (важно):** глобальный сигнал на КЛУБНОЙ поверхности = смысловая ошибка. Зритель читает всё на карточке участника как «про этого человека В ЭТОМ клубе». Отсюда диссонанс «Новичок в клубе (per-club Trust скрыт <3 исходов) + Столп сообщества (global)». Уровень/бейджи — глобальные, гейт «Новичок» — per-club, и они путались.

### Принцип (ЗАЛОЧЕН)
- **Карточка участника (строка списка + модалка) = ТОЛЬКО per-club.** Никакого глобального уровня/бейджей.
- **Глобальная репутация профиля = карточка ЗАЯВКИ** (+ свой профиль, который уже есть). В заявке кандидата ещё НЕТ в клубе → глобальное уместно и спутать не с чем; админу помогает решать.

## 2. Дизайн карточек (ЗАЛОЧЕН — см. final.html)

### 2.1 Карточка участника — «кольца активности» (per-club)
Модалка `MemberProfileModal`:
- Шапка: аватар + имя + `@username` + «в клубе с DATE».
- **О себе** (bio) + **Интересы** (chips) — **НОВОЕ** (сейчас в `MemberProfileDto` их нет).
- **Репутация в этом клубе** — **2 кольца** (SVG-донаты):
  1. **Надёжность** (Trust 0–100) — «умный» показатель (байес + recency-decay + асимметрия). Тир-цвет (зелёный high / accent mid / красноватый low).
  2. **Посещаемость** = посещено/подтверждено (напр. «38/41 = 93%») — сырая прозрачная дробь. Accent.
- Подпись: спонтанные визиты (count) + роль.
- **Бейджей НЕТ** (PO: можно без них, упор на инфографику).
- ⚠️ **Третье кольцо «обещания» УБРАНО как дубль:** `promiseFulfillmentPct = посещено/подтверждено×100` = ровно та же «посещаемость», только в %. НЕ показывать дважды.

Строка в списке (`ClubMembersTab`): per-club only (Trust-число + «Новичок»). **Убрать `levelName`-чип** (откат фронта PR-e).

### 2.2 Карточка заявки — «донат N из M» (global)
Модалка `ApplicationReviewModal`:
- Шапка: аватар + имя + `@username` + город + **пилл уровня** (золото у топ-тира, accent у среднего, серый у базового).
- **О себе** + **Интересы** — **УЖЕ ЕСТЬ** (`ApplicantInfoDto`, модалка их рендерит).
- **Активность на платформе** — донат «надёжна в N из M клубов» (напр. 7/8) + строка «посетила X из Y событий · активна в N клубах».
- Ответ на вопрос + кнопки Принять/Отклонить (как сейчас).
- ⚠️ Дисклеймер «(не организаторский опыт)» **НЕ добавлять** — заголовок «Активность на платформе» сам задаёт рамку (PO решение).

## 3. Формулы / данные (сверено с кодом)

- `promiseFulfillmentPct = totalAttendances / totalConfirmations × 100` — `JooqReputationRepository.kt:219`.
- **Trust** (надёжность) — `TrustPolicy.perClubTrust`: байес-доля сдержанных по KIND, prior .85, K3, ASYM2, HL90. Per-club, on-read.
- **Глобально «N из M»** — `TrustService.global` → `reliableClubs` (N, Trust≥70 среди клубов с ≥3 исходами) / `trackRecordClubs` (M).
- **Глобальный уровень** — `XpPolicy`/`XpService` (по всему ledger; кривая `50·n^2`, 10 уровней Гость→Амбассадор).
- Per-club данные уже в `MemberProfileDto`: `trust`, `promiseFulfillmentPct`, `totalConfirmations`, `totalAttendances`, `spontaneityCount`, `role`, `joinedAt`. Гейт `show` = `outcome_count ≥ 3` → иначе «Новичок».
- ⚠️ **Owner-blind:** `aggregateByUserIds`/ledger исключают СВОЙ клуб владельца (анти-фарм rule 1). Значит глобальная репа = **участницкий след**, слепа к организаторскому опыту. Подпись строго «активность участника», не overclaim как «надёжность организатора».
