# Rate limiting: ключ считается до аутентификации, IP берётся из недоверенного заголовка

**Статус:** бэклог · **Найдено:** 2026-07-13, security-ревью ветки `feature/onboarding`
**Приоритет:** средний · **Не относится к онбордингу** — существует до этой ветки, найдено попутно.

## Что не так

Две связанные проблемы в `backend/src/main/kotlin/com/clubs/common/security/`:

1. **`RateLimitFilter` стоит ПЕРЕД `JwtAuthenticationFilter`** (`SecurityConfig.kt`, цепочка
   `addFilterBefore`). На момент фильтрации `SecurityContext` пуст, поэтому `resolveKey()` всегда
   уходит в ветку `ip:` — ключа `user:` не получает ни один `/api/**`-запрос. То есть лимит
   «по пользователю» в коде есть, но не срабатывает никогда.

2. **`getClientIp()` доверяет первому значению `X-Forwarded-For`**, пришедшему от клиента.
   Заголовок подделывается тривиально, значит лимит обходится его ротацией — включая жёсткий
   лимит 5/мин на `/api/auth/*`, который как раз и защищает от подбора.

## Почему это важно

Второй пункт обесценивает защиту auth-эндпоинтов: именно там лимит и нужен, а обойти его может
кто угодно, кто умеет ставить заголовок. Первый — тише: лимит по IP работает, просто он грубее
задуманного (за одним NAT/корпоративным выходом люди делят бюджет).

## Что сделать

- IP брать только из доверенного источника — заголовок, который проставляет наш Traefik, а всё,
  что пришло от клиента, игнорировать (или парсить `X-Forwarded-For` справа налево, отбрасывая
  недоверенные хопы).
- Ключ лимита считать **после** аутентификации: либо переставить фильтры, либо разделить лимит
  на два — грубый по IP до JWT и точный по `user_id` после.
- Тест: подделанный `X-Forwarded-For` не должен обнулять счётчик.

## Что НЕ трогать

Новый `POST /api/users/me/onboarding` в отдельном лимите не нуждается: операция идемпотентна
(повтор → 409), стоит один индексированный UPDATE, общего `API_LIMIT_PER_MIN` достаточно.
