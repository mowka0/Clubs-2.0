## Проблема

Внутри клубов регулярно возникают сценарии, когда нужно собрать деньги с участников:
- **Pre-event сбор:** забронировать корт, купить инвентарь, оплатить аренду помещения, оптовая закупка
- **Post-event split:** разделить счёт в кафе, компенсировать админу его траты на общий ресурс

Сейчас это делается в обычном TG-чате клуба: админ кидает реквизиты → бегает за каждым → ведёт учёт в голове или табличке. Главная боль — pre-event случай, потому что **нужно собрать ДО действия**, без инструмента это реальный пэйн.

Дополнительно: нет следа ответственности. Тот, кто систематически "забывает" оплатить, не несёт никаких последствий, кроме раздражения админа.

## Идея

Складчина = объект внутри клуба (рядом с событием), со своим жизненным циклом и привязкой к рейтингу.

**Что приложение делает:**
- Создание сбора (название, описание, общая цель, дедлайн, фото-вложение — например чек)
- Выбор участников (всех / конкретных / опционально-добровольных)
- Распределение суммы (поровну / индивидуальные суммы)
- Хранение ссылки на оплату (СБП-перевод, банковский deep-link, что угодно)
- Рассылка через бота указанным участникам
- Трекинг статуса каждого участника (`pending` → `paid_unconfirmed` → `paid_confirmed` / `declined`)
- Прикрепление доказательства оплаты (скрин чека)
- Подтверждение админом (сверяет с историей переводов в своём банке)
- Хук на рейтинг (опциональный — см. ниже)

**Что приложение НЕ делает:**
- Не процессит деньги. Никаких Telegram Stars, ЮMoney, СБП-интеграции на стороне приложения. Все деньги идут напрямую через банк админа.
- Не верифицирует факт оплаты автоматически. Источник истины — банковская выписка админа + скрин от участника.

Это снимает регуляторные риски (115-ФЗ, статус агрегатора, лицензии) — приложение остаётся "доской задач со статусами", не платёжной системой.

## Рейтинговая модель — гибкая, выбор админа

При создании складчины админ выбирает один из трёх режимов:

| Режим | Описание | Влияние на рейтинг |
|---|---|---|
| **Обязательная** | Все указанные участники должны скинуться | Оплатил → +small / Не оплатил без явного отказа → −medium / Отказался явно → −small |
| **Требуется решение** | Все должны ответить (да/нет), но может отказаться | Оплатил → +small / Не ответил вовремя → −small / Отказался явно → 0 |
| **Добровольная** | Кто хочет — скидывается | Оплатил → 0 (или +nano для "спасибо") / Не оплатил → 0 |

**Принцип:** молчание ≠ отказ только в обязательном режиме. В остальных — рейтинг страдает только от явной безответственности (не ответил в "требуется решение"). Это снимает токсичность "минус за неудобный сбор".

Видимость рейтинга в клубе создаёт "цену слова" — у активного, ответственного участника репутация выше, его приглашают чаще, ему доверяют админство в подсборах. Гейм-механика: прокачка индекса ответственности через регулярное участие.

## Use cases (валидированы пользователем)

1. **Бронь корта / зала.** Админ бронирует, должны скинуться 8 из 10 участников. Обязательный режим — кто не скинется, корт не оплачивается, или админ покрывает разницу.
2. **Оптовая закупка / общий инвентарь.** Команда покупает мячи / форму / снаряжение. Обязательно для тех, кому это нужно.
3. **Post-event split (счёт в кафе).** Админ загружает фото чека, делит между участниками встречи. Режим "требуется решение".
4. **Подарок / общий фонд.** Сбор на подарок коллеге, тиммейту. Добровольный режим.

## Антипаттерны / где не работает

- Лекторий, книжный клуб, чисто-общение клубы — у них нет общих расходов. Фича будет лежать. Это окей: модуль опциональный, не навязывается всем клубам.

## Что нужно сделать

### Backend

Новый модуль `skladchina` (по той же слоистой архитектуре, что и `event`):
- `Skladchina` domain: id, club_id, creator_id, title, description, total_goal_kopecks, deadline, payment_link, payment_method_note, mode (mandatory/required/voluntary), receipt_photo_url, status (active/completed/cancelled), created_at
- `SkladchinaParticipant` domain: skladchina_id, user_id, amount_kopecks, status (pending/paid_unconfirmed/paid_confirmed/declined), proof_photo_url, paid_at, confirmed_at, confirmed_by
- Flyway миграция: 2 таблицы + FK на clubs / users
- `SkladchinaService` — бизнес-логика (создание, добавление участников, статус-переходы)
- `SkladchinaReputationHook` — отдельный класс, подписан на статус-переходы, влияет на reputation только если mode ≠ voluntary
- `BotSkladchinaNotifier` — отправка ссылки участникам, напоминания (throttled, см. R-3)
- REST: `POST /api/clubs/{id}/skladchina`, `GET /api/clubs/{id}/skladchina`, `GET /api/skladchina/{id}`, `POST /api/skladchina/{id}/participants/{userId}/mark-paid`, `POST /api/skladchina/{id}/participants/{userId}/confirm`, `POST /api/skladchina/{id}/participants/{userId}/decline`

### Frontend

- **Навигация:** не отдельный таб в bottom nav, а **третий таб внутри ClubPage** рядом с About / Events / Members → добавить "Сборы". Альтернатива — объединить с Events в "Активность" с типом entry. Решается на этапе спеки.
- `SkladchinaCreatePage` — форма (название, описание, сумма, выбор участников, режим, ссылка, опц. фото чека)
- `SkladchinaCard` — рендер карточки сбора с прогресс-баром (collected / goal), счётчик участников
- `SkladchinaDetailPage` — детали + список участников с их статусами + доказательства
- `MySkladchinaPage` (опц.) — мои активные сборы по всем клубам, чтобы не забыть

### Bot

- При создании складчины → рассылка DM с описанием + кнопкой "Открыть в приложении"
- Reminder через 24h до дедлайна для тех, кто `pending`
- Throttling: не более одной DM-агрегации в сутки на юзера, при множественных событиях — одно сообщение со списком

## Acceptance Criteria

- **AC-1:** Админ может создать сбор, выбрать участников из членов клуба, задать сумму (общую с равным дележом или индивидуальную), режим и дедлайн.
- **AC-2:** Указанные участники получают DM от бота с ссылкой на сбор и кнопкой открытия в приложении.
- **AC-3:** Участник в приложении видит складчину, может: открыть платёжную ссылку, отметить "оплатил" (опционально приложив скрин), явно отказаться.
- **AC-4:** Админ видит список участников с их статусами, может подтвердить оплату вручную после сверки с банковской выпиской.
- **AC-5:** При подтверждении оплаты (или при `decline` или при истечении дедлайна без ответа) — срабатывает reputation-hook по правилам режима.
- **AC-6:** Складчина переходит в `completed` когда все участники в финальных статусах (paid_confirmed / declined) ИЛИ когда дедлайн истёк (тогда оставшиеся `pending` → `expired_no_response`).
- **AC-7:** В клубе с режимом `voluntary` — рейтинг не меняется ни у кого, никаких уведомлений-укоров.

## Risks / Open Questions

- **R-1: Хрупкий цикл подтверждения.** Юзер платит в банке → возвращается в приложение → отмечает "оплатил" → админ сверяет → подтверждает. 4 шага, два вне приложения. Слабое звено — шаг 2 (юзер забыл вернуться). Митигация: bot-нотификация через 10-30 мин после клика по платёжной ссылке "ты оплатил? нажми ✅". Без хорошего напоминания данные протухнут.
- **R-2: Ссылки на сбор не стандартизированы между банками.** Тинькофф/Сбер/Альфа/ВТБ — у каждого свой формат. На старте не пытаться структурировать — просто текстовое поле "ссылка" + "метод оплаты" свободным текстом. Если в проде увидим что 80% используют Тинькофф-Сбор — сделать deep-link шаблон отдельной итерацией.
- **R-3: Админ — единая точка доверия.** Если админ ленивый или нечестный — система ломается. Митигация: участник прикладывает скрин чека/перевода → видно другим участникам → у админа нет пространства "забыть подтвердить". Спор разрешается коллективным просмотром. Это критично — иначе одна негативная история убивает доверие к фиче.
- **R-4: Спам бота.** 3 активные складчины × напоминания = много сообщений. Throttling обязателен (см. backend секцию).
- **R-5: Что если участник пометил "оплатил" с фейковым скрином?** Админ при сверке с банком увидит расхождение → `decline` с reason. Возможный штраф рейтинга больше обычного "не оплатил" (за обман). Дизайн правила — в спецификации.
- **R-6: Что если админ не подтверждает оплату долго?** Авто-подтверждение по таймауту не делать (даст способ накручивать рейтинг). Лучше — уведомление админу через 48h "у тебя N неподтверждённых оплат".
- **R-7: Privacy.** Видимость списка участников и их статусов внутри клуба — это нормально (это уже клубная инфа). Но `payment_link` со ссылкой на банк админа — это его телефон/реквизиты в открытом доступе для клуба. Должно быть осознанным решением админа при создании.
- **R-8: Что считать "обязательным" участником в большом клубе?** В клубе на 100 человек обязательный сбор для всех — странно. Возможно, ограничить обязательный режим максимум N участниками (10-20) или сделать UI-warning при превышении.

## Решение откладывания

Не строить сейчас. Причины:
1. Core (events + reputation) ещё не валидирован в проде — нужно убедиться что базовая модель работает.
2. Нужен профиль реальных клубов — для лекториев и хобби-клубов эта фича бесполезна, важно понимать каких клубов в проде больше.
3. Объём работы — 2-3 недели solo fullstack на MVP, нет смысла тратить до того как валидирован MVP.

**Триггер к возврату:** когда 5+ клубов в проде явно спросят "как нам собрать деньги на X" — возвращаемся к этой спеке с конкретными use case'ами от реальных пользователей.
