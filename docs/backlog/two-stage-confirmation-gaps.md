## 1. Процесс как задуман в PRD (§4.4)

- **Этап 1 (§4.4.1):** голосование `going`/`maybe`/`not_going` (за 3–5 дней, по умолч. 14).
  Можно менять до этапа 2. `event.status = upcoming`.
- **Этап 2 (§4.4.2):** cron **за 24ч до события**. Сценарий А (избыток «Пойду» > лимита): уведомить
  всех «Пойду», **первые N подтвердивших → финал, остальные → лист ожидания** (FIFO по timestamp).
  Сценарий Б (недобор): подтвердить всех «Пойду», затем добрать из «Возможно». При отказе
  подтверждённого — первый из листа ожидания автоматически в финал.
- **§4.4.3 Отметка явки (после события):** организатор отмечает Пришёл/Не пришёл; окно оспаривания **48ч**;
  после — финализация → пересчёт репутации.
- **§4.4.4 Начисление:** только confirm/decline-исходы (таблица: Железобетонный/Пустозвон/Передумавший/
  Спонтанный/Вечный сомневающийся/Молчун). **Репутация считается ТОЛЬКО после финализации** (48ч после отметки).
- **§623 (структура):** `scheduler/ — Spring Scheduler (Этап 2, **авто-отклонение**)` — PRD **явно
  предполагает авто-отклонение** не подтвердивших.

## 2. Как реализовано (код)

- **Этап 2 триггер:** `Stage2Service.triggerStage2ForReadyEvents` (@Scheduled 5 мин) →
  `findEventsToTriggerStage2` (`event_datetime <= now + STAGE_2_TRIGGER_HOURS_BEFORE_EVENT`, status=upcoming,
  not triggered) → `transitionToStage2` (status→`stage_2`). При триггере going-voters сверх лимита
  (по timestamp) сразу `waitlisted`; первые N остаются `stage_2_vote=null` (должны подтвердить явно).
- **confirm** (`Stage2Service.confirmParticipation`): только при `status=stage_2`. Если
  `confirmedCount < participantLimit` → `confirmed`, иначе → `waitlisted`. Идемпотентно. Вместимость
  считается по **confirmed**, не по going.
- **decline:** → `declined`; если был `confirmed` — промоут первого `waitlisted` в `confirmed`.
- **Закрытие этапа 2:** только `EventCompletionService.markPastEventsCompleted` (@Scheduled) флипает
  `stage_2 → completed` **после** события (+ grace). Отдельного «дедлайна этапа 2» нет.
- **Отметка явки + финализация:** `AttendanceService` (mark → 48ч → `finalizeAttendance`), затем
  reputation-пайплайн (P1a).

## 3. ДЫРЫ (найдено)

### Дыра A — нет авто-отклонения не подтвердивших (PRD §623 НЕ реализован)
**Нет ни одного scheduler'а, закрывающего этап 2 / авто-отклоняющего.** `going`/`maybe`-голос, который
не подтвердил в окне этапа 2, остаётся `stage_2_vote=null`, `final_status=null` — «ни confirmed, ни
declined». Места он не занимает (вместимость по confirmed), но:
- Ростер не финализируется до завершения события.
- Для репутации `final_status=null` → не в таблице §4.4.4 → 0 (как «Молчун»/«Вечный сомневающийся»).
- **Предложение пользователя (валидно, PRD-aligned):** на дедлайне этапа 2 (старт события?)
  авто-проставлять не подтвердившим `declined` (или отдельный статус `expired_no_confirm`), чтобы:
  ростер был явным; «кто нажал подтвердить — тот в деле»; промоут из waitlist срабатывал и для
  «протухших» (сейчас промоут только на явный decline confirmed-участника).

### Дыра B — этап 2 не закрывается на старте события (баг, который видел пользователь)
`confirmParticipation` требует только `status=stage_2`. После наступления события статус **остаётся**
`stage_2`, пока `markPastEventsCompleted` не флипнет (по расписанию, после event+grace) → есть окно
`[event_start, scheduler]`, где **можно подтвердить прошедшее событие**, и фронт показывает кнопку
(`showStage2 = status==='stage_2'`). Нужно: закрывать этап 2 на `event_datetime` (бэкенд-гард
`event_datetime > now` в confirm/decline + фронт-условие), и/или авто-финализация ростера на старте.

### Открытый вопрос — таймлайн авто-отклонения
Когда авто-отклонять не подтвердивших? Варианты: (а) на старте события `event_datetime`; (б) за X часов
до (отдельный дедлайн подтверждения < 24ч окна). PRD говорит «за 24ч запускается этап 2», но не даёт
явного дедлайна окончания подтверждения. **Решить продуктово.**

## 4. «5 событий подтвердил — всё ещё Новичок» (объяснение, НЕ баг)

Репутация начисляется **только после финализации** (§4.4.4): событие прошло → организатор отметил явку
→ **48ч** окно оспаривания → finalize → reputation-пайплайн. Ничего из сделанного сегодня на staging
**не финализировано** → нет ledger-строк → `outcome_count=0` → «Новичок». Плюс:
- **Анти-фарм правило 1:** в СВОИХ клубах юзер не копит вовсе (если тестит как владелец).
- **Порог 3 исхода** (P1a): даже при паре финализированных — «Новичок» до 3.
- **Проблема тестирования:** 48ч окно (`AttendanceService.DISPUTE_WINDOW_HOURS=48`) делает e2e-тест
  репутации на staging непрактичным. Бэкфилл (V18) уже пересчитал историю; для живого теста — либо
  временно уменьшить окно на staging, либо принять, что калькуляция покрыта тестами (14 reputation-тестов).
