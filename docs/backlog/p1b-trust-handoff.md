## 2. Что P1b строит (scope, из §12 design-дока)

1. **Trust 0–100** — per-club + **глобальный по ВСЕЙ истории** (вкл. покинутые клубы, §7.1).
   Композит осей: явка 0.6 / финансы 0.4 (старт), recency-decay, асимметрия «намерение↔действие».
   Основной показ глобального — «надёжен в N из M клубов», число — вспомогательное.
2. **«Право на ошибку»** = байесовский оптимистичный приор: `Trust = (kept + K·prior)/(total + K)`,
   prior ≈ 0.85, K ≈ 3 (internal). Новичок стартует ~85, одна ошибка слегка просаживает, паттерн роняет.
3. **XP / уровни / бейджи** (геймификация аккаунта, глобальный, копится).
4. **Блок организатора** — прозрачная статистика (провёл N событий · явка % · возвраты % · отзывчивость),
   НЕ вычисляемый балл в MVP (§8). Опора на `clubs.activity_rating`.
5. **Visibility-тиры** (§9.1) — поле `visibility: self | others | internal` у типов сигнала; в UI
   маленькое мотивирующее подмножество, богатая модель внутри. Закрывает F5-19 (грубая раскраска тиров).
6. **Выход с обязательствами** (§7.1) + **UI «История клубов»** в профиле (см. §3).

## 3. Что УЖЕ залочено — НЕ переобсуждать

- **P1a реализация** (ledger/идемпотентность/rule 1/«Новичок») — в проде, фундамент финален.
- **Числа явки:** прогул/зритель = **−200**, приход = **+100** (`ReputationPolicy`, решение 2026-06-11).
  Складчина: оплата +10, молчание −40, отказ/released = 0 без строки.
- **§7.1 (решения 2026-06-13):**
  - глобальная репутация — **по всем клубам с историей** (вкл. покинутые), не только активным;
    текущий active-only-средний на ProfilePage — заглушка P1a, заменить агрегатом по ledger;
  - **выход с обязательствами:** предупреждение «потеряете {sum}» → подтверждение → в одной txn
    штрафные ledger-строки + удаление из событий/сборов + cancel membership + recompute.
    Цена: confirmed-событие = **−200** (+ промоут листа ожидания), реп-складчина pending = **−40**.
    Идемпотентность `UNIQUE(user_id, source_type, source_id)`. Paid-клуб не списывает сразу
    (доступ до конца подписки);
  - **UI «История клубов»:** покинутые клубы — отдельной секцией в профиле с их репутацией; при
    повторном входе клуб переезжает из «Истории» в активный список (две поверхности; делает
    all-history-агрегат прозрачным).
- **Анти-фарм правило 1** (владелец не копит в своём клубе) — live + backfill, действует.

## 4. Открытые продуктовые решения — снять В ДИЗАЙН-СЕССИИ (расширяет §11)

- **Точные веса/константы:** оси (старт 0.6/0.4), период полураспада decay, пороги уровней XP,
  K (мин. разных оплативших для финансового доверия), множитель асимметрии (старт ×2), потолок
  вклада одного клуба в глобал. Все — internal (§9.1), но числа надо зафиксировать.
- **Формула глобального Trust из ledger:** как именно агрегировать per-club → глобал (байес + диверсити-вес),
  устойчивость к выбросам, форма показа («N из M» + число).
- **Формула per-club Trust:** маппинг Σ ledger (неограниченная) → шкала 0–100 с приором/decay.
- **Блок организатора:** точный состав фактов в MVP, где показывать (профиль + дискавери), per-club vs роллап.
- **Выход-с-обязательствами:** сервис перечисления/суммирования обязательств (готовые SQL — в
  facts-report [[project_reputation_v2_design]]), UX-поток модалки, точные тексты.
- **«Новичок» / нет данных:** финальная формулировка и поведение в UI (не «0/100»).
- **Миграция:** нужен ли бэкфилл/пересчёт `user_club_reputation` под новую Trust-формулу (ledger полон,
  данные не трогаем — только формула вывода).

## 5. Подводные камни / не сломать

- **Гейт PR-2 (перенос владения клуба):** анти-фарм rule 1 завязан на текущий `owner_id` → re-backfill
  после трансфера переатрибутирует историю. Заморозить owner на момент записи ДО мержа club-leave PR-2
  (есть RED-характеризация в `ReputationLedgerIntegrationTest`). См. [[project_club_leave_epic]].
- **Ledger — append-only.** Trust-формула выводит число ИЗ ledger; саму историю не переписывать.
  Decay читает `occurred_at` (время поведения), не время обработки.
- **Visibility:** internal-сигналы (диверсити/confidence, константы, сырой минус, фрод-флаги, ranking)
  не должны протечь в API/DTO — поле `visibility` обязательно.
- **Данные пакета 1 F5 теперь чистые** (окно от marked_at, терминальность спора, TOCTOU-гарды) —
  строить Trust поверх них, не воспроизводить старые допущения о грязной явке.
- Дыра B (free-выход мимо штрафа) живёт до P1b — закрывается выходом-с-обязательствами (§3).
