## 1. Что нашли в коде — ДВА бага (симптом пользователя)

### Баг A — голодание пайплайна (это и есть «не ведётся статистика»)
Цепочка: `отметка явки → attendance_marked=true → (48ч) finalizeAttendance →
attendance_finalized=true → (раз в час) ReputationService → пересчёт`.
Первое звено (UI отметки явки) было **потеряно** при унификации создания активностей →
ни одно событие не получало `attendance_marked` → `findFinalizedEvents()` всегда пуст →
репутация никогда не двигалась. **Чинит фикс #2** (`bugfix/attendance-marking-ui`).

### Баг B — идемпотентность (критический, латентный; выстрелит, как только A заработает)
`ReputationService.processReputationForFinalizedEvents()`
([ReputationService.kt:38](../../backend/src/main/kotlin/com/clubs/reputation/ReputationService.kt))
раз в час берёт **все** finalized-события
([`findFinalizedEvents()` без фильтра «обработано»](../../backend/src/main/kotlin/com/clubs/reputation/JooqReputationRepository.kt))
и применяет дельты **аддитивно** (`applyDeltas`: `base + delta`, счётчики `+1`).
→ одно событие переначисляется **каждый час бесконечно**; за сутки +100 станет +2400, счётчики раздуются.
**Решение в редизайне — структурное (ledger), не заплатка** (см. §4).

### Сопутствующие расхождения
- **Расцепленные шедулеры:** `AttendanceService.finalizeAttendance` (1ч) и `ReputationService` (1ч)
  не связаны событием → лаг до 60 мин против PRD «пересчёт сразу после закрытия окна оспаривания».
- **default 0 vs 100:** таблица/сервис стартуют c `reliability_index=0`, а membership-репозиторий
  показывает новичку `COALESCE → 100`. Решение: новичок = **«нет данных / —»**, не 0 и не 100.
- **«Зритель» −20** (maybe→confirmed→absent) есть в коде (ReputationService.computeDeltas),
  но **отсутствует** в извлечённой таблице PRD §4.4.4 — сверить код↔PRD.

## 3. Продуктовые цели (пожелания пользователя — обязательны)

1. **Повышать качество UX**, отсеивать плохих участников, удерживать хороших.
2. **Стимул держать слово**, даже для слабо-обязанных (voluntary-сборы, maybe-голоса, бесплатные клубы).
3. **Геймификация** — чтобы хотелось «раскручивать» аккаунт.
4. **Анти-фарм** — нельзя накрутить себе репутацию (см. §6, ключевой кейс пользователя).
5. **UX-простота** — репутация **не усложняет** пользование, **без перегруза информацией**;
   прогрессивное раскрытие (по умолчанию один простой сигнал, детали — по тапу).
6. (Производное) **Справедливость/прозрачность** — понятно «почему такое число»,
   не наказывать за неконтролируемое, есть путь восстановления.

## 4. Архитектура (РЕШЕНО)

### Фундамент: append-only ledger
**`reputation_ledger`**: `user_id, club_id, axis, kind, points, occurred_at, source_type, source_id`.
- **Идемпотентность by design:** уникальный ключ `(source_type, source_id, kind)` → одно
  событие/сбор начисляется ровно раз. **Баг B исчезает структурно.**
- **Прозрачность (цель 6):** «почему» = развернуть в список начислений.
- **Decay/recency (цель 1 — удержание):** есть `occurred_at` → штрафы гаснут со временем.
- Агрегаты (`user_club_reputation`) становятся **кэшем**, пересчитываемым из ledger.
- **Событийная связь** вместо расцепленных шедулеров: `AttendanceFinalizedEvent` → listener
  пишет в ledger (poll оставить как backfill).

### Две разные величины (ключ к «и геймификация, и отсев»)
| | **XP / Уровень** | **Trust 0–100** |
|---|---|---|
| Природа | копится, почти не падает | **доля** сдержанных слов, recency-взвешенная |
| Цель | геймификация, «раскрутить аккаунт» | отсев плохих + справедливость + прощение |
| Падает | нет (но стрик сгорает) | да, восстанавливается поведением |
| Scope | **глобальный** (на аккаунт) | per-club + **глобальный агрегат** |

### Под-оси (хранятся раздельно, считаются из ledger)
- **Явка** (события) — таблица PRD §4.4.4 (см. §2).
- **Финансы** (сборы) — см. §5.
- **Коммуникация** — опционально, позже.
- **Композитный Trust** = взвешенный блендинг (старт `0.6·явка + 0.4·финансы`, tunable;
  финансы обнуляются без подтверждённых оплат).

## 6. Анти-фарм / Sybil (РЕШЕНО) — ключевой кейс пользователя

> Кейс: создал клуб → создал сбор → сам оплатил → сам подтвердил → +репутация.
> **Та же дыра в явке:** создал событие → записался → сам отметил «пришёл».

**Правило 1 — нельзя начислить себе (судья ≠ выгодоприобретатель).**
Владелец/организатор **не копит репутацию в своём клубе** (ни события, ни сборы) — он
адъюдикатор. Закрывает оба соло-сценария. На ledger: запись отбрасывается, если
`actor==beneficiary` или `beneficiary==club.owner`. (Со-организаторы позже — исключаем подтвердившего.)
*Естественно: репутация в своём клубе самореферентна. Свою копишь в чужих клубах.*

**Правило 2 — диверсификация контрагентов (анти-Sybil).**
Вклад с одного клуба/организатора — **с потолком**; основная масса репутации должна идти из
разных независимых источников. Репутация из одного клуба = низкая уверенность (флаг организатору).
*Этот же механизм — основа глобального агрегата (см. §7).*

**Правило 3 — honor-system деньги структурно слабее.** Плюс только при
`подтверждено орг. И организатор≠плательщик`; сбор начисляет, только если **≥K разных
подтверждённо-оплативших** (не один создатель); вклад с потолком.

**Правило 4 — вырожденные активности не считаются** (событие/сбор, где единственный
«успешный» участник — создатель).

Headline: **«репутацию нельзя выписать себе»** = владелец не копит в своём клубе +
подтвердивший ≠ получатель + потолок на контрагента + минимум разных оплативших.

## 7. Q1 — per-club vs глобальная (РЕШЕНО)

Не «или-или». **Один слой правды + производный взгляд:**
- **Per-club — источник истины** (поведение происходит в клубах; честно к контексту:
  хорош в одном, плох в другом — так и видно).
- **Глобальная — НЕ второе мутируемое число, а агрегат над per-club**, взвешенный по:
  **разнообразию контрагентов**, объёму, свежести; **устойчивый к выбросам**
  (основной показ — **«надёжен в N из M клубов»**, число — вспомогательное; один плохой клуб
  или недобросовестный организатор не обнуляет профиль везде).
- **XP/уровень — глобальный** (геймификация аккаунта); **Trust** — два слоя (по клубу + общий).

Почему честнее и реальнее: вес одного источника ограничен (нельзя «отменить» из одного клуба),
портативный сигнал «держит слово» возникает из паттерна по многим независимым контекстам —
как доверие в жизни (консистентность через много источников, а не один слух). Бонус:
«разнообразие» = тот же анти-Sybil рычаг (§6 правило 2) — один механизм на справедливость и фарм.

## 8. Q3 — рейтинг организатора (РЕШЕНО)

Организатор играет **две роли** → нужны **обе** вещи:
- **Как гость/заявитель в чужой клуб** → обычная **участницкая репутация** (копит в *других*
  клубах; в своём не копит по §6). Тут он как все.
- **Как хозяин своего клуба** → **отдельный организаторский блок** про *результат*, а не про
  вес слова (участницкая репутация это не измеряет, и у «чистого» организатора она пустая).

**MVP: организаторский блок = прозрачная статистика, НЕ вычисляемый балл:**
> «провёл N событий · средняя явка 78% · 60% возвращаются · отвечает на заявки за ~4ч»

Никакой непрозрачной формулы, нечего накручивать-как-балл, участник судит сам. Вычисляемый
«Рейтинг организатора»/уровень — **отложен** (свернём факты позже). Опора на
`clubs.activity_rating` (PRD §5.1, §10).

**Сигналы (outcome-based, считать когда дойдём до вычисляемого):** спрос (доля
проголосовавших/подтвердивших, заполняемость), доставка (show-rate его событий), сплочённость
(повторная явка, удержание), поведение (отзывчивость, нет простоя, мало подтверждённых споров).

**Анти-накрутка организатора** (он сам отмечает явку → мог бы фармить): опираться на сигналы,
которые он **не контролирует** — добровольные голоса участников, повторное поведение во времени,
споры участников, разнообразие реальных участников.

**Показ:** на профиле два разных блока — «Участник» и (если организатор) «Организатор»;
организаторский — в дискавери клубов (помогает выбрать, к кому идти).

> Тонкость: show-rate события — **одни данные под двумя линзами**: для участника «сдержал ли ТЫ
> слово», для организатора «пришли ли люди на ТВОЁ событие (в среднем)».

## 9. UX-простота (РЕШЕНО — пожелание пользователя)

**Прогрессивное раскрытие.** По умолчанию — **один простой сигнал** (уровень/бейдж/одна цифра),
детали (оси, разбивка, «почему», статистика организатора) — по тапу. Репутация **не добавляет
трений** в core-флоу (голосование, оплата, вступление). Не показывать одновременно 3 метрики +
2 оси + организаторскую статистику. Геймификация подаётся как «приятный прогресс», не как
бухгалтерия.

## 9.1. Показываемые vs внутренние сигналы — visibility tiers (РЕШЕНО)

**Разделяем «какие сигналы показываем» и «какие используем».** Богатую модель (много осей,
веса, confidence, фрод-сигналы) держим **внутри** для ранжирования/матчинга/гейтинга, а в UI
выносим **маленькое, мотивирующее, трудно-геймируемое** подмножество. Это снимает конфликт
«качество ранжирования ↔ UX-простота» и усиливает анти-фарм (что не видно — то труднее
оптимизировать).

Каждый параметр тегируется **уровнем видимости**:

- **`self`** (видит сам пользователь): уровень, бейджи, прогресс «до следующего», «почему»
  (разбивка начислений в позитивной рамке). Мотивация + достаточная прозрачность.
- **`others`** (видят организаторы/участники): грубый честный сигнал — уровень/бейдж,
  «надёжен в N из M клубов», организаторская статистика. Достаточно, чтобы решить, **мало**,
  чтобы точно нагеймить.
- **`internal`** (не показываем никому в UI): чистый материал для алгоритмов.

**Кандидаты в `internal`:**
- **Диверсити / Sybil-confidence** (насколько репутация «настоящая»: из скольких независимых
  клубов/организаторов) — драйвит вес в глобальном агрегате и ранжирование, но показ = подсказка
  «размажь себя тонко».
- **Точные константы формулы**: веса осей (0.6/0.4), множитель асимметрии (×2), период
  полураспада decay, потолки, K (минимум разных оплативших). Никогда не показывать.
- **Сырой отрицательный `reliability_index`** — наружу только уровень/бейдж (позитивная рамка),
  не «−180» (демотивация).
- **Анти-фарм флаги подозрительности** — показ типса́ет фармеров.
- **Ranking/match score для дискавери** (порядок клубов/рекомендаций) — composite, внутренний.
- **Провизорные начисления** (неподтверждённая оплата) — пользователю можно мягкий статус
  «ждёт подтверждения», но сами провизорные очки — internal, пока организатор не сверил.
- **Вычисляемый рейтинг организатора** (когда появится) — может быть internal-для-ранжирования
  (порядок в дискавери), а пользователь видит **факты-статистику** (см. §8).

**Принцип:** *signals used ≫ signals shown.* UI остаётся простым (один уровень + пара фактов),
модель внутри — богатой. На ledger/модели — поле `visibility: self | others | internal` у каждого
типа сигнала, чтобы это было невозможно случайно «протечь» в API/DTO.

  блок = **прозрачная статистика** в MVP, вычисляемый рейтинг отложен.
- XP/уровень — глобальный; Trust — per-club + глобальный.
- Анти-фарм — 4 правила (§6), строгий вариант: владелец **полностью** не копит в своём клубе.
- UX — прогрессивное раскрытие, минимальная поверхность по умолчанию.
- **Visibility tiers** (§9.1): часть параметров — **только internal** (диверсити/Sybil-confidence,
  константы формулы, сырой минус, фрод-флаги, ranking score), в UI не показываются; *signals used ≫
  signals shown*. У каждого типа сигнала поле `visibility: self | others | internal`.
- Баги A/B — чиним **внутри редизайна** (ledger), не отдельным хотфиксом (явку в прод сейчас
  не выкатываем отдельно — она часть фикса #2).

## 11. Открытые «ручки» (докрутить в спеке)

- Форма глобального Trust: «N из M» как основной + число вспомогательным (склоняемся к этому).
- Организаторский рейтинг: считать per-club с роллапом в сводный по организатору (показ обоих).
- Точные числа: веса осей (старт 0.6/0.4), пороги уровней XP, период полураспада штрафов (decay),
  K (минимум разных оплативших), множитель асимметрии (старт ×2), потолок вклада одного клуба.
- «Зритель» −20: сверить код↔PRD, узаконить или убрать.
- Новичок: «нет данных / —» вместо 0/100 — финализировать формулировку и поведение в UI.
- Миграция текущих `user_club_reputation` на ledger-модель (бэкфилл из истории event_responses).

## 12. Фазы реализации (предложение)

1. **P1 — фундамент + баги:** `reputation_ledger`, событийная связь (`AttendanceFinalizedEvent`),

4. **P-org — организатор:** статистический блок + показ в дискавери.
5. **P4 — отсев (opt-in):** пороги/флаги для организатора (авто-одобрение, приоритет в waitlist).
