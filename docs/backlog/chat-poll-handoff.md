## Что это и зачем

Последний слайс чат-эпика (§4 стратегии закрывается целиком). Телеграмовский **нативный poll
«Идёшь?»** в группе для Этапа 1 двухэтапки: голос без открытия Mini App вообще. Стратегия §12
(market-analysis, «Этап 1 нативным poll'ом»): «голос без запуска бота, нулевое трение, снимает
половину cold start двухэтапки… внутренняя версия всей стратегии C». Там же: вводить «после
гейта, пересчитав базу метрики Этап1→Этап2» — упомянуть PO при старте.

## Технические рамки (проверить при старте, но с высокой уверенностью)

- `sendPoll` / `stopPoll` / `poll_answer` есть в нашей библиотеке (telegrambots 7.10) — raw HTTP
  не нужен (в отличие от тегов).
- Poll обязан быть **неанонимным** (`is_anonymous=false`) — иначе `poll_answer` (кто как
  проголосовал) не приходит.
- `poll_answer` приходит при нашем пустом `allowed_updates` (не в списке исключений Bot API —
  исключены только chat_member/message_reaction*). Проверить на staging первым делом.
- Отзыв голоса в poll = `poll_answer` с пустым `option_ids`.
- **Синхронизация ОДНОСТОРОННЯЯ (poll → приложение)**: Telegram не позволяет боту менять чужие
  голоса в poll. Голос в Mini App НЕ отразится в poll'е — источник правды по счётчикам остаётся
  живой закреп. Это надо явно проговорить с PO (возможное «расхождение» цифр poll vs закреп).
- Нужен мапинг `poll_id → event_id`: таблица V56 типа
  `event_chat_polls (event_id PK FK→events CASCADE, chat_id, message_id, poll_id UNIQUE, closed_at)`
  — паттерн `event_chat_pins` (V49).

## Скелет дизайна (заготовка для Analyst, НЕ утверждено PO)

- Пост poll'а при создании события (гейт: тумблер; см. вопросы), варианты = наши три:
  «Иду ✅ / Возможно 🤔 / Не иду ❌» (маппинг 1:1 в castVote).
- `poll_answer` → по telegram_id ищем user + membership: участник с доступом → `castVote`
  от его имени; чужой/должник → игнор (или DM? — вопрос PO).
- Закрытие: `stopPoll` при старте события и при отмене (паттерн close-прохода закрепа);
  Этап 2 (подтверждение мест) остаётся ТОЛЬКО в Mini App — poll его не покрывает
  (подтверждение = обязательство, там нужен наш экран с дедлайном/очередью).
- Учёт и flush — по образцу LivePinService (dirty-флаги не нужны: poll сам живой, бот его
  не редактирует).

## Открытые вопросы PO (задать ДО спеки)

1. **Poll vs живой закреп** — вместе или вместо? Дефолт-предложение: вместе (закреп — статус
   и кнопка для Этапа 2, poll — входной канал голосов Этапа 1); но это два поста на событие —
   спам-бюджет. Альтернатива: при включённом poll'е закреп без кнопки «Проголосовать».
2. Отдельный тумблер «Опрос в чате» или расширение «Живого закрепа»? (Паттерн эпика —
   отдельные тумблеры на соц-чувствительные фичи; poll неанонимный — все видят кто как голосует!)
3. Голос НЕ-участника клуба в poll'е: игнорировать молча / DM «вступи в клуб» (paттерн двери)?
4. Голос должника (frozen/expired): игнорировать? (доступа к событиям у него нет)
5. `castVote` бэкенда: сегодня требует «событие в Этапе 1»? Отзыв голоса после старта Этапа 2 —
   что делать с poll_answer, пришедшим поздно (глотать по 409?).
6. Неанонимность poll'а — ок для PO? (Иначе фича невозможна — анонимные не дают poll_answer.)

## Разное / хвосты сессии

- **requestWriteAccess НЕ сделан** (проверено grep'ом 2026-07-10) — «окошко „разрешить боту
  писать“» из стратегии, кандидат №1 по рычагу/усилию (фронт-онли, <1ч). PO выбрал сначала
  голосовалку; окошко — следующее.
- Авто-эскалация должников (§2) — в очереди после (PO спрашивал «сколько сидят в expired»:
  сейчас бессрочно, решает организатор).
- Шероховатость эпика (не блокер): деактивация клуба не отвязывает чат — бот остаётся в группе.
- Крон staging `MEMBERSHIP_EXPIRY_CRON` = `0 0 9 * * *` (Spring, 6 полей!). Правка env —
  через Coolify tinker (id записей 85/86), см. память reference-scheduler-cron-staging.
- jOOQ codegen: НЕ гнать против локальной `clubs` (стоит на V18!). Рецепт: одноразовый
  `docker run postgres:16-alpine` на :5433 → прогнать все миграции psql'ом по порядку →
  `DB_URL=jdbc:postgresql://localhost:5433/clubs DB_USER=clubs DB_PASSWORD=clubs ./gradlew generateJooq`.
- Тест-флейки: vitest иногда роняет один файл без падающих тестов (перегон лечит);
  testcontainers может словить «Ошибка при установке SSL-подсоединения» при параллельном
  докер-чурне — тоже перегон.
