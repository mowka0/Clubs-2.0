# Backend / UX: уникальность названия клуба

Обнаружено 2026-04-25 при тесте `feature/pre-design-rhf-wizard` на staging.

## Что сейчас

`POST /api/clubs` (CreateClubModal через `useCreateClubMutation`) принимает любое название, включая полный дубликат уже существующего клуба. Backend (`ClubService.createClub`) сохраняет — frontend получает success, навигация на `/clubs/<новый-id>/manage`. В каталоге появляется два клуба с одинаковым именем.

## Почему это вопрос

С точки зрения пользователя — не баг (клуб создался), но логически:
- Дубликаты в Discovery → пользователь не различает их без других сигналов (город / организатор / категория)
- Возможный SEO/спам-риск — кто-то регает «клон» популярного клуба для перехвата участников
- Нарушает принцип «название клуба — идентификатор для пользователя» (если такой принцип есть в PRD — проверить)

## Решение пока **не выбрано**

Пользователь зафиксировал: «не знаю какую логику хочу в данном сценарии реализовать».

### Варианты (для будущей дискуссии)

**A. Глобальная уникальность** — `UNIQUE(name)` constraint в БД. POST возвращает 409 «Клуб с таким названием уже есть». Самый строгий — потеряем легитимные кейсы вроде «Книжный клуб» в разных городах
- Backend: миграция Flyway + Bean Validation + 409 в exception handler
- Frontend: показать inline-ошибку под полем «Название» в wizard step 0 (паттерн как у RHF) + `haptic.notify('error')`

**B. Уникальность в пределах города** — `UNIQUE(name, city)`. Допускает «Книжный клуб» в Москве и СПб одновременно. Балансирует UX
- То же что A но composite constraint

**C. Soft-warning** — backend разрешает дубликаты, но возвращает в response флаг `nameConflict: true` если такое имя уже занято. Frontend показывает warning-toast «Клуб с таким названием уже есть в [городе]. Создать всё равно?» с подтверждением
- Не блокирует, информирует — UX-friendly но не решает спам-риск

**D. Никаких ограничений** — оставить как есть. Принять что Discovery должен показывать различающую инфу (organizer name, city, category), и пользователь сам разберётся
- 0 кода, но не решает спам

### Что нужно решить до фикса

1. PRD-уровень: имя клуба — глобальный/городской идентификатор или просто display label?
2. Поведение при существующем клубе с тем же именем у того же организатора — это явная ошибка («не успел создать», ретрай) или valid case?
3. UX: hard-error (блок submit) или soft-warning (подтверждение)?

## Когда фиксить

Не блокирует public release пока не появятся реальные дубликаты в проде. После выбора варианта — `bugfix/club-name-uniqueness`, ~1-2ч (вариант A/B) или ~2-3ч (вариант C).
