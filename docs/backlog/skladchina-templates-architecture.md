# Складчина — шаблоны: архитектура и матрица (design contract)

> **Статус:** дизайн согласован с PO (2026-06-16), реализация на `feature/skladchina-templates`
> (ветка от `feature/skladchina-phase-a`). Источник направления — `skladchina-2.0-roadmap.md`.
> Концептуальная основа — commitment-протокол (см. § «Модель обязательства»).
> Фаза A («упрощение ядра») — фундамент: её движок шаблоны **расширяют**, а не переписывают.

## Принцип

**Один движок + стратегия на шаблон.** Движок (конечный автомат обязательства +
персистентность + DM) один и про шаблоны не знает. Шаблон-специфику он спрашивает у
**стратегии**, которую достаёт из реестра по полю `template`. Шаблон = **маска поверх
одного движка**: он задаёт лишь решения на стадиях, а не свой движок.

Это **Strategy + Registry**. Отвергнутые альтернативы: god-сервис с `when(template)`
(ломает Open/Closed, shotgun surgery), сервис-на-шаблон (дублирует FSM ×5), наследование
(`principles.md`: композиция > наследование).

## Модель обязательства (commitment-протокол)

Каждый участник несёт **обязательство** `C(должник, кредитор, antecedent, consequent)`:
участник обязуется (кредитору — оргу/клубу), что **если** наступит antecedent, **то**
исполнит consequent (заплатит долю). Жизненный цикл 1:1 ложится на наши статусы:

| Состояние commitment | Наш статус участника |
|---|---|
| Conditional | `pending` |
| Detached/Active | `pending` у дедлайна / после порога |
| Satisfied | `paid` |
| Violated | `expired_no_response` |
| Released | `released` |
| Cancelled | `cancelled` (каскад удаления клуба) |

**Шаблон задаёт только две вещи поверх движка:** (1) **antecedent** — когда обязательство
становится обязательным (дедлайн / порог собран / явка отмечена); (2) **маппинг исхода**
в ledger (`outcomeOf` + метка `verified`).

## 8 стадий движка (универсальный скелет)

| # | Стадия | Что решает шаблон |
|---|---|---|
| 1 | **Рождение** (триггер) | вручную «+» / из события / из календаря |
| 2 | **Состав** | орг выбирает / весь клуб / open enrollment / из явки / все кроме X |
| 3 | **Сумма + пресет** | режим, дедлайн, важность, видимость, порог |
| 4 | **Ответ участника** | оплатил (1 тап / сумма) / отказ / pledge |
| 5 | **Проверка** | нет (self-declared) / орг сверяет (A-2) / якорь-явка / рельс оплаты |
| 6 | **Закрытие** | дедлайн / все ответили / порог / вручную |
| 7 | **Исполнение** | нет / куплено / забронировано / подарено / возврат |
| 8 | **Справедливость** | репутация ± / ростер=доступ / только позитив / ничего |

Стадии 1, 4, 5, 6, 7 — это **последовательность действий**, а не поля. Порядок и
«кто-когда» несут справедливость (напр. «состав из явки» убивает мобилизацию случайных).

## Матрица: шаблон × стадии

| Стадия | **custom** | **split_bill** | **gear** | **booking** | **birthday** |
|---|---|---|---|---|---|
| 1 Триггер | «+» вручную | из события / «+» → выбрать событие | «+» «На инвентарь» | «+» / из события (D) | календарь / «+» |
| 2 Состав | орг выбирает | **авто из явки** (attended) | **open enrollment** / весь клуб | open enrollment (pledge) | весь клуб **− именинник** |
| 3 Сумма/пресет | любой режим, важность по выбору | fixed_equal = чек ÷ attended; дедлайн +48ч; важность выкл | фикс-на-голову; «когда куплю» | фикс-на-голову + **порог**; дедлайн жёсткий | voluntary; скрыто от X |
| 4 Ответ | 1 тап / сумма / отказ | 1 тап | 1 тап (добровольно) | **pledge → платёж после порога** | сумма |
| 5 Проверка | нет / орг-отметка | **якорь = явка (verified)** | **орг сверяет банк → A-2** | орг-отметка / рельс (опц.) | нет |
| 6 Закрытие | дедлайн/вручную/все | дедлайн / все оплатили | вручную (орг едет покупать) | **порог собран** / дедлайн-провал | дедлайн / вручную |

## Формат исхода (мост к репутации — «шаг 2»)

Каждый шаблон маппит финальный статус в единый формат, движок пишет в `reputation_ledger`
(+ колонка `verified`):

```kotlin
data class CommitmentOutcome(
    val state: CommitmentState,   // Satisfied | Violated | Released
    val verified: Boolean,        // явка/рельс = true; самодекларация = false
    val kind: ReputationKind?,    // null = в репутацию не пишем
)
```

**Без метки `verified` с первого шаблона «шаг 2» (вывод фин-ответственности) невозможен.**
Оценка ответственности на шаге 2 = `WHERE verified = true`. Ожидаемый итог: твёрдый сигнал
только из **split_bill** (verified=явка), мягкий из **booking** (consented pledge),
**gear/birthday — ноль штрафного сигнала by design**.

## Архитектура — backend

```
com.clubs.skladchina
  SkladchinaService.kt          // ДВИЖОК: FSM/lifecycle, делегирует стратегии
  Skladchina(Controller|Repository|Mapper|Dto).kt
  template/
    SkladchinaTemplate.kt       // strategy interface (растёт инкрементально)
    SkladchinaTemplateRegistry.kt   // Map<type, strategy>, Spring собирает все бины
    CustomTemplate.kt           // = текущее поведение Фазы A
    SplitBillTemplate.kt  GearTemplate.kt  BookingTemplate.kt  BirthdayTemplate.kt
```

- Стратегии — stateless `@Component`, **только правила, не персистентность** (split инжектит
  `EventResponseRepository` для явки → возвращает решение, пишет движок/репо).
- Реестр через `List<SkladchinaTemplate>` (Spring auto-wire) → `associateBy { type }`.
- **Интерфейс растёт от split_bill**, не проектируется «8×5» вперёд (иначе куча no-op).

## Архитектура — frontend

```
features/skladchina/  (или components/skladchina/)
  templates/registry.ts     // template -> { CreateForm, DetailSections, labels, preset }
  templates/splitBill.tsx  gear.tsx  booking.tsx
  shared/                   // PeopleProgress, PayButton, OrgMarkRow, DeficitPanel (из Фазы A)
  SkladchinaPage.tsx        // тонкая оболочка: registry[template] → секции
```

- Тип данных — **discriminated union по `template`** (TS заставит обработать каждый).
- Примитивы Фазы A (people-progress, кнопка оплаты, орг-кнопки, панель «не хватает») →
  shared-компоненты, каждый шаблон собирает их по-своему.

### Точка входа — кнопка «+» (требование PO 2026-06-16)

Шаблон вызывается из **«+» глобального таба «Активности»** (`CreateActivityFlow`), не из
управления клубом. Поток расширяется:

```
тип [Событие | Сбор] → (если Сбор) ШАГ ШАБЛОНА [Разделить счёт | На инвентарь | Бронь |
  Подарок | Свой сбор] → клуб → форма шаблона
```

- `ActivityTypeOptions` / новый `SkladchinaTemplatePicker` — контент внутри того же
  единственного Modal (паттерн `CreateActivityFlow`, чтобы не ломать overlay).
- **Дополнительный контекстный вход для split_bill:** кнопка «🧾 Разделить счёт» на
  прошедшем событии прыгает в ту же форму split с предзаполненным `eventId` (минуя
  шаги тип/шаблон/клуб). «+»-вход остаётся как ручной fallback (там выбираешь событие сам).

## Модель данных

Одна таблица `skladchinas` (решение роадмапа) + `template` enum + **горстка nullable-колонок**
под отдельные шаблоны. **Не** таблица-на-шаблон, **не** JSON-блоб — набор мал и известен.

- `template skladchina_template NOT NULL DEFAULT 'custom'` (enum: custom/split_bill/gear/booking/birthday)
- `event_id UUID NULL REFERENCES events(id)` — split_bill (переиспользуется фазой D)
- `threshold_count INT NULL` — booking (порог пледжей; точная форма уточнится при booking)
- `reputation_ledger.verified BOOLEAN` — метка доверия исхода (мост к шагу 2)
- fulfillment («куплено/забронировано/подарено») — уточнится при gear/booking; кандидат:
  отдельный терминальный признак или статус. **Не вводим, пока gear не потребует.**

## Порядок реализации

1. **split_bill** — самый дешёвый (переиспользует A-1/A-2), самые чистые данные (якорь явки).
   Обкатывает интерфейс стратегии. = роадмап B-2.
2. **gear/касса** — боль безбилетника: open enrollment + ростер=доступ + «куплено».
3. **booking/резерв** — «социальное эскроу» (pledge → порог → платим). Глубже всего.
4. **birthday** — последним или выкинуть (Telegram Gifts отвергнуты PO; денежный сбор — мало пользы).

## Decline-with-approval (V28, split_bill)

Отказ — это **поведение шаблона** (`declinePolicy` в стратегии): `FREE` (custom/voluntary —
мгновенный бесплатный отказ, желаемое поведение) vs `REQUIRES_APPROVAL` (split_bill — благо уже
потреблено, бесплатный отказ = безбилетник). Commitment-смысл: **release контролирует кредитор**
(орг), а не должник.

- `REQUIRES_APPROVAL`: участник подаёт **заявку с обязательной причиной** (`request-decline`),
  остаётся `pending`; орг **одобряет** (`→ declined`) или **отклоняет** (`decline_rejected=true`,
  остаётся `pending` — должен оплатить, повторная заявка закрыта). Мгновенный `decline` для таких
  шаблонов заблокирован (400).
- Без нового статуса участника: открытая заявка = `pending` + `decline_requested_at != null`.
  Зеркалит спор по явке события (`dispute_terminal`). Миграция V28 (3 колонки).
- Эндпоинты: `POST /skladchinas/{id}/request-decline` (участник, `{reason}`),
  `POST /skladchinas/{id}/participants/{userId}/resolve-decline` (орг, `{approve}`).
- **Репутацию пока не трогает** (split важность выкл). На «шаге 2» одобренный отказ = release
  (нейтрально), отклонённый-и-неоплаченный = кандидат на штраф.
- Принуждение «оплатить» — **социальное, не техническое** (деньги офлайн): участник остаётся
  помеченным неоплатившим, путь отказа закрыт.

## Открытые вопросы

- Точная форма fulfillment-состояния (вводим на gear, не раньше).
- booking: порог по людям (`threshold_count`) или по сумме — решить при booking.
- Шаг 2 (репутация из шаблонов): после ≥2 шаблонов — какие `verified`-исходы реально
  питают фин-ответственность; судьба opt-in важного-сбора (−40) и A-2 (+10) из Фазы A.
