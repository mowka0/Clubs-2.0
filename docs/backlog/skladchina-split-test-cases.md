> Цель: системно проверить шаблон split_bill перед мержем трека шаблонов.
> Контекст реализации — `skladchina-templates-architecture.md`, состояние — `skladchina-templates-handoff.md`.
> Staging: https://staging.77-42-23-177.sslip.io, бот `@clubs_v2_test_bot`.
>
> **Легенда покрытия:**
> - ✅ — есть зелёный автотест (`SkladchinaControllerTest`), регресс ловится сам.
> - 🔶 — ручной staging-кейс (UI/UX/интеграция, автотестом не покрыто — основной фокус прогона).
> - ⚠️ — **пробел**: сценарий важен, автотеста нет → проверить руками И/ИЛИ дописать тест.
>
> **Сетап тестового клуба (минимум для ручного прогона):**
> орг (владелец клуба) + 3 активных участника (A, B, C) + 1 не-член клуба (X).
> Нужно прошедшее событие с отмеченной явкой. Деньги в приложении не ходят —
> «оплата» это самодекларация/отметка орга, реальных переводов не делаем.

## A. Создание и точки входа

| ID | Покр. | Сценарий | Шаги | Ожидаемо |
|---|---|---|---|---|
| A1 | 🔶 | Вход «из события» | На прошедшем событии с отмеченной явкой (≥2 пришедших) → «🧾 Разделить счёт» | Открывается форма split с предвыбранным событием (шаг выбора события пропущен). Заголовок = название события. Шапка «Пришли: N чел.» |
| A2 | 🔶 | Вход через «+» → шаблон | Глобальный таб «Активности» → «+» → Сбор → «Разделить счёт» → выбор клуба → форма | Сначала экран «Выберите событие» со списком прошедших событий клуба; после тапа — та же форма, что в A1 |
| A3 | 🔶 | Превью доли в форме | Ввести «Сумма чека» = 4000, пришедших 4 | Подпись «≈ по 1 000 ₽ с каждого (4 чел.)» обновляется на лету |
| A4 | ✅ | Успешное создание линкует событие | API: `POST /clubs/{id}/skladchinas` с `template=split_bill`, `eventId`, `totalGoalKopecks=90000`, 2 пришедших | 201; DTO: `template=split_bill`, `eventId` проставлен, `paymentMode=fixed_equal`, `totalGoalKopecks=90000`, `participantCount=2`, доли по 45000 |
| A5 | 🔶 | Дефолтные пресеты в UI | Открыть форму, ничего не меняя | Дедлайн по умолчанию +48ч; важность сбора **выкл** (тумблера нет/скрыт); название-плейсхолдер «Счёт: {событие}» |
| A6 | 🔶 | Редирект после создания | Заполнить и «Создать сбор» | Переход на `/skladchina/{id}` (replace), карточка сбора показывает ссылку на событие |
| A7 | 🔶 | Участники приходят DM | После создания | Каждому пришедшему активному участнику бот шлёт DM о новом сборе; не-членам и отсутствовавшим — нет |
| A8 | 🔶 | Заголовок по умолчанию | Оставить поле «Название» пустым | Сохраняется `Счёт: {название события}` (или «Счёт за событие», если событие без названия) |

| ID | Покр. | Сценарий | Ожидаемо |

## C. Математика долей (равное деление)

| ID | Покр. | Сценарий | Ожидаемо |
|---|---|---|---|
| C1 | ✅ | Ровное деление | 90000 / 2 → 45000 + 45000; 80000 / 2 → 40000 + 40000 |
| C2 | ⚠️ | Неровное деление — остаток последнему | Чек 100000 коп. (1000 ₽), 3 пришедших → 33333 + 33333 + **33334** (остаток последнему по сортировке userId). **Автотеста на остаток для split нет** — только ровные суммы. Проверить инвариант `сумма долей == чек` |
| C3 | ⚠️🔶 | Округление в UI vs точность на сервере | UI показывает «≈ по X ₽» округлённо, но сервер пишет точные копейки. Чек 1000 ₽ ÷ 3: UI «≈ 333 ₽», сервер 33333/33333/33334. `collectedKopecks` при полной оплате == `totalGoalKopecks` (без потери копеек — это и был баг округления A-1) |
| C4 | 🔶 | Сумма чека дробная | Ввести «4000,50» → 400050 коп.; проверить парсинг запятой/точки в `rubToKopecks` |

## E. Отказ с одобрением организатора (V28 — `REQUIRES_APPROVAL`)

| ID | Покр. | Сценарий | Ожидаемо |
|---|---|---|---|
| E1 | ✅ | Мгновенный `decline` заблокирован | `POST /skladchinas/{id}/decline` для split → 400 «Для этого сбора отказ оформляется заявкой с причиной» |
| E2 | 🔶 | UI: кнопка «Запросить отказ» | У участника split-сбора кнопка называется **«Запросить отказ»** (не «Отказаться»); тап → форма ввода причины (textarea, обязательно) |

| E5 | ✅ | Заявка → отклонение → must pay | Орг `resolve-decline {approve:false}` → `decline_rejected=true`, участник остаётся `pending`; повторная `request-decline` → 400 «Ваш отказ отклонён — нужно оплатить счёт» |
| E6 | 🔶 | UI состояния заявки у участника | После подачи: баннер «⏳ Запрос на отказ отправлен — ждём решения организатора», кнопка отказа скрыта. После отклонения: баннер «Запрос на отказ отклонён — нужно оплатить счёт» |
| E7 | 🔶 | UI у организатора | В списке участников у заявителя бейдж «Просит отказаться» (`rd-warn`), показана причина «‹текст›», кнопки «Одобрить отказ» / «Отклонить». Кнопки оплаты для него скрыты, пока заявка открыта |
| E8 | ✅ | Идемпотентность заявки | Повторный `request-decline` при открытой заявке → 200, состояние не меняется (не плодит) |
| E9 | ✅ | Resolve не-организатором | Участник B вызывает `resolve-decline` по заявке A → 403 |
| E10 | ✅ | `request-decline` на custom-сборе | Сбор без `REQUIRES_APPROVAL` → 400 «Этот сбор не поддерживает заявки на отказ» (политика именно split) |
| E11 | 🔶 | Заявка после оплаты | Участник уже `paid` → «Запросить отказ» недоступно; API `request-decline` → 400 «только до оплаты или ответа» |

| E13 | 🔶 | Resolve без открытой заявки | Орг `resolve-decline` по участнику без заявки → 400 «Нет открытой заявки на отказ» |
| E14 | 🔶 | Принуждение «оплатить» только социальное | После отклонения отказа участник остаётся `pending` (помечен неоплатившим), путь отказа закрыт — технически платить его не заставляем (деньги офлайн) |
