**Статус:** аудит (код не начат) · **Дата:** 2026-07-17 · **Метод:** 7 агентов-читателей по кластерам + синтез (59 находок / 55 экранов), выводы сверены с кодом вручную.

## Аудит пустых состояний Clubs 2.0 — сводный отчёт

## 1. Итог одним абзацем

Обследовано ~55 пустых/стартовых состояний по 6 кластерам фронта. Реальной работы требуют **19 экранов** (2×P0, 7×P1, 10×P2); остальные ~36 — либо структурно недостижимы (IMPOSSIBLE, ~20 шт., их проверили и заглушка не нужна), либо намеренно скрыты и это верно (условные секции «Мои клубы», лист ожидания и т.п.), либо уже эталонные (5 GOOD-образцов). **Самое больное — то, что видит новичок сразу после онбординга.** Дверь «Найти клубы» ведёт на каталог `/`, где в раннем/небольшом городе показывается вводящее в заблуждение «Клубы не найдены · измените фильтры» без единого CTA (**P0**). Дверь «Создать клуб» ведёт в свежесозданный клуб, где первый же экран — вкладка «Активности» — это голая серая строка «В клубе пока нет активностей» без роли и без действия (**P0**), а вкладка «Управление → Статистика» встречает владельца тревожными нулями («Вовлечённость 0%» красным + нудж «напомните о встрече», хотя встреч ещё не было) (**P1**). Таб «Активности» слеп к роли: организатору нового клуба говорит «Найди клубы в Поиске» вместо «Создай событие» (**P1**). Профиль прячет «Интересы» вместо приглашения их заполнить (**P1**). Отдельный системный риск (урок F5-20) — два экрана выдают **ошибку загрузки за пустоту**: каталог и «Мои клубы» при сбое сети рисуют «пусто» без «Повторить», причём у юзера клубы реально есть (**P1**). Вся работа — чистый фронт, бэкенд не нужен.

## 4. Единый примитив

`FeedEmpty.tsx` — правильная база, но текущего API `{ icon?, title, description, ctaLabel?, onCta? }` **не хватает** для трёх задач, всплывших в аудите: (1) вторая кнопка (у «Мои клубы»-empty две двери онбординга), (2) вариант «ошибка» с `role="alert"` и «Повторить» (нужен минимум на 4 экранах — иначе везде дублируется own error-верстка), (3) интеграция с подсветкой онбординга (`useHighlight` / `location.state.highlight`). Роль-развилка при этом остаётся **на стороне вызывающего** (передаём разные `title/description/cta` по `isManager`) — это правильно, компонент про роли знать не должен.

**Рекомендация:** продвинуть FeedEmpty в `shared` как `EmptyState` (или расширить на месте, если импортеров немного), с расширенным API:

```ts
type EmptyStateProps = {
  variant?: 'empty' | 'error';          // error → role="alert" + акцент на retry
  icon?: ReactNode;
  title: string;
  description?: string;
  primaryCta?: { label: string; onClick?: () => void; to?: string; highlight?: string };
  secondaryCta?: { label: string; onClick?: () => void; to?: string };
  skeleton?: ReactNode;                  // «скелет будущей раскладки» под заставкой
};
```

- `variant='error'` закрывает Каталог-ошибку, Мои-клубы-ошибку, ошибки Интересов/Уровня — единый вид «заголовок + описание + Повторить».
- `to` + `highlight` дают декларативную связь с онбордингом: `{ to: '/my-clubs', highlight: 'create-club' }` пробрасывается в `navigate(to, { state: { highlight } })` — так пустые состояния переиспользуют ровно те же две двери, что карусель первого входа.
- **Скелет раскладки (`skeleton`) нужен точечно, не везде:** каталог (2-3 rd-club-card с shimmer), «Мои клубы» (1-2 строки карточки клуба), Активности/События и Клуб/Активности (контур карточки события), Управление/Статистика (тускло-серые рычаги). Для профиля, приглашения, EventPage — скелет избыточен, там хватает текста + CTA.

Мигрировать точечно: сначала P0/P1 экраны на новый EmptyState, telegram-ui `<Placeholder>` (ClubActivitiesTab, EventPage:287) убрать заодно для консистентности с Banco-Plata.

## 5. План внедрения

Весь объём — **чистый фронтенд, бэкенд не требуется**: все данные для роль-развилок уже приходят (`useMyClubsQuery` знает про owner-роль, ClubPage знает `isManager`, EventsTab нужно лишь прокинуть уже существующий сигнал).
