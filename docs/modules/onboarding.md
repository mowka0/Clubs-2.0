# Онбординг (срез 1: первый вход)

**Статус:** спека · **Ветка:** `feature/onboarding` · **Создано:** 2026-07-13
**Мокап:** `docs/design/onboarding/mockups/onboarding-flow.html` (реальные бренд-токены)

## Цель

Человек впервые открыл мини-апп и не понимает, что это и зачем. Онбординг делает ровно две вещи:
**объясняет продукт** и **разводит по двум дверям** — ходить на чужие встречи или вести свой клуб.

Эпик из трёх срезов, здесь — первый:
1. **Первый вход** (этот документ): три слайда-карусель, без вопросов.
2. Новый организатор: чек-лист первых шагов (создать событие, привязать чат, позвать людей).
3. Новичок в клубе: правила клуба и объяснение взносов/репутации после вступления.

## Решения PO (залочены 2026-07-13, НЕ пересматривать)

1. **Никаких вопросов.** Город, категории, интересы **не спрашиваем**. Клубов в системе ещё нет — подбирать не из чего, и вопрос про интересы превращается в ритуал: человек платит временем и не получает ничего. Когда клубы появятся, подбор станет осмысленным — тогда и вернём (см. «Не входит»).
2. **Онбординг — карусель из 3 слайдов**, смахивается влево-вправо. Стрелки по краям — подсказка, что слайд листается.
3. **Кнопка живёт на своём слайде**, а не сваливается кучей на последнем: слайд участника заканчивается «Найти клубы в твоём городе», слайд организатора — «Создать клуб и пригласить друзей».
4. **Роль не спрашиваем.** Организатор — не порода человека, а тот, у кого есть клуб; один и тот же человек ведёт книжный клуб и ходит на пробежки. Обе двери показываем, выбор происходит действием, а не декларацией.
5. **Приглашённый онбординг не проходит вообще.** Пришёл по инвайт-ссылке — вступает в клуб как обычно, по существующей фиче инвайтов. Он пришёл к конкретным людям, а не искать клуб.
6. **Бэкфилла существующих пользователей нет** — реальных людей в проде ещё нет, заботиться не о ком.
7. **Тон — на «ты».** Слоган: «Найди своих для живых встреч!» · подзаголовок: «Наполни свою жизнь активностями!».

## Слайды

### Слайд 1 — что это и что даёт
Арт 🙌, слоган, подзаголовок и **четыре причины пользоваться**. Только то, что одинаково важно и участнику, и организатору — без углубления в механику:

| | |
|---|---|
| 📍 **Люди рядом, а не в переписке** | Клубы по интересам в твоём городе: пробежка, ужин, настолки, книги. |
| 🤝 **Приходят те, кто обещал** | У каждого репутация: видно, кто держит слово. Встречи не срываются. |
| 🗓 **Клуб не тонет в переписке** | Встречи, состав, деньги — в одном месте. Не листать чат, чтобы понять, что и когда. |
| ⚡ **Всё внутри Telegram** | Ничего не ставить, не регистрироваться, не заводить пароли. |

Кнопка — «Дальше». Дверей на этом слайде нет.

### Слайд 2 — путь участника
Арт 🎟, «Ходи на встречи, которые готовят другие. Вступил в клуб — приходишь. Организовывать ничего не нужно.» Три сильнейших преимущества для участника:

| | |
|---|---|
| 🔎 **Свои люди — в пару тапов** | Клубы твоего города по интересам: бег, настолки, книги, кухня. Не искать по чатам и не проситься «а можно к вам». |
| 📅 **Ничего не пропустишь** | Все встречи клуба в одном месте, напоминание придёт само. Видно, кто идёт — и стоит ли идти. |
| ⭐ **Репутация работает на тебя** | Ходишь и не подводишь — растёт надёжность. С ней берут в закрытые клубы и зовут первым, когда мест мало. |

- Главная кнопка: **«Найти клубы в твоём городе»** → Discovery (`/`).
- Второстепенная: «Хочу вести свой клуб →» (листает на слайд 3).

### Слайд 3 — путь организатора
Арт 🎪, «Или веди свой клуб. Душа компании и прирождённый организатор? Свои правила, свой состав, свой блэкджек.» Три сильнейших преимущества для организатора:

| | |
|---|---|
| 👋 **Зови друзей в один тап** | Приглашение уходит карточкой клуба от твоего имени — прямо в личку. Или ссылкой, кому удобнее. |
| 💬 **Клуб живёт в твоём чате** | Привяжи чат клуба — анонсы встреч и напоминания приходят туда сами. Ушёл из клуба — вышел из чата. |
| 💸 **Взносы без таблиц и неловкости** | Видно, кто оплатил, а кто нет — напомнит приложение, а не ты. Складчины считаются сами. |

Под кнопкой — честная сделка: «Платный клуб — **взносы твои**, платформа не берёт процент. Взамен — клуб держится на тебе».

- Главная кнопка: **«Создать клуб и пригласить друзей»** → форма создания клуба (`CreateClubModal`).
- Второстепенная: «← Сначала посмотрю клубы» (листает назад).

Любая из главных кнопок завершает онбординг.

## Модель данных

### Миграция V60 — одна колонка

```sql
ALTER TABLE users ADD COLUMN onboarded_at TIMESTAMPTZ;
```

**Почему явная колонка, а не производный признак** («нет клубов → показать онбординг»): производный признак ломается при ошибке запроса — данные не пришли ≠ данных нет, и человек получает онбординг поверх заполненного аккаунта (ровно баг F5-20). Признак должен быть фактом, а не догадкой. Плюс честная метрика: сколько людей дошло до конца.

## API

### `POST /api/users/me/onboarding`
Отмечает онбординг пройденным. Тела запроса нет — собирать нечего.

- Проставляет `onboarded_at = now()`, возвращает обновлённый `UserDto`.
- Повторный вызов уже онбординенным — 409 (защита от гонки двойного тапа).
- Только свой профиль (JWT), чужой `userId` не принимается.

### `UserDto.onboardedAt`
Новое поле — фронту нужно для гейта.

## Frontend

### Гейт
`Layout.tsx` уже держит auth-гейт: пока не авторизован — роуты не рендерятся. Онбординг встаёт **туда же**: `user.onboardedAt == null` → рендерится `OnboardingFlow` вместо `<Outlet/>`. **Не роут** — иначе его обходят навигацией.

**Важно:** `isPending`/`isError` запроса ≠ «онбординг не пройден». Пока данные не пришли — спиннер, не онбординг (урок F5-20).

**Исключение — инвайт:** если в launch params есть `startParam` (пришёл по приглашению), онбординг не показывается вовсе — человек идёт в клуб через существующий `DeepLinkHandler`.

### Компонент
`OnboardingFlow` — карусель: свайп (touch) + стрелки по краям + точки-индикатор. Три слайда с текстами выше. Главная кнопка слайда вызывает `POST /me/onboarding`, затем навигация: `/` для «найти клубы», открытие `CreateClubModal` для «создать клуб».

## Acceptance Criteria

**AC-1.** Новый пользователь (`onboardedAt == null`) после авторизации видит карусель онбординга, а не Discovery. Навигацией её не обойти.
**AC-2.** Карусель листается свайпом влево-вправо и стрелками; точки показывают текущий слайд.
**AC-3.** «Найти клубы в твоём городе» → онбординг помечен пройденным, человек на Discovery.
**AC-4.** «Создать клуб и пригласить друзей» → онбординг помечен пройденным, открыта форма создания клуба.
**AC-5.** Повторный вход онбординг **не** показывает.
**AC-6.** Пришедший по инвайту (`startParam`) онбординг не видит — попадает в клуб.
**AC-7.** Ошибка запроса профиля не показывает онбординг поверх заполненного аккаунта (регресс F5-20).
**AC-8.** Вопросов о городе, категориях и интересах в потоке нет.

## Нефункциональные

- **Безопасность:** `POST /me/onboarding` — только свой профиль; тела нет, валидировать нечего.
- **Логирование:** завершение онбординга — INFO с выбранной дверью (участник/организатор). Единственная метрика намерения, которая у нас есть.

## Осознанно НЕ входит

- **Сбор города, категорий и интересов.** Вернём, когда в системе появятся клубы и подбор станет осмысленным. Тогда же — таксономия «категория → интерес», разметка клубов интересами и эндпоинт рекомендаций. Проработанный дизайн этого лежит в истории ветки (коммиты `bda16be`…`947b3be`): миграция таксономии, затравка словаря на ~100 интересов, правила подбора. Не выбрасывать — переиспользовать.
- **Предложение заполнить интересы при первом заходе в профиль** — отдельная задача, когда интересы снова понадобятся.
- **DM-напоминания** — бот не может писать первым, нужен `requestWriteAccess` (отдельная задача из очереди работ).
- **Чек-лист организатора** и **приветствие новичка в клубе** — срезы 2 и 3.
