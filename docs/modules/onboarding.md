# Онбординг (срез 1: первый вход)

**Статус:** as-built (реализовано, ждёт проверки на staging) · **Ветка:** `feature/onboarding` · **Создано:** 2026-07-13 · **Сверено с кодом:** 2026-07-13
**Мокап:** `docs/design/onboarding/mockups/onboarding-flow.html` (реальные бренд-токены)
**Хэндофф:** `docs/backlog/onboarding-handoff.md`

## Цель

Человек впервые открыл мини-апп и не понимает, что это и зачем. Онбординг делает ровно две вещи:
**объясняет продукт** и **разводит по двум дверям** — ходить на чужие встречи или вести свой клуб.

Эпик из трёх срезов, здесь — первый:
1. **Первый вход** (этот документ): три слайда-карусель, без вопросов.
2. Новый организатор: чек-лист первых шагов (создать событие, привязать чат, позвать людей).
3. Новичок в клубе: правила клуба и объяснение взносов/репутации после вступления.

## Решения PO (залочены 2026-07-13, НЕ пересматривать)

1. **Никаких вопросов.** Город, категории, интересы **не спрашиваем**. Клубов в системе ещё нет — подбирать не из чего, и вопрос про интересы превращается в ритуал: человек платит временем и не получает ничего. Когда клубы появятся, подбор станет осмысленным — тогда и вернём (см. «Не входит»).
2. **Онбординг — карусель из 3 слайдов**, смахивается влево-вправо. Стрелки по краям — подсказка, что слайд листается.
3. **Кнопка живёт на своём слайде**, а не сваливается кучей на последнем: слайд участника заканчивается «Найти клубы в своём городе», слайд организатора — «Создать клуб и пригласить друзей».
4. **Роль не спрашиваем.** Организатор — не порода человека, а тот, у кого есть клуб; один и тот же человек ведёт книжный клуб и ходит на пробежки. Обе двери показываем, выбор происходит действием, а не декларацией.
5. **Инвайт важнее онбординга — карусель ОТКЛАДЫВАЕТСЯ, а не отменяется** (PO 2026-07-13). Пришёл по инвайт-ссылке — вступает в клуб как обычно, по существующей фиче инвайтов: он пришёл к конкретным людям, а не искать клуб. `onboarded_at` при этом **остаётся NULL**, поэтому при следующем ОБЫЧНОМ запуске (без `startParam`) карусель показывается: человек уже в клубе, но так и не узнал, что можно искать другие клубы и вести свой.
6. **Бэкфилла существующих пользователей нет** — реальных людей в проде ещё нет, заботиться не о ком.
7. **Тон — на «ты».** Подзаголовок первого экрана: «Наполни свою жизнь активностями!».
   Исключение — сам слоган: «**Объединяйтесь** в Clubs по интересам, чтобы встречаться вживую!» — здесь
   обращение к людям во множественном числе намеренно: объединяются не в одиночку.
8. **Название продукта — на первом экране, внутри слогана** (PO 2026-07-13). Набрано каллиграфией Allura.
   Шрифт самохостится **сабсетом ровно под слово «Clubs»** (1.4 КБ вместо 232 КБ) — другой текст этим
   начертанием набрать нельзя, пока не пересобран сабсет.

## Кому и когда показываем (сводка)

| Ситуация | Видит карусель? |
|---|---|
| Первый запуск, `onboarded_at = NULL` | **Да** — вместо всего приложения (гейт, не роут) |
| Запуск по любой deep-ссылке (`startParam` есть: инвайт, клуб, событие, складчина), `onboarded_at = NULL` | **Нет** — идёт по ссылке; карусель отложена до следующего обычного запуска |
| Следующий обычный запуск после инвайта, `onboarded_at` всё ещё NULL | **Да** |
| Закрыл мини-апп на 2-м слайде, не дойдя до кнопки | **Да** — при следующем запуске, с начала |
| `onboarded_at != NULL` | **Нет** — никогда, даже если вышел из всех клубов и вычистил профиль |
| Профиль ещё грузится / запрос упал | **Нет** — спиннер, не карусель (регресс F5-20) |

Единственное условие показа — **явный факт `onboarded_at = NULL`**. Никаких производных признаков вроде «нет клубов» или «пустой профиль»: производный признак ломается при ошибке запроса (данные не пришли ≠ данных нет) и выкатывает онбординг поверх обжитого аккаунта.

**Пройденным считается только по тапу главной кнопки слайда** («Найти клубы в своём городе» / «Создать клуб и пригласить друзей»). Кнопки «Пропустить» нет, поэтому бросивший на середине увидит карусель снова — он ведь так и не узнал, что здесь можно.

## Модель данных

### Миграция V60 — одна колонка

```sql
ALTER TABLE users ADD COLUMN onboarded_at TIMESTAMPTZ;
COMMENT ON COLUMN users.onboarded_at IS 'Когда пользователь завершил онбординг; NULL — ещё не проходил';
```

Индекс не заводим: колонку читают только по PK (`WHERE id = ?`), выборок «все неонбординженные» нет.
Бэкфилла нет намеренно (решение PO) — все существующие строки остаются с `NULL` и увидят карусель.

**Почему явная колонка, а не производный признак** («нет клубов → показать онбординг»): производный признак ломается при ошибке запроса — данные не пришли ≠ данных нет, и человек получает онбординг поверх заполненного аккаунта (ровно баг F5-20). Признак должен быть фактом, а не догадкой. Плюс честная метрика: сколько людей дошло до конца.

## API

### `POST /api/users/me/onboarding`
Отмечает онбординг пройденным.

Тело: `{ "door": "MEMBER" | "ORGANIZER" }` — та дверь, которой человек вышел из карусели.
Изначально спека объявляла запрос без тела («собирать нечего»), но это противоречило
нефункциональному требованию логировать выбранную дверь: без неё сервер её попросту не знает.
Дверь всегда известна — её задаёт нажатая кнопка, — поэтому поле обязательное.

- Проставляет `onboarded_at = now()`, возвращает обновлённый `UserDto`.
- **Дверь в БД не хранится**, уходит только в INFO-лог: это метрика намерения, а колонка под неё
  была бы данными без потребителя. Понадобится аналитика — берётся из логов.
- Условие `onboarded_at IS NULL` стоит **внутри `UPDATE`**, а не отдельной проверкой перед ним,
  поэтому двойной тап кнопки не создаёт гонки: строку обновит ровно один запрос, второй получит 409.
- Только свой профиль: `userId` берётся из JWT, в запросе его негде передать.

| Код | Когда |
|---|---|
| 200 | Отметили сейчас. Тело — обновлённый `UserDto` (с непустым `onboardedAt`) |
| 400 | `door` отсутствует или неизвестен (не `MEMBER`/`ORGANIZER`). `onboarded_at` не проставляется |
| 401 | Нет/невалидный JWT |
| 404 | Пользователя из токена нет в БД (различаем с 409, чтобы не отвечать «уже пройден» тому, кого нет) |
| 409 | Онбординг уже пройден. **Для фронта это не ошибка** — см. § «Нефункциональные» |

### `UserDto.onboardedAt`
Новое поле (`OffsetDateTime?` / ISO-строка на фронте) — фронту нужно для гейта. Приезжает и в ответе
`POST /api/auth/telegram` (`user`), и в `GET /api/users/me`. Смежные доки, где описан `UserDto`:
[`profile.md`](./profile.md) · [`frontend-stores.md`](./frontend-stores.md).

## Frontend

### Гейт
`Layout.tsx` уже держит auth-гейт: пока не авторизован — роуты не рендерятся. Онбординг встаёт **туда же**,
на `useAuthStore.user` (профиль приезжает вместе с токеном из `POST /api/auth/telegram`, отдельного запроса
для гейта нет): `user.onboardedAt == null` → рендерится `OnboardingFlow` вместо `<Outlet/>`. **Не роут** —
иначе его обходят навигацией.

Порядок веток в `Layout` (сверху вниз):

| Состояние | Что рендерится |
|---|---|
| `!isAuthenticated && error` | Плашка «Не удалось авторизоваться. Откройте приложение через Telegram.» |
| `!isAuthenticated` (login в полёте) | Спиннер |
| `!user` (профиля ещё нет) | Спиннер — **не** карусель |
| `user.onboardedAt == null` и нет `startParam` | `OnboardingFlow` вместо всего приложения |
| Иначе | `<Outlet/>` + док |

**Важно:** «данные не пришли» ≠ «онбординг не пройден». Пока профиля нет — спиннер, не онбординг (урок F5-20).

**Исключение — deep-link:** если в launch params есть `startParam`, онбординг откладывается — человек идёт по ссылке через существующий `DeepLinkHandler`. Правило намеренно шире, чем «только инвайт»: пришедший по ссылке на клуб, событие или складчину тоже шёл к конкретной вещи, а не знакомиться с продуктом. `onboarded_at` остаётся NULL, поэтому карусель покажется при следующем обычном запуске.

### Компонент
`OnboardingFlow` — карусель: свайп (touch) + стрелки по краям + точки-индикатор. Три слайда с текстами выше.

- **Свайп** засчитывается от сдвига ≥ **50px** (`SWIPE_THRESHOLD_PX`) — меньше считается дрожанием пальца.
- **Стрелки** контекстные: левая скрыта на первом слайде, правая — на последнем; правая пульсирует
  (подсказка «листается»), при `prefers-reduced-motion` пульсация выключена.
- **Главная кнопка** («дверь») вызывает `POST /api/users/me/onboarding`, и **только после ответа сервера** —
  навигацию. Оптимистично гасить карусель нельзя: упавший запрос оставил бы человека с пустым
  `onboarded_at`, но уже внутри приложения. На время запроса кнопка дизейблится и показывает «Секунду…».
- **Порядок шагов двери жёсткий и не переставляется** (баг, найденный PO на staging 2026-07-13):
  `ответ сервера → навигация → запись профиля в стор`. Профиль в сторе — это и есть гейт; положи его
  раньше навигации, и гейт откроется, карусель размонтируется, а TanStack **не вызовет** колбэки
  `mutate(...)` у наблюдателя без слушателей. Навигация тогда теряется целиком, вместе с меткой
  подсветки: обе двери молча оставляли человека там, где он стоял. Именно поэтому
  `useCompleteOnboardingMutation` **не пишет в стор сам** — это делает вызывающий, после `navigate`.
  Регресс закрыт тестами в `OnboardingGate.test.tsx` — они гоняют дверь **через гейт**; карусель,
  отрендеренная напрямую, не размонтируется и такой баг не ловит.
- **Запрос упал** (не 409) — тост «Не удалось продолжить. Проверьте связь и попробуйте ещё раз.»,
  человек остаётся в карусели.

Файлы: `components/onboarding/{OnboardingFlow,OnboardingSlide}.tsx` + `slides.ts` (тексты данными,
не JSX) · `hooks/useHighlight.ts` · `queries/profile.ts` (`useCompleteOnboardingMutation`) ·
`api/profile.ts` (`completeOnboarding`) · стили `styles/redesign.css` (`.ob-*`, `.rd-highlight-pulse`).

### Подсветка цели после онбординга (PO 2026-07-13)
Кнопка онбординга **не открывает форму сама** — она приводит человека туда, где эта форма живёт, и **показывает пальцем**, что нажать. Так человек с первого раза запоминает, где эта штука находится, и во второй раз найдёт её сам.

- «Создать клуб и пригласить друзей» → `navigate('/my-clubs', { state: { highlight: 'create-club' } })` → на «Моих клубах» кнопка «+ Клуб» на пару секунд получает пульсирующее акцентное кольцо.
- «Найти клубы в своём городе» → `navigate('/', { state: { highlight: 'city' } })` → в каталоге клубов так же подсвечивается селектор города.

Механика общая — хук `useHighlight(target)`: страница читает `location.state.highlight`, вешает CSS-класс
`rd-highlight-pulse` на свой элемент и снимает его через **2200 мс** (кольцо пульсирует 2 раза по 1.1s —
числа в хуке и CSS парные). Метка снимается из history сразу (`replace: true`), иначе возврат назад зажигал
бы кольцо снова; при этом **вычищается только ключ `highlight`**, остальной `state` сохраняется — на «Моих
клубах» рядом живут `state.toast` и deep-link `?focus=inbox`. При `prefers-reduced-motion` пульсация
заменяется статичной акцентной рамкой.

## Acceptance Criteria

**AC-1.** Новый пользователь (`onboardedAt == null`) после авторизации видит карусель онбординга, а не Discovery. Навигацией её не обойти.
**AC-2.** Карусель листается свайпом влево-вправо и стрелками; точки показывают текущий слайд.
**AC-3.** «Найти клубы в своём городе» → онбординг помечен пройденным, человек в каталоге клубов, селектор города подсвечен.
**AC-4.** «Создать клуб и пригласить друзей» → онбординг помечен пройденным, человек на «Моих клубах», кнопка «+ Клуб» подсвечена. Форма создания **не открывается сама** — см. § «Подсветка цели»: человек должен запомнить, где эта кнопка живёт.
**AC-5.** Повторный вход онбординг **не** показывает. Закрывший карусель на середине (не нажав главную кнопку) при следующем запуске видит её снова, с начала.
**AC-6.** Пришедший по инвайту (`startParam`) карусель не видит — попадает в клуб; при следующем обычном запуске (без `startParam`) карусель показывается.
**AC-7.** Пока профиль не приехал (или авторизация упала) — спиннер/плашка ошибки, а не онбординг поверх заполненного аккаунта (регресс F5-20).
**AC-8.** Вопросов о городе, категориях и интересах в потоке нет.
**AC-9.** Упавший запрос двери (не 409) оставляет человека в карусели с тостом об ошибке — приложение не открывается «наполовину онбординженным».

## Нефункциональные

- **Безопасность:** `POST /me/onboarding` — только свой профиль (`userId` из JWT, в запросе его негде передать). Тело валидируется: `door` — обязательный enum, неизвестное значение → 400. Онбординг — **UX-гейт, а не граница безопасности**: он ничего не защищает, каждый реальный эндпоинт независимо закрыт JWT. Обойти карусель на клиенте (подсунув `startParam`) можно, ценой обхода будет лишь непоказанная карусель.
- **409 на фронте — не ошибка:** «уже пройден» означает, что цель достигнута (например, онбординг завершён на другом устройстве, а профиль в этой сессии устарел). Фронт перечитывает профиль и продолжает навигацию, а не запирает человека в карусели.
- **Логирование:** завершение онбординга — INFO с выбранной дверью (участник/организатор). Единственная метрика намерения, которая у нас есть.

## Осознанно НЕ входит

- **Сбор города, категорий и интересов.** Вернём, когда в системе появятся клубы и подбор станет осмысленным. Тогда же — таксономия «категория → интерес», разметка клубов интересами и эндпоинт рекомендаций. Проработанный дизайн этого лежит в истории ветки (коммиты `bda16be`…`947b3be`): миграция таксономии, затравка словаря на ~100 интересов, правила подбора. Не выбрасывать — переиспользовать.
- **Предложение заполнить интересы при первом заходе в профиль** — отдельная задача, когда интересы снова понадобятся.
- **DM-напоминания** — бот не может писать первым, нужен `requestWriteAccess` (отдельная задача из очереди работ).
- **Чек-лист организатора** и **приветствие новичка в клубе** — срезы 2 и 3.
