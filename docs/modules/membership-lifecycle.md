# Membership lifecycle — статусная модель членства (де-Stars, honor-system)

**Статус:** спека фичи `feature/expired-status-revival` (решение PO 2026-07-06,
источник: `docs/backlog/freeze-flow-rethink.md` § «✅ РЕШЕНИЕ PO»).
Здесь же живёт каноническая **матрица status×surface**, на которую ссылается
`MembershipAccess.kt` (раньше ссылка вела в `payment-v2.md`, где матрицы не было — drift закрыт).

## 1. Статусная модель (enum `membership_status`, V3 + V37)

| Статус | Смысл (целевая модель 2026-07-06) | Кто пишет |
|---|---|---|
| `active` | Состоит в клубе, доступ к контенту открыт | join бесплатного клуба; «Взнос получен» (markDuesPaid); «Своя дата» (setAccessUntil) |
| `frozen` | **Единственный смысл: новый платный участник ждёт ПЕРВОГО взноса.** Состоит, занимает слот, доступа нет | вступление/одобрение в платный клуб (createFrozen / reactivateFrozen); легаси-эндпоинт `/freeze` (с UI не вызывается) |
| `expired` | **Просрочено продление**: доступ был, оплаченное окно истекло. Состоит, занимает слот, доступа нет — «должник» | шедулер `expireOverdueAccess` (ежедневно 9:00): `active` + `subscription_expires_at <= now` → `expired`; V46 (разовая): легаси frozen-должники |
| `cancelled` | Вышел / удалён / отказано — не состоит | leave, kick (remove), reject |
| `grace_period` | **Мёртвый легаси Stars.** Не пишется и не читается кодом (уборка выполнена этой же веткой); застрявшие строки флипнуты в `expired` (V46); из PG-типа не удалён — Postgres не умеет DROP VALUE | никто |

Путь возврата из `expired` и `frozen` одинаковый (honor-system): участник платит организатору
вне платформы → «Оплатить взнос» (claimDues) → организатор «Взнос получен» (markDuesPaid) → `active`
(+30 дн от максимума(текущий срок, сейчас)). `expired` НЕ возвращается через повторную заявку —
он всё ещё участник (`findActiveByUserAndClub` его находит, join/apply отвечают «Already a member»).

## 3. Переходы и события `membership_history` (V31)

| Переход | Событие |
|---|---|
| ∅ → active/frozen (вступление) | `joined` |
| cancelled → active/frozen (повторное вступление) | `rejoined` |
| active/frozen/expired → cancelled (выход/кик/отказ) | `left` |
| active → expired (шедулер, просрочка продления) | **не логируется** |
| expired → active («Взнос получен» / «Своя дата») | **не логируется** |

**Решение (2026-07-06, зафиксировано этой фичей): лапс продления НЕ логируется как churn.**
Обоснование: `expired`-участник остаётся в ростере должником и обычно возвращается за дни;
club-quality (`JooqClubRankRepository`, `JooqClubStatsRepository`) считает события `left`+`expired`
оттоком — лог на каждый лапс штрафовал бы ранг клуба за транзиентные просрочки и давал бы двойной
счёт, когда должник потом реально уходит (`left`). Реальный отток фиксируется в момент выхода/кика.
Enum-значение `expired` в `membership_event` остаётся зарезервированным (историческая семантика V31).

## 4. Уведомления

- За 3 дня до истечения (active): DM-текст «истекает через 3 дня» — без кнопки (платить пока нечего:
  claim доступен только frozen/expired).
- В момент истечения (шедулер, ДО транзакции processExpiry): DM «доступ закрыт — подписка истекла»
  **с inline-кнопкой-диплинком «Оплатить взнос»** на `/clubs/{clubId}` (аналог `sendAccessFrozenDM`).
- `ExpiringSubscriptionNotification` расширяется полем `clubId` (для диплинка).

## 5. UI-поверхности

- **Таб «Участники» (организатор, бакеты):**
  - «⛔ Доступ истёк» — статусный бакет: `accessStatus === 'expired'` ИЛИ (safety-окно до тика
    шедулера) `active` с прошедшим `subscriptionExpiresAt`. Мета «истекла N дн назад»,
    бейдж «Оплата заявлена» при claim, действие «Взнос получен».
  - «💸 Оплата вступления» — только `frozen` (хинт «Вступили — подтвердите взнос» перестаёт врать).
- **Карточка участника (организатор):** ветка `expired` — статус «⛔ Доступ истёк · была до DATE»,
  claim-линия + скриншот как у frozen, действие «Взнос получен · открыть доступ». Без
  «Отказать · вернуть» (это только про первое вступление); «Удалить из клуба» доступен.
- **ClubPage (сам expired-участник):** аналог frozen-вида — «Подписка истекла, доступ закрыт»,
  CTA «Оплатить взнос» (claim-флоу, DuesPaymentSheet), после claim — «Оплата на проверке». Табов нет.
- **MyClubsPage (сам участник):** expired-членство попадает в блок «🔒 Доступ закрыт — оплатите»
  вместе с frozen (та же личная срочность, тот же путь).
- **MyClubsPage (организатор, кросс-клуб):** секция объединяет frozen+expired; заголовок
  «💸 Ждут оплаты»; мета строки: frozen → «вступил(а) …», expired → «истекла N дн назад».
- `accessStatus` в API: `'active' | 'frozen' | 'expired'` (организатору), null обычным зрителям.

## 6. Миграция V46 (данные)

Разовый флип легаси-должников: `frozen` + `subscription_expires_at IS NOT NULL AND <= now()` →
`expired`. Это строки, которые старый шедулер заморозил за просрочку (или орг заморозил вручную
уже после первого оплаченного окна) — в новой модели они «истёкшие», не «ждущие первого взноса».
`frozen` с NULL-окном (настоящее первое вступление) и с будущим окном (редкая ручная пауза
эпохи кнопки «Закрыть доступ») не трогаем. Плюс безусловный флип всех застрявших `grace_period`
строк → `expired` (семантика совпадает; тупик, который PO видел на «Американском клубе»).
+ `COMMENT ON` обновляет текст жизненного цикла V37.

## 7. Критерии приёмки

- AC-1: шедулер переводит `active` с истёкшим окном в `expired` (не `frozen`); UPDATED_AT обновлён;
  `access_frozen_at` не проставляется; `membership_history` не пишется.
- AC-2: `expired`-участник в «Мои клубы» видит клуб в блоке «Доступ закрыт — оплатите»; на ClubPage —
  вид «подписка истекла» с работающим claim-флоу; после claim — «Оплата на проверке».
- AC-3: организатор видит должника в статусном бакете «⛔ Доступ истёк» (стабильно, не только до тика);
  «Взнос получен» открывает доступ (+30 дн) и убирает из бакета; карточка участника показывает
  «Доступ истёк», claim и скриншот.
- AC-4: бейдж таб-бара «Мои клубы» загорается от claimed-`expired` так же, как от claimed-`frozen`;
  без claim — не загорается. (Red-dot на табе «Управление» удалён целиком — PO 2026-07-07.)
- AC-5: `expired` занимает слот (лимит участников), находится `findActiveByUserAndClub`
  (повторный join/apply → «Already a member»), не получает DM о событиях, не виден в ростере
  обычным участникам.
- AC-6: DM «подписка истекла» приходит с кнопкой-диплинком «Оплатить взнос» на страницу клуба.
- AC-7: выход/кик `expired`-участника логирует `left`; сам флип active→expired ничего не логирует;
  club-quality churn-метрики не меняются от лапсов.
- AC-8: V46 перевела легаси frozen-должников (окно в прошлом) в `expired`; frozen без окна остались.
- AC-9: DM «истекает через 3 дня» — прежнее поведение (текст без кнопки).
