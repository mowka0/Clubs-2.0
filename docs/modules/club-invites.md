# Приглашения в клуб + механика полного клуба (club-invites)

> Статус: спека утверждённого дизайна. Решения PO зафиксированы 2026-07-11,
> мокапы: `docs/design/club-invites/mockups/01-invite-flow.html` (путь инвайта, кадры A–E)
> и `02-full-club.html` (полный клуб, кадры F–I).
>
> **Пререквизит:** апгрейд `telegrambots` 7.10.0 → 9.5.0 (ветка `chore/telegrambots-9.5`,
> отдельный staging-прогон до фичи). Даёт prepared messages (Bot API 8.0) для карточки-приглашения
> и типизированные методы Member Tags (сносим `rawApiCall` в `ChatTelegramGateway`).

## Зачем

Организатор зовёт людей в клуб лично: выбирает получателей в **нативном пикере Telegram**,
приглашение уходит **от его имени** (не от бота — бот не может писать незнакомцам и не видит
контакты). Получатель открывает карточку клуба в Mini App и вступает в один тап.
Вторая половина — механика полного клуба: «хочу вступить, а мест нет» превращается в заявку,
которую орг разруливает из инбокса, расширяя клуб.

## Решения PO (2026-07-11)

1. **Вступление по инвайту — сразу член** (существующий `joinByInvite`), без заявки.
   В платном клубе «сразу член» = `frozen` до подтверждения взноса — как везде.
2. **Клуб полон** — инвайт и витрина деградируют в обычную заявку («Попроситься в клуб»).
   В инбоксе орга заявки полного клуба собираются в один блок: крестик отсеивает лишних
   (обычный отказ с причиной), поле нового лимита (подсказка — текущий лимит числом),
   кнопка **«Расширить клуб и принять всех (N)»** — атомарно.
3. **Платный клуб** — текущее поведение: цена видна на посадочной, после вступления/одобрения
   участник `frozen`, платит оргу напрямую, орг подтверждает взнос.
4. **Текст приглашения** — фиксированный шаблон, орг своё не дописывает.
5. **Кто приглашает** — владелец и роль `organizer`.
6. **Бейдж «по вашему приглашению»** — отклонён, заявки без пометки источника.
7. **Механика отправки — prepared message** (rich-карточка с кнопкой). Фолбэк для старых
   клиентов без `shareMessage` — «Скопировать ссылку».

## Ключевой инвариант (платные клубы)

**Оплата доступна только membership'у в статусе `frozen` — заявитель платить не может физически**
(экран взноса живёт в membership-флоу, заявка membership не создаёт). Поэтому сценарий
«оплатил, а заявку отклонили» исключён порядком шагов: заявка → одобрение → `frozen` → оплата.

Матрица флоу:

| Ситуация | Флоу |
|---|---|
| Открытый платный, есть места | вступил сразу → `frozen` → платит → орг подтвердил |
| По заявке (`closed`), есть места | заявка → approve → `frozen` → платит |
| Любой клуб, мест нет | «Попроситься» → блок в инбоксе → «Расширить и принять» → `frozen` → платит |
| Личный инвайт, есть места | сразу член (`frozen` в платном) → платит |
| Личный инвайт, мест нет | как «Попроситься» |

Лимит мест считает `active + frozen + expired` (существующий `countActiveByClubId`) — не меняем.

## Часть 1. Личные приглашения

### Инвайт-код: расширение на все типы клубов

Сейчас `clubs.invite_code` генерится только при создании `private`-клуба; у `open`/`closed` — NULL.
Новое: код нужен любому клубу — **ленивая генерация**: эндпоинт шаринга (см. ниже) создаёт код,
если его ещё нет (существующий `generateInviteCode()`). Миграция не нужна.
Семантика `private` не меняется (не виден в каталоге, вход только по ссылке).
Регенерация кода — существующий `POST /api/clubs/{id}/regenerate-invite` (owner only), отзывает
старую ссылку.

### Новый эндпоинт шаринга

```
POST /api/clubs/{id}/invite-share            (owner или organizer этого клуба)
→ 200 { "inviteUrl": "https://t.me/<bot>?startapp=invite_<code>",
        "preparedMessageId": "<id>" }
```

- Гарантирует наличие invite-кода (ленивая генерация).
- Вызывает Bot API `savePreparedInlineMessage` для `telegram_id` вызывающего:
  карточка-приглашение (InlineQueryResultArticle) с фиксированным текстом
  («<Имя орга> приглашает в клуб «<название>» · <категория> · <город>») и inline-кнопкой
  **«Открыть клуб»** → `inviteUrl`.
- `preparedMessageId` живёт недолго (TTL на стороне Telegram) — фронт запрашивает при каждом
  открытии шита, не кэширует.
- 403 — не owner/organizer; 404 — клуба нет.

### Фронт: организаторская половина

| Место | Что |
|---|---|
| `ClubPage`, таб «Участники» | строка-кнопка «＋ Пригласить в клуб» над ростером, видна owner/organizer (кадр A) |
| Боттом-шит (кадр B) | «Отправить в Telegram» → `shareMessage(preparedMessageId)`; «Скопировать ссылку» → `inviteUrl` в буфер. Если `shareMessage` недоступен на клиенте — остаётся только копирование |
| Успех создания клуба (кадр E) | CTA «Пригласить участников» → тот же шит |
| `DeepLinkHandler` | новый префикс `invite_<code>` → `/invite/<code>` (по образцу `club_`/`event_`) |

### Посадочная приглашённого

Существующая `/invite/:code` (`InvitePage`): карточка клуба + «Вступить в клуб» =
`POST /api/invite/{code}/join` (сразу член; `frozen` в платном — тексты уже есть).
Дополнение: `GET /api/invite/{code}` в `ClubDetailDto` добавляет имя владельца
(`ownerFirstName`, `ownerLastName`) — подпись «Приглашение от <имя>» (кадр D).
Если клуб полон — ветка «Попроситься» (часть 2, кадр G).

## Часть 2. Полный клуб

### Подача заявки в полный клуб

`submitApplication` сейчас требует `accessType == closed`. Ослабляем:

```
заявка принимается, если accessType == closed  ИЛИ  клуб полон (любой тип)
```

Всё остальное без изменений: дедуп активной заявки, rate limit 3/день, вопрос клуба
(для open-клуба вопроса обычно нет — `answerText` не требуется), DM оргу о новой заявке
(в текст добавляется «клуб полон» при полноте).

### Фронт: витрина и посадочная

- `ClubPage` (посетитель) и `InvitePage`: если `memberCount >= memberLimit` — баннер
  «Клуб сейчас полон» + CTA **«Попроситься в клуб»** → `submitApplication` (кадры F, G).
- После подачи — обычное состояние «заявка отправлена» (существующие «Мои заявки»).

### Инбокс орга: блок полного клуба (кадры H, I)

Группировка — **чисто фронтовая**: `useMyPendingApplicationsQuery` уже отдаёт заявки с
`ClubBriefDto`; заявки клубов, где `клуб полон`, собираются в блок по клубу
(нужно знать `memberCount`/`memberLimit` клуба — добавить оба поля в `ClubBriefDto`).

- Шапка блока: аватар + название + «N заявок» + пилюля «Клуб полон · X/X».
- Строка заявителя: имя + краткий peerStats; крестик (квадратный, r=9) = **существующий
  reject-флоу** с обязательной причиной и DM заявителю.
- Футер: поле «Новый лимит участников» (подсказка-плейсхолдер — текущий лимит числом),
  строка-калькулятор «мест X, заявок N — чтобы принять всех, укажите не меньше X+N»,
  кнопка «Расширить клуб и принять всех (N)».
- Заявки НЕполных клубов — отдельными карточками, как сейчас.
- После расширения блок рассыпается в обычные карточки автоматически (клуб больше не полон).

### Новый эндпоинт: расширить и принять атомарно

```
POST /api/clubs/{id}/expand-and-approve      (owner only — как approve сегодня)
{ "newMemberLimit": 23, "applicationIds": ["...", "..."] }
→ 200 { список ApplicationDto }
```

Одна транзакция:
1. owner-check; все заявки принадлежат клубу и в статусе `pending`;
2. `newMemberLimit > текущего лимита` и `newMemberLimit >= countActiveByClubId + applicationIds.size`;
3. обновить `member_limit`;
4. одобрить каждую заявку той же логикой, что `approveApplication`
   (платный клуб → `frozen`, бесплатный → `active`; DM «заявка одобрена / оплатите вступление»).

DM-ки шлются после коммита (best-effort, как в существующем approve).

## Вне скоупа (осознанно)

- Пометка источника заявки «по приглашению» (решение PO №6).
- «Принять всех» без расширения; батч-approve для неполных клубов.
- Nudge «в клубе 3 из 20 — пригласи своих» на странице клуба.
- Кастомный текст приглашения от орга.
- `requestWriteAccess` (отдельная задача, следующая по очереди).

## Критерии приёмки

1. Owner/organizer видит «Пригласить в клуб» в табе «Участники»; обычный участник — нет.
2. «Отправить в Telegram» открывает нативный пикер; получатель получает карточку от имени
   орга с кнопкой «Открыть клуб»; кнопка ведёт на `/invite/<code>` в Mini App.
3. «Скопировать ссылку» кладёт в буфер рабочий deep-link (открывается с холодного старта).
4. Вступление по инвайту: бесплатный клуб → сразу `active`; платный → `frozen` + существующий
   взнос-флоу; посадочная показывает цену и имя орга.
5. Инвайт-ссылка работает для `open`, `closed`, `private` клубов; регенерация отзывает старую.
6. Полный клуб (любой тип): витрина и посадочная показывают «полон» + «Попроситься»;
   заявка попадает в инбокс; НЕполный open-клуб заявку по-прежнему не принимает.
7. Инбокс: заявки полного клуба сгруппированы в блок; отказ по крестику работает как обычный
   отказ (причина + DM); «Расширить клуб и принять всех» поднимает лимит и принимает
   оставшихся одной операцией; валидация лимита (п. 2 эндпоинта) отбивает 400.
8. После расширения: принятые в платном клубе — `frozen` с DM «оплатите»; в бесплатном —
   `active`; блок в инбоксе рассыпался.
9. Заявитель НЕ видит экран оплаты до одобрения (инвариант).
10. Регресс: обычные заявки в `closed`-клуб с местами работают как раньше.
