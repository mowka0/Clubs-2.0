# Гео к событию — Яндекс.Карты (event-geo)

> Статус: **реализовано 2026-07-11** (ветка `feature/event-geo`).
> Мокапы (одобрены PO 2026-07-11): `docs/design/event-geo/mockups/01-geo-flow.html`
> (кадры A — форма, B — пикер, C — страница события; mockups/ в .gitignore, файл локальный).
> Хэндофф-план: `docs/backlog/event-geo-handoff.md`.
> Принятые PO Low-риски (CSP, quota-burn DoS): `docs/backlog/event-geo-hardening.md`.

## Зачем

«Место» события перестаёт быть свободным текстом: организатор ставит **точку на карте**
(поиск по адресу + уточнение пином — двор, вход, беседка), а участник получает
**статичную мини-карту** и кнопку «Маршрут», открывающую Яндекс.Карты с построением пути.
Убирает главную боль офлайн-встреч — «а где именно собираемся?».

## Решения PO (2026-07-11)

1. **Вариант 2 с поиском по адресу**: в форме создания события вместо текстового поля —
   кнопка «Добавить место» → шит-пикер: строка поиска → «Найти» (геокодер) → карта прыгает
   к результату → пин закреплён в центре, тянешь карту — уточняешь точку → «Готово».
2. **Участнику — статичная мини-карта с пином** (Static API) + кнопки «🧭 Маршрут» и
   «Открыть в Картах» (deep-link, бесключевой).
3. **Опциональное поле-уточнение** («Вход со двора, домофон 12») — отдельное от адреса,
   ≤ 200 символов.
4. **FAIL-CLOSED** (решение на старте сессии реализации): при недоступности CDN/ключа
   Яндекса создание события **блокируется** («Карты недоступны, попробуйте позже»),
   текстового фолбэка нет. Следствие: `locationLat`/`locationLon` обязательны в
   `CreateEventRequest` и на бэке; колонки БД nullable только ради легаси-событий.
5. Редактирования события в продукте нет (только создание/отмена) — пикер живёт
   только в форме создания.

## Провайдер и ключи

- Яндекс.Карты, бесплатный тариф **~100 запросов/сутки на сервис**. Ключи — фронтовые
  публичные, в env (в git не попадают). В кабинете PO продукты **раздельные** (связки
  «JavaScript API и HTTP Геокодер», чьи ключи принимает ymaps3/v3, там нет — v3 отвечает
  403 «Invalid api key», проверено curl 2026-07-11; поэтому пикер работает на **JS API v2.1**):
  - `VITE_YANDEX_MAPS_API_KEY` — продукт «JavaScript API» (v2.1), скрипт карты пикера;
  - `VITE_YANDEX_STATIC_API_KEY` — продукт «Static API», мини-карта события;
  - `VITE_YANDEX_GEOCODER_API_KEY` — продукт «API Геокодера», поиск адреса + реверс.
- Экономия лимита: геокодинг **только по кнопке «Найти»** (никакого live-саджеста),
  обратный геокодинг — **один раз** по «Готово».
- **Изоляция провайдера**: весь Яндекс-специфичный код — в `frontend/src/utils/yandexMaps.ts`
  и `LocationPickerSheet.tsx`; смена на 2ГИС/OSM не трогает страницы.
- Referer-ограничение ключей настраивает PO в кабинете developer.tech.yandex.ru
  (домены sslip.io + localhost).

## Схема БД (V57)

```sql
ALTER TABLE events
    ADD COLUMN IF NOT EXISTS location_lat  DOUBLE PRECISION NULL,
    ADD COLUMN IF NOT EXISTS location_lon  DOUBLE PRECISION NULL,
    ADD COLUMN IF NOT EXISTS location_hint VARCHAR(200) NULL;

ALTER TABLE events
    ADD CONSTRAINT chk_events_location_pair CHECK ((location_lat IS NULL) = (location_lon IS NULL));
```

- `location_lat`/`location_lon` — WGS-84; NULL = легаси-событие без точки (создано до фичи).
  Инвариант «оба NULL или оба заданы» закреплён **CHECK-констрейнтом `chk_events_location_pair`
  в БД** (любой будущий write-путь не запишет половинную точку) + `@NotNull` в DTO для новых
  событий.
- `location_hint` — опциональное уточнение организатора; NULL = нет.
- `location_text` — не меняется (NOT NULL), для новых событий заполняется адресом из
  обратного геокодера (фолбэк — «lat, lon» строкой, если реверс не ответил).

## API-контракты

### POST /api/clubs/{clubId}/events — новые поля запроса

| Поле | Тип | Валидация |
|---|---|---|
| `locationLat` | Double | **обязательно**, ∈ [-90, 90] |
| `locationLon` | Double | **обязательно**, ∈ [-180, 180] |
| `locationHint` | String? | опционально, ≤ 200 символов, пусто → null |

Остальное без изменений (`locationText` по-прежнему обязателен ≤ 500 — адрес).

### GET /api/events/{id} — новые поля EventDetailDto

`locationLat: Double?`, `locationLon: Double?`, `locationHint: String?` — null у легаси-событий.

Списковые DTO (`EventListItemDto`, `MyEventListItemDto`, `ActivityItemDto`) координаты
**не получают** — карта показывается только на странице события (YAGNI).

## Фронтенд

### URL-форматы Яндекса (`utils/yandexMaps.ts`)

- Лоадер JS API: `https://api-maps.yandex.ru/2.1/?apikey=<MAPS-KEY>&lang=ru_RU` (v2.1,
  см. «Провайдер и ключи» — почему не v3).
- Геокодер: `https://geocode-maps.yandex.ru/1.x/?apikey=<GEOCODER-KEY>&geocode=<text>&format=json&results=1&lang=ru_RU`
  (и обратный: `geocode=<lon>,<lat>`). Таймаут запроса — 10 с (`AbortSignal.timeout`,
  в старых webview без него).
- Статичная карта: `https://static-maps.yandex.ru/v1?apikey=<STATIC-KEY>&ll=<lon>,<lat>&z=16&size=650,300&pt=<lon>,<lat>,pm2rdm`.
- Маршрут: `https://yandex.ru/maps/?rtext=~<lat>,<lon>` (от текущей геопозиции).
- Открыть точку: `https://yandex.ru/maps/?pt=<lon>,<lat>&z=17`.

⚠️ Ловушка: в `ll`/`pt`/`geocode` у Яндекса порядок **lon,lat**; в `rtext` — **lat,lon**;
а JS API **v2.1** внутри использует **[lat, lon]**. Наружу порядки не торчат: интерфейс
`PickerMap` (обёртка над v2.1) оперирует только `GeoPoint {lat, lon}`.

### Кадр A — CreateEventPage

- Вместо input «Место»: дашед-кнопка «Добавить место» (реюз `.rd-invite-row`) с подписью
  «Найдите адрес и уточните точку на карте» → открывает пикер.
- После выбора кнопка сменяется карточкой адреса (📍 адрес + «точка уточнена на карте» +
  кнопка «Изменить» → снова пикер).
- Ниже — поле «Уточнение к месту (необязательно)», ≤ 200 символов.
- Сабмит требует выбранную точку («Укажите место на карте»). Fail-closed: без точки
  событие не создаётся, текстовый ввод адреса недоступен.

### Кадр B — LocationPickerSheet

- Боттом-шит по паттерну InviteSheet (`createPortal`, `rd-sheet-*`).
- Строка поиска + кнопка «Найти» (🔍) → `geocode()` → карта летит к результату
  (строка «✓ Найдено: … — уточните точку, потянув карту»). Ошибки поиска различаются:
  пустой ответ — «Не нашли адрес — попробуйте сформулировать иначе.», HTTP-ошибка геокодера
  (лимит/ключ, `GeocoderHttpError`) — «Поиск временно недоступен, попробуйте позже.»,
  сетевая — «Не получилось найти адрес — проверьте соединение и попробуйте ещё раз.».
- Карта JS API v2.1 через провайдер-нейтральную обёртку `createPickerMap` → `PickerMap`
  (`getCenter`/`panTo`/`destroy`), **пин — абсолютный div в центре контейнера** (не маркер
  на карте): тянешь карту — точка под пином. Подпись-пилюля «Пин в центре — тяните карту».
- «Готово» → координаты центра карты + `reverseGeocode()` (1 запрос) → адрес в форму;
  если реверс упал — адресом становится «lat, lon» строкой (6 знаков после запятой).
- «Отмена» → закрыть без изменений. Дефолт-центр — Москва (55.751244, 37.618423), zoom 12;
  повторное открытие с выбранной точкой — центр на ней, zoom 16.
- Ошибка загрузки CDN → в шите сообщение «Карты недоступны, попробуйте позже» и
  кнопка «Готово» замьючена (fail-closed).

### Кадр C — EventPage

- События **с координатами**: карточка места = адрес (`locationText`) + серое уточнение
  (`locationHint`) + `<img>` статичной карты (тап → открыть точку в Яндекс.Картах) +
  кнопки «🧭 Маршрут» / «Открыть в Картах» (`window.open`, `_blank`).
- События **без координат** (легаси): текущий блок без изменений — текст без карты.
- Если картинка Static API не загрузилась — `<img>` скрывается (`onError`), адрес и
  кнопки остаются.

## Env-проводка (фронт собирается в Docker)

- `frontend/Dockerfile`: `ARG` + `ENV` для всех трёх `VITE_YANDEX_*` ключей перед `npm run build`.
- `docker-compose.prod.yml` → `frontend.build.args`: три переменные из env приложения Coolify.
- `.env.example`: три плейсхолдера.
- `frontend/.env.local` (gitignored) для локальной разработки создаёт PO руками.
- `frontend/src/vite-env.d.ts`: типы обеих переменных.

## Критерии приёмки

1. Организатор: «Добавить место» → поиск «Покровка 47» → «Найти» → карта на адресе →
   потянуть карту → «Готово» → в форме карточка с адресом; создать → событие создано,
   в БД lat/lon/hint заполнены.
2. Участник: на странице события — адрес, уточнение, мини-карта с пином; «Маршрут»
   открывает Яндекс.Карты с построением пути; «Открыть в Картах» — точку.
3. Легаси-событие (lat/lon NULL) — блок места как раньше, без карты и кнопок.
4. Создание без выбранной точки — блокируется с ошибкой «Укажите место на карте».
5. CDN Яндекса недоступен → пикер показывает «Карты недоступны, попробуйте позже»,
   «Готово» неактивна, событие создать нельзя (fail-closed).
6. Бэкенд отклоняет: lat/lon вне диапазонов, отсутствующие lat/lon, hint > 200 символов (400).
7. Регресс: событие в закрытом/платном клубе создаётся и открывается как раньше.

## Нефункциональные

- Лимит 100 зап/сутки: ≤ 2 геокод-запроса на одно создание события (поиск + реверс);
  JS API грузится только при открытии пикера (ленивая загрузка скрипта), Static API — 1
  картинка на просмотр страницы события (браузерный кэш переиспользует).
- Ключи — публичные фронтовые, но в git не хранятся (env-indirection, referer-ограничение
  в кабинете Яндекса).
- Безопасность: координаты валидируются на границе (Bean Validation); `locationHint`
  рендерится React'ом (экранирование по умолчанию); URL статичной карты строится на
  клиенте из чисел, не из строк пользователя.
