# Путь назад + перенос репутации в «Мои клубы» + асимметричная видимость Trust

> Статус: **реализовано, ждёт staging-теста PO** (ветка `feature/path-back`, 2026-07-05).
> Отклонения от первоначальной спеки (уточнены в ходе разработки):
> 1. Строка-мотиватор на событии скрыта не только у `confirmed`, но и у `declined` /
>    `expired_no_confirm` (звать отказавшегося/просрочившего бессмысленно). AC-8 уточнён.
> 2. В раскрытой карточке CTA «Открыть клуб» для всех ролей (в мокапе у организатора было
>    «Управление клубом» — управление доступно со страницы клуба, отдельный роут не тянем).
> Правки по staging-тесту PO (2026-07-05): (а) кольца Посещаемость/Сборы убраны — статистика
> только текстом («… · сборы 6/6»); (б) проекция моделирует ВРЕМЯ (см. ниже) — иначе у
> пользователя с большой историей она обещала «+1 балл за встречу»; (в) UI-заголовок блока —
> **«Путь наверх»** (имя фичи/файлов остаётся path-back); (г) в раскрытой карточке — чипы
> клубных наград вызывающего (`awards` в DTO, батч `AwardService.getUserAwardsByClub`, те же
> `rd-award-chip`, что в ростере); (д) шеврон раскрытия 24px, при раскрытии — акцентный цвет.
> Решения PO зафиксированы 2026-07-05. Мокапы: `docs/design/reputation-path-back/mockups/`
> (01 — варианты A/B/C «пути назад», выбран **A**; 02 — раскрывающаяся карточка «Моих клубов»).
> Родительская спека репутации: `reputation-v2.md` (формула Trust — § P1b, не меняется).

## Цель

1. **«Путь назад»** — после просадки Trust показывать владельцу не «приговор», а рецепт
   восстановления с числами: проекция «сейчас → +1 встреча → +2 встречи» (стратегия §1:
   «после промаха показывать дорогу назад, а не приговор»).
2. **Перенос** — блок «Репутация по клубам» уезжает из Профиля в «Мои клубы»: клуб и репутация
   в нём — одна сущность, одна раскрывающаяся карточка. Профиль освобождается под будущие данные.
3. **Асимметричная видимость** — чужой Trust рядовым участникам не показывается (позитив публично,
   оценки — только себе и организатору). Убирает «дно таблицы», противоречащее стратегии §3.

## Scope

### Входит
- Бэкенд: проекция «пути назад» в `UserClubReputationDto` (поля `projectedNext1`,
  `projectedNext2`, `meetingsToReliable`) — только для активных клубов вызывающего.
- Бэкенд: `skladchinaPaid`/`skladchinaTotal` в `UserClubReputationDto` (кольцо «Сборы»).
- Бэкенд: асимметричная видимость в `getClubMembers` (trust=null чужим + нейтральная сортировка
  для не-организаторов) и `getMemberProfile` (репутационный блок только организатору и себе).
- Фронт: раскрывающаяся карточка клуба в «Моих клубах» (свёрнуто = текущий вид + Trust справа;
  раскрыто = метрики + кольца + путь назад (вариант A) + ближайшая встреча + «Открыть клуб»).
- Фронт: удаление панели «Репутация» (per-club список) из Профиля.
- Фронт: строка-мотиватор (вариант C) на странице события.

### НЕ входит
- Изменение формулы Trust / порогов (`TrustPolicy` — только рефакторинг выделения весов).
- Тренды/стрелки чужого Trust, тиры для участников — отвергнуто (см. память PO).
- Глобальный блок Профиля («надёжен в N из M», уровень, интересы) — остаётся в Профиле.
- Карточка ЗАЯВКИ (глобальная репутация организатору) — без изменений.
- DM/уведомления о просадке — отдельная тема (нужно «окошко разрешить боту писать»).

## API-контракты (изменения)

### `GET /api/users/me/reputation` — расширение `UserClubReputationDto`
```jsonc
{
  // ... существующие поля без изменений ...
  "trust": 60,
  // НОВОЕ. Проекция «пути назад» по той же TrustPolicy (чистый пересчёт с весами +1/+2
  // посещения, decay-вес нового исхода = 1.0). null, когда путь назад не показывается:
  // trust null (гейт MIN_OUTCOMES) ИЛИ trust >= 70 (RELIABLE_THRESHOLD) ИЛИ клуб неактивен.
  "projectedNext1": 66,        // Trust после +1 посещения
  "projectedNext2": 70,        // Trust после +2 посещений
  "meetingsToReliable": 2,     // посещений до >= 70; capped 9 (больше → 9, UI пишет «9+»)
  // НОВОЕ. Кольцо «Сборы» (как в карточке участника): оплачено / (оплачено+просрочено)
  // РЕПУТАЦИОННЫХ складчин. null-безопасно: 0/0 → кольцо не рендерится.
  "skladchinaPaid": 3,
  "skladchinaTotal": 3
}
```
Проекция — **детерминированный пересчёт с учётом времени**: посещение j моделируется через
j × `PATH_BACK_MEETING_INTERVAL_DAYS` (14 дней ≈ клубный ритм «2 встречи/месяц»); к шагу k старые
веса затухают на d^k (d = 0.5^(14/90) по тому же HALF_LIFE_DAYS), свежие посещения добавляются с
весом на момент шага. Восстановление честно идёт двумя механизмами живой формулы: новые посещения
+ затухание старых промахов. Без учёта времени формула инертна при большой истории (staging-кейс
PO: 17 подтв. → «+1 балл за встречу»). Новых механик и хранимых полей нет, миграций нет.

### `GET /api/clubs/{id}/members` — асимметричная видимость
- Вызывающий **организатор**: как сейчас (trust у всех, сортировка organizer → trust DESC).
- Вызывающий **участник**: `trust` заполнен ТОЛЬКО у собственной строки (`userId == caller`),
  у остальных `null`; сортировка **нейтральная**: organizer первым, дальше по `joinedAt` ASC
  (стабильно, без «дна таблицы»). Награды (awards) видны всем, как раньше (позитив публичен).

### `GET /api/clubs/{clubId}/members/{userId}` (профиль участника)
- Организатор ИЛИ свой профиль (`userId == caller`): как сейчас.
- Иначе (участник смотрит чужой): `trust`, `promiseFulfillmentPct`, `totalConfirmations`,
  `totalAttendances`, `spontaneityCount`, `skladchinaPaid`, `skladchinaTotal` = **null** —
  фронт показывает ту же подачу, что для «Новичка» (неоднозначность по дизайну: «нет данных»
  неотличимо от «скрыто»). `bio`/`interests`/`awards`/`role` — без изменений.

## Поведение UI

### «Мои клубы» — раскрывающаяся карточка (мокап 02, фрейм 1-3)
- Данные: join `useMyClubsQuery` (клубы) × `useMyReputationQuery().activeClubs` по `clubId` —
  оба запроса на странице уже есть, новых нет.
- **Свёрнуто**: текущая карточка (аватар, имя, «категория · N/M») + справа Trust с tier-цветом
  и подписью «надёжность» (паттерн покинутых клубов). Состояния без числа: «Новичок · пока нет
  данных» (гейт), «Организатор · ваш клуб». Шеврон ▾, поворот при раскрытии.
- **Раскрыто** (тап по шапке): чипы клубных наград вызывающего (если есть); строка метрик
  «обещания N% · X подтв. · Y посещ. [· Z спонт.] [· сборы P/T]» — вся статистика ТОЛЬКО текстом
  (кольца убраны по staging-фидбеку PO); блок «Путь наверх» (см. ниже); CTA «Ближайшая встреча»
  (`nearestEvent`, скрыто если null) → страница события; «Открыть клуб» → страница клуба.
- **Блок «Путь наверх» (вариант A)**: три кольца «сейчас → +1 встреча → +2 встречи» со
  значениями `trust / projectedNext1 / projectedNext2`; третье кольцо зелёное при
  `projectedNext2 >= 70` с подписью «надёжная зона». Текст: «Две ближайшие встречи вернут вас
  в надёжную зону» при `meetingsToReliable <= 2`, иначе «До надёжной зоны — N посещений»
  (N=meetingsToReliable, «9+» при 9). Рендерится ТОЛЬКО при `projectedNext1 != null`.
- Раскрытие — чистый клиентский стейт (useState set of clubId), без запросов.
- Секция «История» (покинутые клубы) — без изменений (там Trust уже показывался).

### Профиль
- Панель «Репутация» со списком клубов (activeClubs/historyClubs rows) — **удаляется**.
- Остаются: глобальный блок («надёжен в N из M», Надёжность-статистика), «Уровень», «Интересы».
- `useMyReputationQuery` на Профиле остаётся ради global-блока.

### Страница события — строка-мотиватор (вариант C)
- Место: внутри блока Этапа 2 / голосования, под кнопками действия.
- Условия показа (все): вызывающий — участник клуба; его `projectedNext1 != null` (т.е. trust<70
  и есть история); окно действия открыто (`showVoting` ИЛИ `showStage2`); пользователь ещё НЕ
  `confirmed` (после подтверждения строка исчезает — обещание уже дано).
- Текст: «↗ Придёте на эту встречу — надёжность вырастет: {trust} → {projectedNext1}».
- Данные: `useMyReputationQuery` + find по `event.clubId` (запрос кэшируется, не дублируется).

## Acceptance Criteria

### AC-1: проекция считается по формуле (с учётом времени)
GIVEN участник клуба со свежими весами kept=1, broke=1 (trust=59)
WHEN он запрашивает `/api/users/me/reputation`
THEN `projectedNext1=66`, `projectedNext2=72`, `meetingsToReliable=2`
AND значения равны `TrustPolicy.projectedTrust(weights, k)` для k=1,2 (decay-aware, интервал 14 дн).

### AC-2: путь назад скрыт у здоровой репутации
GIVEN trust=82 (>=70) THEN `projectedNext1/2 = null`, `meetingsToReliable = null`.
GIVEN trust=null (новичок ниже гейта) THEN все поля проекции null.

### AC-3: участник не видит чужой Trust
GIVEN вызывающий — рядовой участник клуба
WHEN он запрашивает список участников
THEN `trust != null` только у его собственной строки
AND порядок: организатор первым, далее по joinedAt ASC.

### AC-4: организатор видит всё как раньше
GIVEN вызывающий — организатор клуба
THEN trust заполнен у всех (как сейчас), сортировка по trust DESC сохранена.

### AC-5: чужая карточка участника без оценок
GIVEN участник открывает карточку ДРУГОГО участника
THEN репутационные поля null → UI рендерит подачу «Новичок» (без колец Надёжность/Сборы)
AND bio/интересы/награды видны.
GIVEN открывает СВОЮ карточку или смотрит организатор THEN как раньше.

### AC-6: карточка «Моих клубов»
GIVEN клуб с trust=60 и историей WHEN раскрыть карточку
THEN видны метрики, кольца, «Путь назад» (60→66→70), ближайшая встреча (если есть).
GIVEN trust>=70 THEN раскрытый вид БЕЗ блока «Путь назад».
GIVEN «Организатор»/«Новичок» THEN раскрытый вид: пояснительная строка, без метрик/колец.

### AC-7: Профиль без per-club списка
WHEN открыть Профиль THEN панели со списком клубов нет; глобальный блок/уровень/интересы на месте.

### AC-8: строка на событии
GIVEN участник с projectedNext1=66, событие в Этапе 2 (или голосование Этапа 1 открыто),
его статус НЕ в {confirmed, declined, expired_no_confirm}
THEN видна строка «↗ Придёте на эту встречу — надёжность вырастет: 60 → 66».
WHEN он подтверждает участие THEN строка исчезает.
GIVEN trust>=70 ИЛИ новичок ИЛИ статус терминальный THEN строки нет.

## Non-functional
- **Производительность**: проекция — O(1) поверх уже читаемых outcomes (`computeForUser`);
  новых запросов к БД нет. `/members` — без изменения количества запросов.
- **Безопасность (A01)**: видимость режется НА СЕРВЕРЕ (null в DTO), фронт-скрытие — только UX.
  Внутренние константы формулы (веса, K, prior) наружу не отдаются — только итоговые числа.
- **Логи**: без новых логов (чтение); изменения видимости не логируются (не событие безопасности).

## Интеграции
- `TrustPolicy` — выделить подсчёт весов (kept/broke) в переиспользуемую функцию; формула не меняется.
- `TrustService.computeForUser` — обогащение `ClubTrust` весами/проекцией.
- `MembershipService.getMyReputation` → mapper → DTO (skladchina-счётчики из тех же outcomes).
- `MemberService.getClubMembers`/`getMemberProfile` — гейт по зрителю (прецедент: `forOrganizer`).
- Фронт: `MyClubsPage`, `ProfilePage`, `EventPage`, `MemberProfileModal`, `ClubMembersTab`.

## Риски / открытые вопросы
- Сортировка для не-организатора меняет привычный порядок списка — осознанно (убираем рейтинг).
- «Сборы 0/0» у старых клубов — кольцо просто не рендерится (как в карточке участника).
- meetingsToReliable при огромной просадке — cap 9 («9+»), чтобы не обещать марафон точным числом.
