# Монетизация v2 — залоченная модель (buildable spec)

**Статус:** РЕШЕНО (design-LOCK). Продуктовая модель закрыта; код по сервисному сбору — gated, не строится сейчас (§7).
**Дата LOCK:** 2026-06-25.
**Источник решений:** `docs/design/payment-monetization-v2.md` (decision record) + design-сессия 2026-06-25 (4 развилки через PO).
**Отношение к v1:** `docs/modules/payment.md` описывает as-built Stars-путь (нейтрализуется decoupling-слайсом). Этот документ — **целевая** модель.

> Эта спека фиксирует **что** мы построим, когда будем монетизировать, и в **каком порядке**. Она НЕ ставит дату пэйвола, НЕ выбирает финальные цены (tuning-knob, §6), НЕ заказывает acquirer-интеграцию. Включение сбора gated на (а) юридику и (б) доказанную активность бесплатного цикла (design-док §10).

## 1. Резюме залоченных решений

| # | Решение | Итог |
|---|---|---|
| 1 | **Кто платит** | **Оба плательщика обязательны.** v1 — **organizer-pays** (строим первым). Фаза 2 — **member-pays** (committed, строится позже). |
| 2 | **Структура планов** | **Три тарифа** `FREE`(1 платный клуб, 0₽) / `TRIO`(до 3, 200₽) / `UNLIMITED`(∞, 400₽). Единица ёмкости = **платный клуб**; бесплатные клубы безлимитны и не считаются. Инвариант — **free-floor + no-cliff volume-discount** (§3.2). |
| 3 | **Триггер** | **Месячная рекуррентная подписка**. Пэйвол показывается на **последнем шаге создания 2-го+ платного клуба** (§3.6). |
| 4 | **Формат подписки** (PO-override 2026-06-25) | Подписка работает весь оплаченный период **независимо от наличия активных клубов**, **не паузится**, выключается **только в конце периода**. Отменяет freeze-by-hours (§7.3 дизайн-дока) и aggregate-flag pause (§7.5). |

**Залочено ранее (design-док §1/§5, не переоткрывается):**
- Платформа **вне потока денег** клуба: взносы участник→организатор идут мимо платформы (P2P-out, механика складчины).
- Платформа биллит **только свой собственный сервисный сбор** (УСН 6%, свои чеки, без сплита). Рекуррентность чистая именно потому, что это собственный сбор (design-док §5.2).
- **Telegram Stars отвергнуты** для РФ как денежный поток.

## 2. Последовательность реализации (who-pays)

PO зафиксировал: **оба плательщика обязательны**. Сиквенс делегирован разработке → выбран **organizer-first**.

### 2.1 v1 — organizer-pays (строим первым)
- Платит **организатор** — за тариф-ёмкость (сколько **платных** клубов он может вести; см. §3 — единица = платный клуб, бесплатные не считаются).
- **Первый платный клуб — бесплатно** (тир `FREE`). Сбор начинается со **2-го** платного клуба.
- Членство для участника **бесплатно** на стороне платформы; взнос за клуб участник платит организатору напрямую (P2P-out, мимо платформы).
- **Один плательщик на клуб** → минимум биллинг-поверхности на старте.
- **Обоснование сиквенса:** design-док §10 — «не брать с участников рано, сначала доказать цикл»; организатор — сторона с очевидным ROI; обходит непроверенную гипотезу «платить за бывшее бесплатным» на самой ценочувствительной стороне.

### 2.2 Member-pays — рельсы строим сейчас, свет за фиче-флагом (default OFF)
- Участник платит платформе **отдельный мелкий сервисный сбор** (платформенная premium-подписка «Clubs+»; что разблокирует — §2.3).
- **Build-решение (2026-06-25):** движок member-подписки **строится сразу** (один engine с `payer_role`, см. §5.1), но **скрыт за feature-флагом, default OFF** (паттерн как `CLUB_RANK_BADGE_ENABLED`). Включается тогглом, когда наберётся аудитория + появится HQ-премиум для разблокировки. Удовлетворяет §10 («не брать с участников рано») — рельсы готовы, свет выключен.
- **Таблица memberships остаётся чистой** — member-подписка живёт в своей сущности (`service_subscription`, `payer_role=MEMBER`), НЕ колонкой в membership.

### 2.3 Что разблокирует member-подписка (граница «платформенное vs продукт организатора»)
**Несущая линия:** участник платит платформе только за **платформенное**, НЕ за доступ к контенту организатора (за доступ/события платит организатору взносом P2P — иначе двойной платёж + риск переквалификации в комиссию).
- **Платформенное (можно гейтить премиумом):** усиленная принадлежность к штаб-квартире, расширенный профиль в каталоге, **приоритет/ранний доступ к ограниченным местам** (механика гонки за места — платформенная), премиальные репутационные фичи.
- **НЕ гейтить:** базовую принадлежность, доступ к событиям как таковой, ядро штаб-квартиры (иначе ломается «доказать цикл на бесплатной базе», §10).
- **HQ (штаб-квартира) — отдельный продуктовый трек** (стена/каталог/идентичность/опросы), строится бесплатно ради retention; member-премиум — тонкий слой поверх.

## 3. Структура планов (Решение 2)

### 3.1 Лестница тарифов
`plan` = **enum/lookup**, не свободное число. Ёмкость считается в **платных клубах**:

| plan | Ёмкость (`max_paid_clubs`) | Цена/мес (launch) |
|---|---|---|
| `FREE` | 1 | 0₽ |
| `TRIO` | 3 | 200₽ |
| `UNLIMITED` | ∞ | 400₽ |

- **Единица ёмкости = платный клуб** (клуб с взносом участников / гейтом доступа, см. decoupling-слайс). **Бесплатные клубы (без взноса) — безлимитны на всех тирах и в ёмкость НЕ считаются** (защита discovery-движка).
- Ёмкость = **ceiling-чек** при создании/активации платного клуба: организатор не может иметь активных **платных** клубов больше `max_paid_clubs` своего плана.
- Первый платный клуб покрыт тиром `FREE` (0₽). Сбор начинается со 2-го.

### 3.2 Ценовой инвариант (free-floor + no-cliff volume-discount)
Несущее правило — не сами числа, а форма лестницы:

- **Free-floor:** `FREE` (1 платный клуб, 0₽) — giveaway-пол; защищает одноклубного хоббиста и discovery. Он **вне** метрики «цена за клуб».
- **No-cliff guard:** цена `UNLIMITED` **≤ 2× entry-paid тира** (`TRIO`). Launch: `400 ≤ 2×200` — проходит. Гарантирует, что безлимит открывается **дёшево**, активный мульти-клуб организатор не наказывается.
- **Volume-discount по ceiling:** цена-за-клуб **на потолке** платного тира **монотонно невозрастающая** вниз по лестнице (TRIO 200/3 ≈ 66,7₽ ≥ UNLIMITED → 0 при росте клубов).
- **Жёсткий анти-паттерн (запрещён):** лестница, где переход к следующему тиру стоит дороже входа в платность (отвергнутая сетка-штраф). Известный артефакт (НЕ нарушение): на границе `TRIO→UNLIMITED` цена-за-клуб 4-го клуба временно подскакивает (66,7→100₽), затем падает — неизбежно для любого безлимита, т.к. TRIO кончается на 3.

### 3.3 Где живут цены
- Числа выше (200/400₽) — **launch-значения, залочены PO 2026-06-25** как стартовая сетка.
- Хранятся в **config/pricing-таблице, keyed by `plan` + `effective_date`**, никогда не хардкод, никогда от клиента (сервер считает).
- `effective_date` позволяет менять цены **без миграции** — поддерживает будущий пересчёт под НДС-22% эквайринга (legal-блок §8), не переоткрывая модель.

### 3.4 Инвариант enforced тестом
- Unit-тест утверждает: (а) `UNLIMITED ≤ 2× TRIO` (no-cliff), (б) цена-за-клуб на потолке платных тиров невозрастающая. Сторожит анти-паттерн §3.2 навсегда, не только доком.

### 3.5 Смена тарифа
- **Без авто-proration.** Смена плана = plan-swap на **следующей биллинг-границе**, не mid-cycle перерасчёт.

### 3.6 Когда показываем пэйвол (триггер)
- Пэйвол-модалка всплывает на **последнем шаге создания клуба** — данные введены, осталось нажать «Создать».
- Условие показа: создаваемый клуб — **платный** И он **превышает** `max_paid_clubs` текущего плана организатора (т.е. это его **2-й+** платный клуб на тире `FREE`).
- 1-й платный клуб и **любые** бесплатные клубы создаются без модалки.
- Реализация: capacity ceiling-чек срабатывает в обработчике «создать клуб»; при превышении — блок создания + предложение оформить/апгрейдить план. Зависимость: концепт «платный клуб» вводит decoupling-слайс (Трек B); enforcement пэйвола — Трек C (gated).

## 4. Биллинг-стейт-машина (Решение 3 + 4, упрощённый формат)

> **PO-override 2026-06-25:** подписка = обычная месячная рекуррентная подписка. Работает весь оплаченный период **независимо** от того, есть ли у организатора активные клубы. **Не паузится** по активности. Выключается **только** в конце оплаченного периода (после отмены или несостоявшегося продления). Это отменяет freeze-by-hours и aggregate-flag-pause из дизайн-дока §7.3/§7.5.

### 4.1 Состояния подписки

| Состояние | Доступ к премиум-ёмкости | Сбор |
|---|---|---|
| `ACTIVE` | Есть | Авто-продление раз в месяц |
| `CANCELLED_PENDING_END` | Есть до конца периода | Больше не продлевается; в конце периода → `ENDED` |
| `PAST_DUE` (dunning) | Есть в течение grace | Ретраи продления; успех → `ACTIVE`, исчерпание ретраев → `ENDED` |
| `ENDED` | Нет | Отменён |

- **Нет** состояния `Frozen` на уровне подписки. **Нет** банка часов. **Нет** aggregate-flag.
- Переходы — **forward-only** и `@Transactional` вместе с грантом/ревоком доступа (защита от out-of-order webhooks).

### 4.2 Золотое правило (§7.1 дизайн-дока) — сохранено
**Нет доступа ⇒ нет списания.** Заряд следует за статусом доступа атомарно. В organizer-pays «доступ» = доступ к премиум-ёмкости платформы (вести платные клубы / инструменты). Отмена → потеря доступа в конце периода → списания прекращаются. Правило выполняется тривиально.

### 4.3 Кто может завершить + гард сверх-лимита (РЕШЕНО 2026-06-25, decision A)
- **Организатор** отменяет свою подписку сам (`ACTIVE → CANCELLED_PENDING_END`, доигрывает до конца периода). Завершение = отмена авто-продления; доступ держится до конца оплаченного периода.
- **Над-лимитный гард (решение A):** **отмена И даунгрейд блокируются (409), пока у организатора платных клубов больше ёмкости целевого плана.** Сначала сделать лишние клубы бесплатными или удалить, потом менять тариф / отменять. Enforced в `SubscriptionService.cancel` (против FREE-ёмкости) и `subscribe` (против ёмкости целевого плана).
- Ёмкость держится не только на создании: **создание** платного клуба, **редактирование** free→paid (`ClubService.updateClub`), **отмена** и **даунгрейд** — все проверяют лимит. Иначе организатор обходит потолок.

### 4.4 Что происходит с клубами при `ENDED`
- Из-за гарда §4.3 организатор **не может добровольно дойти до `ENDED`, будучи сверх лимита** → к моменту `ENDED` платных клубов уже ≤ FREE-ёмкости. Поэтому отдельного «over-capacity на ENDED» поведения не требуется (вопрос §4.4 снят гардом).
- **Открытый край:** при будущем **недобровольном** `ENDED` (провал продления → `PAST_DUE → ENDED`, когда появится реальный acquirer) организатор может оказаться сверх лимита. Тогда нужен over-capacity-обработчик (freeze/архив, данные целы) — отложено вместе с dunning (§4.6), сейчас не достижимо (стаб продление не валит).

### 4.5 Заморозка организатором ≠ состояние подписки
- «Заморозка доступа участнику» (организатор не получил P2P-взнос) — это **гейт доступа к КЛУБУ** (организатор↔участник, honor-system, decoupling-слайс), **ортогональный** платформенной подписке.
- В organizer-pays это **не влечёт** платформенного заряда с участника. Не путать с биллинг-состоянием.

### 4.6 Dunning
- При сбое продления: `ACTIVE → PAST_DUE`, ретраи в течение grace. Лочим **состояния**; **расписание** ретраев (сколько, как долго) — tuning-knob.

## 5. Модель данных (high-level, forward-compatible под фазу 2)

> Строится в Треке C (gated). Здесь — целевая форма, чтобы member-pays встал без миграции-перелопачивания.

### 5.1 `service_subscription` (платформенный сбор)
| Поле | Назначение |
|---|---|
| `id` | PK |
| `payer_user_id` | Кто платит платформе |
| `payer_role` | `ORGANIZER` (v1) \| `MEMBER` (фаза 2) — payer-dimension с самого начала |
| `plan` | `FREE`/`TRIO`/`UNLIMITED` для organizer; member-tier для фазы 2 |
| `subject_club_id` | NULL для organizer (platform-wide); club-scoped для member-pays |
| `status` | enum §4.1 (`ACTIVE`/`CANCELLED_PENDING_END`/`PAST_DUE`/`ENDED`) |
| `current_period_end` | Конец оплаченного периода (драйвер «выключается в конце периода») |
| `provider_token` | Merchant-tokenization (saved card / СБП-подписка) |
| `provider_event_id` | **UNIQUE** — входящая идемпотентность webhook |
| timestamps | created/updated |

- **memberships не трогаем** — никакого `member.subscription_id`. Member-pays живёт здесь (`payer_role=MEMBER`, `subject_club_id` заполнен).
- `plan` ёмкость (`max_clubs`) — в lookup-таблице плана, не в подписке.

### 5.2 `pricing` (config)
- Keyed by `plan` + `effective_date`. Сервер считает сумму, клиент не доверяется. Числа — отложены (§3.3).

### 5.3 Эндпоинты (design-док §9)
- `create-subscription` (свой сбор, merchant-tokenization).
- `webhook` (verify signature + идемпотентность по `provider_event_id UNIQUE`).
- `cancel` (→ `CANCELLED_PENDING_END`).
- `status`.

### 5.4 `PaymentProvider`-seam
- Единый интерфейс над провайдером собственного сбора. Конкретный acquirer (ЮKassa/Т-Банк) — legal/ops-блок (§8), дропается реализацией позже. As-built: введён `StubPaymentProvider` (`@Component`, синхронно «активирует», денег не двигает, `parseWebhook → Ignored`); реальный acquirer — второй `@Component` за тем же сеамом.

## 6. Что НЕ залочено (tuning-knobs, не блокеры)
- Цены **залочены как launch-сетка** (0 / 200 / 400₽), но остаются config-tunable через `effective_date` под будущий НДС-пересчёт (§3.3, §8) — без переоткрытия модели.
- Расписание dunning-ретраев.
- Over-capacity UX при `ENDED` (§4.4).
- Member-tier структура для фазы 2 (форма subject-scoped подписки задана, числа/тиры — нет).

## 7. Реализация — gated, не сейчас

- **§7 биллинг (Трек C)** — стейт-машина §4 + реальный acquirer + рекуррент. **Gated** на (а) юридику (§8) и (б) доказанную активность бесплатного цикла (design-док §10). **Не строить раньше.**
- **Decoupling-слайс (Трек B)** — независим от who-pays/pricing, **может стартовать сейчас**: де-Stars + организатор-гейт доступа к клубу (honor-system, клон складчины). Детали — `docs/backlog/monetization-v2-model-handoff.md` §7 (B1–B11). Платформенный сбор пристёгивается позже через `PaymentProvider`-seam.

### 7.1 HARD-гейт перед реальным acquirer (из Security/Reviewer ревью Slice 1)
Когда стаб-`PaymentProvider` заменяется реальным (ЮKassa/Т-Банк), ОБЯЗАТЕЛЬНО до включения:
1. **Верификация подписи вебхука** (HMAC/RSA по raw-body, секрет из env, constant-time, + replay-окно по timestamp). Сейчас `/api/subscriptions/webhook` = permitAll и стаб возвращает `Ignored` (no-op) → риска нет; с реальным провайдером `handleWebhook` начнёт менять состояние по неаутентифицированному телу. Сеам готов (`X-Signature` уже прокидывается), но enforce нет. **CWE-345 / OWASP A07-A08.**
2. **Тайтовый rate-limit на `POST /api/subscriptions*`** (несколько/мин на юзера, как auth-bucket) — сейчас только общий 60/мин.
3. **Over-capacity при НЕдобровольном `ENDED`** (провал продления → `PAST_DUE→ENDED`) — обработчик freeze/архив (данные целы). Добровольный путь уже защищён гардом §4.3.
4. **TOCTOU на capacity-гарде** (`COUNT` без лока): при реальных деньгах закрыть advisory-локом/`SELECT … FOR UPDATE` на подписке владельца. Сейчас риск низкий (один юзер, double-submit; first-time race уже даёт чистый 409).

### 7.2 Долг Slice 2 (de-Stars), не блокер Slice 1
- В форме создания клуба ещё висит v1-копирайт «Цена подписки (Stars/мес)» + «зарабатывать N Stars (80%)» (`CreateClubModal`). Рублёвый сервисный сбор и Stars-формулировки сейчас на соседних экранах — снять в decoupling-слайсе.

## 8. Вне scope — legal/человек (нужна юр-консультация, design-док §11)
1. Юрлицо и налоговый режим (ИП vs ООО на УСН 6%).
2. Выбор acquirer (ЮKassa vs Т-Банк) — от него зависит tokenization Решения 3.
3. Формулировка оферты «сервисный сбор» (не переквалифицировать в агентскую комиссию).
4. 289-ФЗ «О платформенной экономике» (в силе 2026-10-01).
5. Для опциональной SPLIT-формы: 54-ФЗ чеки, НПД-статус, лимит 2,4 млн₽/год.
6. Пересчёт 22% НДС эквайринга (в силе 2026-01-01) — **почему точные цены §3.3 отложены**.
