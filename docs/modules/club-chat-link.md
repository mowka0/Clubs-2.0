## Данные

### Миграция V47: таблица `club_chat_links`, дроп `clubs.telegram_group_id`

```sql
CREATE TABLE club_chat_links (
    club_id           UUID PRIMARY KEY REFERENCES clubs(id) ON DELETE CASCADE,
    chat_id           BIGINT NOT NULL UNIQUE,        -- id группы/супергруппы Telegram
    chat_title        VARCHAR(255),                  -- название чата на момент привязки/обновления
    linked_by_user_id UUID NOT NULL REFERENCES users(id),
    linked_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    bot_status        VARCHAR(20) NOT NULL DEFAULT 'member',  -- administrator|member|left|kicked
    can_pin_messages  BOOLEAN NOT NULL DEFAULT FALSE,
    can_invite_users  BOOLEAN NOT NULL DEFAULT FALSE,
    door_enabled      BOOLEAN NOT NULL DEFAULT FALSE,
    door_invite_link  TEXT,                          -- ссылка createChatInviteLink(creates_join_request)
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

- Один клуб = один чат, один чат = один клуб (PK + UNIQUE).
- Легаси `clubs.telegram_group_id` (V2/V10, ручной REST-эндпоинт `POST /{id}/link-group`,
  фронтом никогда не использовался): непустые значения переносятся в `club_chat_links`
  (bot_status='member', права false, linked_by = владелец клуба — состояние C «проверьте права»),
  колонка дропается, эндпоинт и методы репозитория удаляются.
- Все таблицы/колонки — с русскими `COMMENT ON` (DoD миграции).

## Слайс 1 — Привязка

### Флоу
1. Владелец в табе «Чат» (состояние A) жмёт «Привязать чат» → фронт открывает
   `https://t.me/<bot>?startgroup=<club_id>&admin=pin_messages+invite_users+restrict_members`
   (URL приходит с бэка в `ChatLinkStatusDto.startGroupUrl` — username бота живёт на сервере;
   restrict_members нужен, чтобы снимать баны — см. «Дверь»).
2. Telegram открывает пикер групп; после добавления бота клиент шлёт в группу
   `/start <club_id>` от имени добавившего.
3. `ClubsBot.consume` (сообщение в group/supergroup, `/start` с payload) → `ChatLinkService.link`:
   - payload парсится как UUID клуба; клуб существует и активен;
   - **гейт безопасности**: `telegram_id` отправителя `/start` = владелец клуба
     (`clubs.owner_id`). Иначе — сообщение в чат «Привязать чат может только владелец клуба…»
     и бот выходит из чата (`LeaveChat`);
   - конфликты: клуб уже привязан к другому чату ИЛИ чат уже привязан к другому клубу →
     объясняющее сообщение + выход;
   - `GetChatMember(chat, botId)` → фиксируем `bot_status`, `can_pin_messages`, `can_invite_users`;
   - INSERT в `club_chat_links`;
   - **invite-ссылка (`creates_join_request`) создаётся сразу при привязке** (если есть право
     приглашать; иначе — как только право появится) — по ней работает кнопка «Чат клуба»,
     не дожидаясь тумблера двери.
4. **Петля подтверждения** (решение PO): бот пишет в чат «✅ Чат привязан к клубу „X“»
   и DM владельцу «Чат „Y“ привязан к клубу „X“. Это были вы?» с inline-кнопкой
   **[Отвязать чат]** (`callback_data = chatlink:unlink:<club_id>`). Callback обрабатывается
   только если жмёт владелец клуба.

### Health-мониторинг (`my_chat_member`)
Обновление статуса самого бота в привязанном чате (приходит из коробки, проверено в notes.md):
- `left`/`kicked` → `bot_status` обновляется, фичи гаснут (UI — состояние C). Привязка НЕ удаляется
  (вернут бота — всё оживёт после «Проверить ещё раз»/refresh).
- `administrator`/`member` → обновляем `bot_status` и флаги прав.
- **Пересоздание invite-ссылки**: Telegram отзывает ссылки удалённого админа, поэтому на переходе
  «бот снова может приглашать» (вернулся в чат / вернули право) старая ссылка отзывается и
  создаётся свежая. Срабатывает из `my_chat_member` И из повторного `/start` (порядок апдейтов
  не гарантирован; двойного пересоздания нет — условие сравнивает со строкой ДО обновления).
  При миграции group→supergroup ссылки старой группы умирают — пересоздаётся в migrate-хендлере.
- **Повторный `/start <club_id>` в уже привязанном чате** — идемпотентен: освежает права,
  пересоздаёт ссылку при необходимости и шлёт в чат ТО ЖЕ сообщение, что при первой привязке
  («✅ Чат привязан…»). DM-петля подтверждения не дублируется.
- Смена названия чата ловится при refresh (`GetChat`).
- Миграция group→supergroup: `message.migrate_to_chat_id` → обновляем `chat_id`.

### REST API (все — только владелец клуба, 403 иначе)
| Метод | Путь | Что делает |
|---|---|---|
| GET | `/api/clubs/{id}/chat-link` | Статус: `{ linked, chatTitle, linkedAt, botStatus, canPinMessages, canInviteUsers, doorEnabled, doorInviteLink, startGroupUrl }` |
| POST | `/api/clubs/{id}/chat-link/refresh` | Перечитать `GetChat` + `GetChatMember`, обновить строку, вернуть статус («Проверить права ещё раз») |
| PATCH | `/api/clubs/{id}/chat-link` | `{ doorEnabled?: boolean, livePinEnabled?: boolean }` — тумблеры двери (слайс 2) и живого закрепа (слайс 3); меняются только присланные поля |
| DELETE | `/api/clubs/{id}/chat-link` | Отвязать: revoke invite link (если есть), бот выходит из чата, строка удаляется |

### UI: таб «Чат» в «Управлении»
Табы: `Статистика · Финансы · Чат · Настройки`. Состояния по мокапу 01:
- **A (не привязан)**: карточка-объяснялка (лестница ценности), CTA «Привязать чат» →
  `openTelegramLink(startGroupUrl)`, хинт про права.
- **B (привязан, здоров)**: карточка чата (название, «привязан {дата}»), пиллы прав
  (`✓ бот в чате`, `✓/✕ закреп разрешён`, `✓/✕ приглашения разрешены`); в карточке — invite-ссылка
  с кнопкой «Копировать» и подписью «Данная ссылка уже активна и работает, поменяйте старую,
  если где-то её используете» (видна всегда, когда ссылка есть — независимо от двери). Тумблеры:
  «Живой закреп» (слайс 3 — требует права закрепа), «Вход в чат через заявки» (слайс 2),
  «Строгий режим» — «скоро» (disabled). Кнопка «Отвязать чат» (с подтверждением).
- **C (деградация)**: алерт что сломалось (бот кикнут / права отняты), пиллы с ✕;
  при кикнутом боте прямо под алертом — кнопка «Привязать бота заново» (тот же startgroup
  deep link). Кнопка «Проверить права ещё раз» (refresh), «Отвязать чат».

## Слайс 2 — Дверь

### Принцип
Членство в чате = членство в клубе. Бот не может добавить человека в группу принудительно
(нет такого Bot API) — поэтому вход всегда через `chat_join_request`, который бот одобряет.

### Тумблер «Вход в чат через заявки» (PATCH doorEnabled=true)
Тумблер = **политика для ЧУЖИХ**. Invite-ссылка живёт НЕЗАВИСИМО от него (создаётся при
привязке, V48 фиксирует семантику комментами):
- Включение требует `can_invite_users` (409 с объяснением); переиспользует ссылку привязки,
  создаёт только если её ещё нет.
- Выключение НЕ отзывает ссылку — по ней продолжает работать кнопка «Чат клуба» и разосланные DM.
- Участников с доступом бот впускает по заявке ВСЕГДА, независимо от тумблера.
- Организатор сам публикует эту ссылку вместо старой (шаг «закрой чат» ЕМУ объясняем, но не делаем
  за него — Bot API не умеет менять настройки группы).

### `chat_join_request` (бот получает, когда человек стучится по door-ссылке)
Ключ: `chat_join_request` даёт боту законное право написать человеку в личку один раз.
По telegram_id ищем пользователя и его membership в привязанном клубе:
- **active / cancelled-в-периоде** → `ApproveChatJoinRequest` сразу (человек уже свой) —
  НЕЗАВИСИМО от тумблера двери (через эту ветку работает кнопка «Чат клуба»).
- **frozen / expired** (должник) → заявка остаётся висеть; DM: «Ты в клубе, но доступ ещё не
  открыт — оплати взнос организатору, и я впущу автоматически» + кнопка на страницу клуба.
- **не участник / неизвестный аккаунт** → заявка висит; DM по мокапу 02-B: «Чат принадлежит клубу X.
  Вступление через клуб: 1) подай заявку 2) организатор одобрит — я сразу впущу» +
  кнопка «Открыть клуб» (WebApp-кнопка на `/clubs/{id}`).
- Если дверь выключена — заявки ЧУЖИХ и должников игнорируются (организатор разбирает их сам
  в клиенте Telegram); участники с доступом впускаются всегда (см. выше).

### Доступ открылся → впустить в чат
Новое доменное событие `MembershipAccessOpenedEvent(clubId, userId)` публикуется в:
- `MembershipActivator.activate(frozen=false)` — бесплатное вступление (join, approve, complete-free);
- `AccessGateService.markDuesPaid` — взнос подтверждён (frozen→active, expired→active, продление);
- `AccessGateService.unfreezeAccess` — разморозка.

AFTER_COMMIT-листенер (паттерн `EventBotNotifier`): если у клуба привязан чат, бот жив и
ссылка существует (НЕЗАВИСИМО от тумблера двери) →
1. `ApproveChatJoinRequest(chat, telegramId)` — если человек уже стучался (ошибку
   «нет заявки» глотаем); стучавшийся попадает в чат вообще без действий;
2. иначе `GetChatMember`: точно уже в чате → ничего (защита от DM-спама при продлениях);
   **забанен (`kicked`)** → `UnbanChatMember(only_if_banned)` перед приглашением — «удалить из
   группы» в Telegram = бан, забаненному любая ссылка «недействительна» (главный корень бага
   со staging, USER_KICKED). Требует права «Блокировка пользователей» (restrict_members);
3. далее DM «Организатор открыл тебе доступ. Вступай в чат клуба» + `door_invite_link`
   (тап → join request → авто-одобрение по ветке active, п.1 — UX «нажал и в чате»).
   При неизвестном статусе (Telegram молчит) DM шлём: потерять вход хуже лишнего DM.

### Отказ/исключение → закрыть заявку в чат
- `ApplicationService.rejectApplication` → событие → `DeclineChatJoinRequest` (блайндовый вызов,
  ошибки глотаем). Существующий DM с причиной отказа не трогаем.
- `AccessGateService.rejectMember` / `removeMember` → тоже decline висящей заявки.
  (Бан в чате уже-состоящих — слайс 5 «строгий режим», здесь НЕ делаем.)

### Витрина на странице клуба
`ClubDetailDto` дополняется:
- `chatLinked: Boolean` — чат привязан и бот жив (`bot_status` in administrator|member);
- `chatDoorEnabled: Boolean`;
- `chatInviteLink: String?` — invite-ссылка, отдаётся ТОЛЬКО участникам с доступом
  (active / cancelled-в-периоде) и владельцу; гостям/frozen/expired — null (least exposure,
  как `paymentLink`). БЕЗ условия «дверь включена» — кнопка «Чат клуба» работает сразу
  после привязки.

UI (мокап 02-C):
- гость/заявитель при `chatLinked && chatDoorEnabled`: чип «💬 У клуба есть чат — вход
  откроется после одобрения заявки»;
- участник с доступом при `chatInviteLink`: кнопка «💬 Чат клуба» → `openTelegramLink(ссылка)`
  (уже в чате → Telegram просто откроет чат; нет → заявка + авто-впуск). Доступна сразу после
  привязки, дверь включать не обязательно.

## Слайс 3 — Живой закреп + итог встречи

> Мокап: `docs/design/club-chat-link/mockups/03-live-pin-and-summary.html`.
> Принцип: у бота ОДНО закреплённое сообщение-статус на событие, которое он редактирует
> (метка «изменено» вместо ленты дублей), и один пост-итог после встречи. Всё остальное — тишина.

### Данные

**Миграция V49**: колонка `club_chat_links.live_pin_enabled BOOLEAN NOT NULL DEFAULT FALSE`
(тумблер) + таблица «какие сообщения бот ведёт в чате по событию»:

```sql
CREATE TABLE event_chat_pins (
    event_id           UUID PRIMARY KEY REFERENCES events(id) ON DELETE CASCADE,
    chat_id            BIGINT NOT NULL,      -- чат, куда постился статус (снимок на момент поста)
    message_id         BIGINT,               -- id сообщения-статуса (NULL = пост не создавался/не удался)
    closed_at          TIMESTAMPTZ,          -- NULL = закреп живой (редактируется); NOT NULL = закрыт
    summary_message_id BIGINT,               -- id поста-итога (NULL = итог ещё не постился — дедуп-гейт)
    created_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Тумблер «Живой закреп» (PATCH livePinEnabled)

`UpdateChatLinkRequest` становится частичным: `{ doorEnabled?: boolean, livePinEnabled?: boolean }`
(меняются только присланные поля; фронт шлёт по одному).

- **Включение** требует бота в чате + `can_pin_messages` (409 с объяснением — зеркалит дверь).
  Сразу **backfill**: для всех БУДУЩИХ неотменённых событий клуба (upcoming/stage_2,
  `event_datetime > now`) без живого закрепа бот постит статус и закрепляет — организатору
  не нужно ждать следующего события, чтобы увидеть фичу.
- **Выключение**: живые закрепы открепляются, их строки удаляются (сообщения остаются в истории
  чата обычными постами — права удалять сообщения у бота нет и не просим). Закрытые строки
  (прошедшие события, итоги) не трогаются. Повторное включение создаёт свежие пины.
- Потеря права закрепа ПОСЛЕ включения: тумблер не сбрасывается, бот продолжает редактировать
  уже созданные закрепы (редактирование своего сообщения права не требует), новые пины не
  закрепятся (пост уйдёт без pin). UI показывает алерт деградации (как у двери).

### Жизненный цикл сообщения-статуса

| Триггер | Действие |
|---|---|
| `EventCreatedEvent` (AFTER_COMMIT) | если чат привязан, бот жив и `live_pin_enabled` → пост-статус Этапа 1 + pin (`disable_notification`, без пуша всем) + строка в `event_chat_pins` |
| Голос Этапа 1 (`castVote`), подтверждение/отказ Этапа 2, промоут из очереди (обе ветки: отказ и выход из клуба) | новое доменное событие `EventRosterChangedEvent(eventId)` / существующий `WaitlistPromotedEvent` → **dirty-флаг** в памяти |
| `Stage2StartedEvent` | dirty-флаг → при ближайшем flush сообщение перерисуется в режим «гонка за места» |
| Flush-планировщик (`chatlink.live-pin-flush-ms`, дефолт 30 000) | раз в период: все dirty-события перерисовываются из БД одним `editMessageText` на событие — дебаунс ≥30 сек гарантирован периодом; ошибка «message is not modified» глотается |
| Событие стартовало (тот же flush, close-проход по `event_datetime <= now`) | финальный edit «Сбор закрыт · подтверждено N из L» + **unpin** + `closed_at` |
| `EventCancelledEvent` | немедленный edit «❌ событие отменено (+причина)» + unpin + `closed_at` |

Текст сообщения (plain text, как все DM бота; время — МСК, формат `dd.MM.yyyy HH:mm МСК`):

- **Этап 1**: `📅 {title}` / `🗓 {дата} · 📍 {место}` / `✅ Идут — N` / `🤔 Возможно — M` /
  `👥 Мест — {limit}` + url-кнопка **«Проголосовать»**.
- **Этап 2**: `🏁 {title} — подтверждение мест` / `🗓 …` / `✅ Подтвердили — N из {limit}` /
  `📋 В очереди — K` / `⏳ Подтвердить до — {старт события}` + url-кнопка **«Подтвердить участие»**.
  (Дедлайн = старт события — честная граница окна подтверждения на бэке; отказ от
  подтверждённого места закрывается раньше, T-4ч, но в закрепе показываем именно окно
  подтверждения.)
- ⚠️ Кнопки — **только url** `https://t.me/<bot>/app?startapp=event_<uuid>` (WebApp-кнопки в
  группах запрещены Telegram; `DeepLinkHandler` фронта уже парсит `event_*`).

### Итог после встречи

Триггер: `AttendanceMarkedEvent` (первая отметка явки организатором — тёплый момент «вечером
после встречи», а не финализация через 48ч). Гейты: чат привязан, бот жив, `live_pin_enabled`,
итог ещё не постился (`summary_message_id IS NULL` — повторные отметки явки итог не дублируют).
Обычный пост (НЕ в закреп):

```
Встреча №{N} прошла ✅          ← N = все прошедшие неотменённые события клуба
👥 Пришли — {X} из {Y}          ← X = отмечены present, Y = подтверждённые Этапа 2
🎉 {Имя} — впервые на встрече клуба   ← участники, у кого это первый present в клубе (если есть)
Следующая — {дата}: {title}     ← ближайшее будущее событие клуба (если есть)
```

- url-кнопка «Иду на следующую» → `startapp=event_<nextId>` — только если следующее событие есть.
- Имена «впервые» — именительный падеж без склонения («Наташа и Марк — впервые…»), максимум 3,
  дальше «и ещё k».
- «Серия клуба N встреч подряд» из мокапа — **НЕ входит** в этот слайс: метрики «серия» нет в
  модели данных (осознанный gap, вернёмся при геймификации качества клуба).

### Известные ограничения (осознанные)

- Dirty-флаги живут в памяти — рестарт бэкенда теряет несброшенные правки; следующее изменение
  ростера перерисует закреп. Close-проход не зависит от dirty (сканирует БД) — закрытие
  закрепов надёжно.
- Выход участника из клуба обновляет закреп только для событий, где у него была подтверждённая
  бронь (через `WaitlistPromotedEvent`/`EventRosterChangedEvent`); его голоса Этапа 1 в других
  событиях исчезают из счётчиков при следующем любом изменении ростера.
- Как и вся чат-механика — best-effort: сбой Telegram логируется и не валит бизнес-операцию.

## Безопасность
- Верификация владельца при привязке — обязательный гейт (см. флоу). Одноразовые коды отвергнуты PO.
- Все REST-эндпоинты — ownership-check (`club.ownerId == caller`), 403 иначе.
- Callback `chatlink:unlink` — проверка, что жмёт владелец (по telegram_id → user → owner).
- `door_invite_link` не светится гостям; никаких секретов в логах (chat_id/club_id — ок).
- Payload deep link — только UUID (парсинг с отказом на мусор), никакого исполнения ввода.
- Fail-open запрещён: при недоступности Telegram API привязка/дверь падают с понятной ошибкой,
  членство в клубе НЕ блокируется (DM/approve — best-effort, ошибки логируются).

## Критерии приёмки (staging, ручной тест PO)
Привязка:
1. Таб «Чат» виден владельцу в «Управлении» (4-й таб между «Финансы» и «Настройки» — порядок: Статистика·Финансы·Чат·Настройки).
2. «Привязать чат» открывает пикер Telegram; после добавления бота в группу: сообщение в чате
   «Чат привязан…», DM владельцу с кнопкой «Отвязать», таб показывает состояние B с пиллами прав.
3. Не-владелец добавляет бота по чужой ссылке → бот отказывает и выходит.
4. Кик бота из группы → таб показывает C-алерт; возврат бота + «Проверить ещё раз» → B.
5. «Отвязать чат» → бот выходит из группы, таб снова A. Кнопка [Отвязать] в DM делает то же.
Дверь:
6. Тумблер требует «приглашения разрешены»; при включении появляется door-ссылка.
7. Чужой человек по door-ссылке → заявка висит, ему приходит DM бота с кнопкой на клуб.
8. Организатор одобряет заявку в приложении (бесплатный клуб) → человек автоматически в чате + DM.
9. Платный клуб: одобрение заявки НЕ впускает (frozen); «Взнос получен» → впускает.
10. Участник с доступом видит кнопку «Чат клуба»; гость видит чип; frozen — ничего.
11. Отказ по заявке → висящая chat-заявка отклоняется.
Живой закреп:
12. Тумблер «Живой закреп» активен при `✓ закреп разрешён`; включение при будущем событии →
    в чате появляется закреплённый статус с голосами и кнопкой «Проголосовать».
13. Голос в приложении → закреп обновляется в пределах ~30 сек (метка «изменено», без нового поста).
14. Переход события в Этап 2 → то же сообщение становится «подтверждение мест» (подтверждено/
    очередь/дедлайн), кнопка «Подтвердить участие».
15. Создание нового события при включённом тумблере → бот сам постит и закрепляет статус.
16. Старт события → закреп открепляется с финальным текстом «Сбор закрыт…».
17. Отмена события → закреп открепляется с текстом «событие отменено».
18. Организатор отмечает явку → бот постит итог (№ встречи, «пришли X из Y», «впервые» — если есть,
    следующее событие + кнопка). Повторная отметка явки итог НЕ дублирует.
19. Выключение тумблера → живые закрепы открепляются; включение обратно → свежие пины.
20. Кнопки в группе открывают Mini App на странице события (url t.me/<bot>/app?startapp=event_…).

## Рамки/факты Telegram (проверено, notes.md)
- `my_chat_member` и `chat_join_request` приходят при пустом `allowed_updates` (наш стартер).
- `chat_member` (чужие статусы) — НЕ приходит и не нужен в слайсах 1–2.
- Бот не может: добавить юзера в группу, закрыть чат «по заявкам» за организатора.
- `?startgroup=<payload>&admin=…` — payload ≤64 символа [A-Za-z0-9_-], UUID проходит.
