## Данные

### Миграция V47: таблица `club_chat_links`, дроп `clubs.telegram_group_id`

```sql
CREATE TABLE club_chat_links (
    club_id           UUID PRIMARY KEY REFERENCES clubs(id) ON DELETE CASCADE,
    chat_id           BIGINT NOT NULL UNIQUE,        -- id группы/супергруппы Telegram
    chat_title        VARCHAR(255),                  -- название чата на момент привязки/обновления
    linked_by_user_id UUID NOT NULL REFERENCES users(id),
    linked_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    bot_status        VARCHAR(20) NOT NULL DEFAULT 'member',  -- administrator|member|left|kicked
    can_pin_messages  BOOLEAN NOT NULL DEFAULT FALSE,
    can_invite_users  BOOLEAN NOT NULL DEFAULT FALSE,
    door_enabled      BOOLEAN NOT NULL DEFAULT FALSE,
    door_invite_link  TEXT,                          -- ссылка createChatInviteLink(creates_join_request)
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

- Один клуб = один чат, один чат = один клуб (PK + UNIQUE).
- Легаси `clubs.telegram_group_id` (V2/V10, ручной REST-эндпоинт `POST /{id}/link-group`,
  фронтом никогда не использовался): непустые значения переносятся в `club_chat_links`
  (bot_status='member', права false, linked_by = владелец клуба — состояние C «проверьте права»),
  колонка дропается, эндпоинт и методы репозитория удаляются.
- Все таблицы/колонки — с русскими `COMMENT ON` (DoD миграции).

## Слайс 1 — Привязка

### Флоу
1. Владелец в табе «Чат» (состояние A) жмёт «Привязать чат» → фронт открывает
   `https://t.me/<bot>?startgroup=<club_id>&admin=pin_messages+invite_users+restrict_members+manage_tags`
   (URL приходит с бэка в `ChatLinkStatusDto.startGroupUrl` — username бота живёт на сервере;
   restrict_members нужен, чтобы снимать баны — см. «Дверь»).
2. Telegram открывает пикер групп; после добавления бота клиент шлёт в группу
   `/start <club_id>` от имени добавившего.
3. `ClubsBot.consume` (сообщение в group/supergroup, `/start` с payload) → `ChatLinkService.link`:
   - payload парсится как UUID клуба; клуб существует и активен;
   - **гейт безопасности**: `telegram_id` отправителя `/start` = владелец клуба
     (`clubs.owner_id`). Иначе — отказ (см. § «Отказ в привязке: уходить или остаться»);
   - конфликты: клуб уже привязан к другому чату ИЛИ чат уже привязан к другому **живому**
     клубу → отказ по тем же правилам;
   - **осиротевшая строка** (за строкой стоит удалённый клуб) чат НЕ занимает и освобождается
     на месте — см. § «Перехват чата с осиротевшей привязкой»;
   - `GetChatMember(chat, botId)` → фиксируем `bot_status`, `can_pin_messages`, `can_invite_users`;
   - INSERT в `club_chat_links`;
   - **invite-ссылка (`creates_join_request`) создаётся сразу при привязке** (если есть право
     приглашать; иначе — как только право появится) — по ней работает кнопка «Чат клуба»,
     не дожидаясь тумблера двери.
4. **Петля подтверждения** (решение PO): бот пишет в чат «✅ Чат привязан к клубу „X“»
   и DM владельцу «Чат „Y“ привязан к клубу „X“. Это были вы?» с inline-кнопкой
   **[Отвязать чат]** (`callback_data = chatlink:unlink:<club_id>`). Callback обрабатывается
   только если жмёт владелец клуба.
5. **Приглашение сидящим в чате** (фидбек PO 2026-07-08): следом бот постит в чат приглашение
   с url-кнопкой **«Вступить в клуб»** (`t.me/<bot>?startapp=club_<uuid>`, DeepLinkHandler
   фронта парсит `club_*`) — чат мог существовать до клуба, зовём его участников. Только при
   ПЕРВИЧНОЙ привязке: повторный `/start` (возврат кикнутого бота) приглашение не дублирует.

### Отказ в привязке: уходить или остаться (`bugfix/chat-link-deadlock`, 2026-07-20)

> Было (слайсы 1–5): любой отказ = сообщение в чат + `LeaveChat`. Это ломало чужую работающую
> интеграцию — посторонний одним `/start` с произвольным payload'ом выгонял бота из чата,
> который обслуживал живой клуб (дверь, закрепы, теги, баны умирали вместе с ним).

**Правило:** бот покидает чат **только если за чатом нет живой привязки**. «Живая привязка» =
строка `club_chat_links` этого `chat_id`, за которой стоит **видимый** клуб (`findById` его
находит; строка удалённого клуба живой не считается — см. § «Перехват…»).

Два независимых решения на каждом отказе:

| Решение | Правило |
|---|---|
| **Выйти из чата** | Да — только если чат свободен (живой привязки нет). Чат с живой привязкой отказ разрушать не имеет права. |
| **Написать в чат** | Да — всегда, **кроме** случая «отправитель не верифицированный владелец запрашиваемого клуба И чат держит живую привязку». Иначе (бот уйти уже не может) любой участник группы гонял бы `/start` с произвольным UUID и превращал бота в спамера чужого чата. |

Верифицированному владельцу (гейт владельца пройден) сообщение приходит **всегда** — иначе он
не поймёт, почему привязка не проходит.

Матрица всех веток отказа:

| # | Ветка отказа | Отправитель верифицирован? | Чат свободен → | Чат держит живую привязку → |
|---|---|---|---|---|
| 1 | Клуб не найден / неактивен | нет (проверка не дошла) | сообщение + выход | молча остаётся |
| 2 | Отправитель не владелец клуба | нет | сообщение + выход | молча остаётся |
| 3 | У клуба уже привязан другой чат | да | сообщение + выход | сообщение, остаётся |
| 4 | Чат занят другим живым клубом | да | — (ветка недостижима: чат занят ⇒ не свободен) | сообщение, остаётся |
| 5 | Перехват сироты, отправитель не админ чата | да | сообщение + выход | — (сирота ⇒ живой привязки нет) |

Каждый отказ логируется `WARN`: `requestedClubId`, `chatId`, `verifiedOwner`, `botLeftChat`,
`chatOccupiedByClubId`.

> **Чат с осиротевшей строкой считается свободным** для решения «уходить ли»: живого клуба за
> ним нет, значит ломать нечего — бот уходит и пишет отказ на общих основаниях. Саму
> осиротевшую строку отказ **не трогает** — она освобождается только при **успешном** перехвате
> (все гейты пройдены), иначе неудачная попытка чужого клуба стирала бы чужую историю чата.

**Тексты отказов (точные):**

| Ветка | Текст |
|---|---|
| Клуб не найден / неактивен | «Клуб не найден. Откройте «Управление клубом» в приложении Clubs и нажмите «Привязать чат» ещё раз.» |
| Не владелец | «Привязать чат может только владелец клуба в приложении Clubs.» |
| У клуба уже есть чат | «У клуба «X» уже привязан другой чат. Сначала отвяжите его в «Управлении клубом».» |
| Чат занят живым клубом | «Этот чат уже привязан к другому клубу. Один чат — один клуб: сначала отвяжите его в том клубе, «Управление» → «Чат».» |
| Перехват сироты не-админом чата | «Этот чат раньше принадлежал другому клубу. Перепривязать его может только администратор чата в Telegram.» |

> **Название клуба в тексте «не владелец» отсутствует намеренно** (security): отказ уходит ДО
> проверки прав, и подстановка названия подтверждала бы существование приватного клуба любому
> обладателю его UUID. В ветках 3–5 отправитель уже верифицирован как владелец — там имя клуба
> не утечка.

### Перехват чата с осиротевшей привязкой

**Сирота** — строка `club_chat_links`, чей клуб soft-deleted. Возникала до
`bugfix/chat-link-deadlock`, когда удаление клуба строку не трогало: чат оставался занят
навсегда — клуб скрыт `is_active`-фильтром, `findById` его не видит, отвязать из приложения
нечем (эндпоинты chat-link владельческие и ходят через `findById`). Теперь удаление клуба
чат освобождает (см. § «Освобождение чата при удалении клуба»), но **самолечение оставлено**
для строк, накопившихся до фикса. Отдельной миграции нет: на проде сирот нет (проверено SQL),
на staging — одна.

Поведение при `/start <new_club_id>` в чате с сиротой:
1. Гейт владельца нового клуба — как обычно.
2. **Дополнительный гейт (security):** отправитель обязан быть `creator`/`administrator`
   **этого чата** в Telegram (`ChatTelegramGateway.isChatAdmin`). **Fail-closed:** Telegram не
   ответил / статус не получен = «не админ» = отказ.
   *Почему:* сироты живут ровно там, где бот сидит админом, и перехват отдаёт новому клубу все
   права над чатом (инвайты, закрепы, мьюты, баны, теги). Без гейта рядовой участник легаси-группы
   завёл бы свой клуб и увёл группу под себя, а настоящие админы чата отменить это не смогли бы —
   отвязка владельческая.
3. Гейт пройден → строка сироты освобождается через `releaseKeepingBotInChat` (та же чистка,
   что при отвязке: закрепы, статус-посты сборов, мьюты, баны, теги, revoke invite-ссылки,
   удаление строки) — **но бот из чата НЕ выходит**: его только что добавили ради новой привязки,
   выход её сорвал бы. Логируется `WARN` с `staleClubId` / `chatId` / `newClubId`.
4. Дальше — обычная первичная привязка (INSERT, invite-ссылка, сообщения, DM-петля).

> **Гейт админа чата НЕ распространяется на обычную привязку** (сознательно): там Telegram сам
> гейтит того, кто добавляет бота, а лишняя проверка сломала бы легальный сценарий «бота добавили
> обычным участником» (слайс 1, состояние C «проверьте права»).

### Health-мониторинг (`my_chat_member`)
Обновление статуса самого бота в привязанном чате (приходит из коробки, проверено в notes.md):
- `left`/`kicked` → `bot_status` обновляется, фичи гаснут (UI — состояние C). Привязка НЕ удаляется
  (вернут бота — всё оживёт после «Проверить ещё раз»/refresh).
- `administrator`/`member` → обновляем `bot_status` и флаги прав.
- **Пересоздание invite-ссылки**: Telegram отзывает ссылки удалённого админа, поэтому на переходе
  «бот снова может приглашать» (вернулся в чат / вернули право) старая ссылка отзывается и
  создаётся свежая. Срабатывает из `my_chat_member` И из повторного `/start` (порядок апдейтов
  не гарантирован; двойного пересоздания нет — условие сравнивает со строкой ДО обновления).
  При миграции group→supergroup ссылки старой группы умирают — пересоздаётся в migrate-хендлере.
- **Повторный `/start <club_id>` в уже привязанном чате** — идемпотентен: освежает права,
  пересоздаёт ссылку при необходимости и шлёт в чат ТО ЖЕ сообщение, что при первой привязке
  («✅ Чат привязан…»). DM-петля подтверждения не дублируется.
- Смена названия чата ловится при refresh (`GetChat`).
- Миграция group→supergroup: `message.migrate_to_chat_id` → обновляем `chat_id`.

### REST API (все — только владелец клуба, 403 иначе)
| Метод | Путь | Что делает |
|---|---|---|
| GET | `/api/clubs/{id}/chat-link` | Статус: `{ linked, chatTitle, linkedAt, botStatus, canPinMessages, canInviteUsers, canRestrictMembers, canManageTags, doorEnabled, doorInviteLink, livePinEnabled, skladchinaStatusEnabled, strictModeEnabled, awardTagsEnabled, startGroupUrl }` |
| POST | `/api/clubs/{id}/chat-link/refresh` | Перечитать `GetChat` + `GetChatMember`, обновить строку, вернуть статус («Проверить права ещё раз») |
| PATCH | `/api/clubs/{id}/chat-link` | `{ doorEnabled?, livePinEnabled?, skladchinaStatusEnabled?, strictModeEnabled?, awardTagsEnabled? }` — частичные тумблеры (дверь / живой закреп / статус сборов / строгий режим / теги наград); меняются только присланные поля |
| DELETE | `/api/clubs/{id}/chat-link` | Отвязать: полное освобождение чата (см. § «Освобождение чата») + бот выходит из чата |

### Освобождение чата (единый механизм)

Одна процедура (`ChatLinkService.releaseLink`) обслуживает три входа. Отличаются они **только**
тем, выходит ли бот из чата:

| Вход | Кто зовёт | Бот выходит? |
|---|---|---|
| Владельческая отвязка (REST `DELETE` + кнопка [Отвязать] в DM) | `doUnlink` | ✅ да |
| **Удаление клуба** (`DELETE /api/clubs/{id}`) | `releaseOnClubDeleted(clubId)` | ✅ да |
| Перехват чата с осиротевшей привязкой | `releaseKeepingBotInChat` | ❌ нет (бота только что добавили ради новой привязки) |

Состав чистки (одинаковый для всех трёх, порядок значим — всё делается, **пока бот ещё в чате**):
1. снять живые закрепы событий (`livePinService.disableForClub`);
2. снять статус-посты складчин (`skladchinaChatStatusService.disableForClub`);
3. вернуть голос замьюченным должникам строгого режима (unmute);
4. снять ВСЕ учтённые баны строгого режима (`chat_strict_bans`) — после ухода бота впустить
   забаненного будет некому;
5. снять теги наград (`memberTagService.disableForClub`);
6. отозвать door-invite-ссылку;
7. `LeaveChat` — **только** если вход этого требует (таблица выше);
8. удалить строку `club_chat_links`.

Telegram-шаги best-effort: строка удаляется в любом случае, иначе мёртвая привязка блокировала
бы повторную. Лог: `Chat link released: clubId={} chatId={} botLeftChat={}`.

### Освобождение чата при удалении клуба (`bugfix/chat-link-deadlock`, 2026-07-20)

`ClubService.deleteClub` вызывает `chatLinkService.releaseOnClubDeleted(id)` **до**
`clubRepository.softDelete(id)` — чтобы освобождение работало на ещё видимом клубе. Порядок
относительно остальных каскадов (события / складчины / заявки) значения не имеет: закрепы и
статус-посты снимаются по `chat_id`, а не по статусу активностей. Нет привязки — no-op.

Счётчик `chatUnlinked` (boolean) добавлен в лог удаления клуба —
см. `docs/modules/clubs.md` § «DELETE /api/clubs/{id}».

> **Почему это обязательно, а не «шероховатость v1»:** без освобождения чат навсегда числился бы
> за скрытым клубом. Отвязать нечем (владельческие эндпоинты ходят через `findById`, который
> скрытый клуб не видит), а повторная привязка упиралась бы в «Этот чат уже привязан к другому
> клубу» — тупик, из-за которого заведён этот багфикс.

  `openTelegramLink(startGroupUrl)`, хинт про права.

  если где-то её используете» (видна всегда, когда ссылка есть — независимо от двери). Тумблеры:
  «Живой закреп» (слайс 3 — требует права закрепа), «Вход в чат через заявки» (слайс 2),
  «Строгий режим» — «скоро» (disabled). Кнопка «Отвязать чат» (с подтверждением).

## Слайс 2 — Дверь

### Принцип
Членство в чате = членство в клубе. Бот не может добавить человека в группу принудительно
(нет такого Bot API) — поэтому вход всегда через `chat_join_request`, который бот одобряет.

### Тумблер «Вход в чат через заявки» (PATCH doorEnabled=true)
Тумблер = **политика для ЧУЖИХ**. Invite-ссылка живёт НЕЗАВИСИМО от него (создаётся при
привязке, V48 фиксирует семантику комментами):
- Включение требует `can_invite_users` (409 с объяснением); переиспользует ссылку привязки,
  создаёт только если её ещё нет.
- Выключение НЕ отзывает ссылку — по ней продолжает работать кнопка «Чат клуба» и разосланные DM.
- Участников с доступом бот впускает по заявке ВСЕГДА, независимо от тумблера.
- Организатор сам публикует эту ссылку вместо старой (шаг «закрой чат» ЕМУ объясняем, но не делаем
  за него — Bot API не умеет менять настройки группы).

### `chat_join_request` (бот получает, когда человек стучится по door-ссылке)
Ключ: `chat_join_request` даёт боту законное право написать человеку в личку один раз.
По telegram_id ищем пользователя и его membership в привязанном клубе:
- **active / cancelled-в-периоде** → `ApproveChatJoinRequest` сразу (человек уже свой) —
  НЕЗАВИСИМО от тумблера двери (через эту ветку работает кнопка «Чат клуба»).
- **frozen / expired** (должник) → заявка остаётся висеть; DM: «Ты в клубе, но доступ ещё не
  открыт — оплати взнос организатору, и я впущу автоматически» + кнопка на страницу клуба.
- **не участник / неизвестный аккаунт** → заявка висит; DM по мокапу 02-B: «Чат принадлежит клубу X.
  Вступление через клуб: 1) подай заявку 2) организатор одобрит — я сразу впущу» +
  кнопка «Открыть клуб» (WebApp-кнопка на `/clubs/{id}`).
- Если дверь выключена — заявки ЧУЖИХ и должников игнорируются (организатор разбирает их сам
  в клиенте Telegram); участники с доступом впускаются всегда (см. выше).

### Доступ открылся → впустить в чат
Новое доменное событие `MembershipAccessOpenedEvent(clubId, userId)` публикуется в:
- `MembershipActivator.activate(frozen=false)` — бесплатное вступление (join, approve, complete-free);
- `AccessGateService.markDuesPaid` — взнос подтверждён (frozen→active, expired→active, продление);
- `AccessGateService.unfreezeAccess` — разморозка.

AFTER_COMMIT-листенер (паттерн `EventBotNotifier`): если у клуба привязан чат, бот жив и
ссылка существует (НЕЗАВИСИМО от тумблера двери) →
1. `ApproveChatJoinRequest(chat, telegramId)` — если человек уже стучался (ошибку
   «нет заявки» глотаем); стучавшийся попадает в чат вообще без действий;
2. иначе `GetChatMember`: точно уже в чате → зависит от `wasAccessClosed` в событии
   (фидбек PO 2026-07-08): доступ ДО этого был закрыт (вступление, разморозка, взнос
   frozen/expired-должника) → DM «Организатор открыл тебе доступ — ты уже в чате»
   (кейс кика: кик из клуба не убирает из чата, и после повторного вступления человек
   не получал ничего); продление при живом доступе → ничего (защита от DM-спама);
   **забанен (`kicked`)** → `UnbanChatMember(only_if_banned)` перед приглашением — «удалить из
   группы» в Telegram = бан, забаненному любая ссылка «недействительна» (главный корень бага
   со staging, USER_KICKED). Требует права «Блокировка пользователей» (restrict_members);
3. далее DM «Организатор открыл тебе доступ. Вступай в чат клуба» + `door_invite_link`
   (тап → join request → авто-одобрение по ветке active, п.1 — UX «нажал и в чате»).
   При неизвестном статусе (Telegram молчит) DM шлём: потерять вход хуже лишнего DM.

### Отказ/исключение → закрыть заявку в чат
- `ApplicationService.rejectApplication` → событие → `DeclineChatJoinRequest` (блайндовый вызов,
  ошибки глотаем). Существующий DM с причиной отказа не трогаем.
- `AccessGateService.rejectMember` / `removeMember` → тоже decline висящей заявки.
  (Бан в чате уже-состоящих — слайс 5 «строгий режим», см. ниже.)

### Витрина на странице клуба
`ClubDetailDto` дополняется:
- `chatLinked: Boolean` — чат привязан и бот жив (`bot_status` in administrator|member);
- `chatDoorEnabled: Boolean`;
- `chatInviteLink: String?` — invite-ссылка, отдаётся ТОЛЬКО участникам с доступом
  (active / cancelled-в-периоде) и владельцу; гостям/frozen/expired — null (least exposure,
  как `paymentLink`). БЕЗ условия «дверь включена» — кнопка «Чат клуба» работает сразу
  после привязки.

UI (мокап 02-C):
- гость/заявитель при `chatLinked && chatDoorEnabled`: чип «💬 У клуба есть чат — вход
  откроется после **одобрения заявки**» для закрытого клуба (`accessType === 'closed'`) и
  «…после **вступления**» для открытого — хвост фразы зависит от типа доступа, чтобы не обещать
  гостю несуществующую модерацию (`ClubPage.tsx:559-566`);
- участник с доступом при `chatInviteLink`: кнопка «💬 Чат клуба» → `openTelegramLink(ссылка)`
  (уже в чате → Telegram просто откроет чат; нет → заявка + авто-впуск). Доступна сразу после
  привязки, дверь включать не обязательно.

### Панель подключения чата на странице клуба

> Дизайн утверждён PO 2026-07-20. Источник истины по вёрстке и текстам:
> `docs/design/club-chat-banner/mockups/variant-c-fox-card.html` (вариант C — «лис-карточка»;
> отвергнутые варианты A/B лежат рядом). Функционально относится к слайсу 1 (привязка),
> но описана здесь — рядом с остальным UI страницы клуба.
>
> **As-built (2026-07-20, ветка `feature/empty-states-wave-3`):** компонент
> `frontend/src/components/club/ClubChatConnectBanner.tsx` (презентационный, роль-гейт снаружи),

> тесты — `frontend/src/test/pages/ClubPageChatBanner.test.tsx` (9 шт.).

#### Цель

Организатор, который на экране «Клуб создан» нажал «Позже», сейчас узнаёт о чат-интеграции
только если сам зайдёт в «Управление → Чат». Панель — второй, ненавязчивый заход: она стоит
на самом посещаемом экране клуба, объясняет ценность тремя строками и ведёт в тот же таб «Чат».
Метрика смысла: доля клубов с привязанным чатом (привязка повышает выживаемость клуба —
обоснование то же, что у CTA на экране создания).

#### Scope

**Входит:** одна презентационная карточка на странице клуба; переход в существующий таб «Чат»;
локальное скрытие «Позже в настройках». Новых эндпоинтов, полей DTO и миграций нет.

**НЕ входит:** привязка чата прямо со страницы клуба (deep link `startgroup` остаётся в табе
«Чат» — там же живут объяснение прав и состояния A/B/C); серверное хранение факта скрытия;
показ панели со-организатору; аналитика показов/кликов; вариации текста под тип клуба.

#### User Story

**Как** владелец клуба без привязанного чата
**Я хочу** видеть на странице клуба короткое объяснение, что даёт подключение чата, и кнопку к нему
**Чтобы** не искать функцию в «Управлении» и подключить чат, когда до этого дойдут руки.

#### Размещение и вёрстка

- Позиция: на странице клуба (`ClubPage`) **под хиро, до секции «О клубе»**. У владельца
  read-only заметка «Подписка отменена» не рендерится (`!isOwner` в её условии), поэтому
  конфликта порядка нет.
- Контейнер: карточка `rd-glass`, внутренний паддинг `14px 16px`, `overflow: hidden`.
- Раскладка: флекс-строка, `align-items: center`. Слева арт лиса фиксированной ширины **124px**,
  справа текстовый блок (`flex: 1`, `text-align: left`). **Все три зазора равны 16px**:
  граница→лис, лис→текст, текст→граница (в мокапе — токен `--pad`, он же `gap` и горизонтальный
  паддинг). Лис центрирован по вертикали панели.
- Арт: `frontend/src/assets/mascot/fox-chat.png`.
- Заголовок — **одной строкой** (`white-space: nowrap`, кегль 15px/700): иконка
  `frontend/src/assets/icons/telegram-plane.png` шириной **17px** + текст. Иконка — PNG из
  брендового логотипа (единый вид на всех платформах), не эмодзи.
- **Выравнивание текста — по левому краю у всей колонки** (решение PO 2026-07-20): заголовок,
  буллеты и кнопки делят одну вертикаль слева. Промежуточный вариант «центрированный заголовок
  над левыми буллетами» отвергнут — читался как рассинхрон. По центру свободного места стоит
  колонка целиком, за это отвечают равные зазоры панели, а не `text-align`.
  Маркер «•» — акцентным цветом.
- Кнопки в столбик (`gap: 7px`). Акцентная — **переиспользует общий `rd-btn-primary`**
  (он и есть `--accent-grad`), локально ужат только паддинг до `10px 14px`: штатные `15px 20px`
  в тесной колонке панели выглядят раздутыми. Ghost — **собственный класс
  `.rd-chat-panel-ghost`** (прозрачный фон, рамка `--glass-border`, `width: 100%`), а не общий
  `rd-ghost-btn`: тот заливной (`--surface`) и не тянется на всю ширину, то есть в столбике
  встал бы уже акцентной кнопки. Отклонения от мокапа — только стилевые, вёрстка совпадает.

#### Гейт видимости

Панель рендерится **тогда и только тогда**, когда одновременно:

1. `isOwner` — `club.ownerId === user.id` (`ClubPage.tsx:121`);
2. `club.chatLinked === false`;
3. для этого `clubId` нет отметки о скрытии.

**Почему `isOwner`, а не `isManager` (проверено по коду).** Все эндпоинты `chat-link` —
владельческие: `ChatLinkService.requireOwner` (`chatlink/ChatLinkService.kt:303`) кидает
`ForbiddenException("Only the club owner can manage the chat link")`, и гейт стоит на **каждом**
методе (getStatus/refresh/update/unlink и всех операциях слайсов). Капабилити `MANAGE_CHAT`
объявлена в `common/auth/ClubCapability.kt:60` в блоке «Владельческие (только у organizer, НЕ делегируются)»
и на этих эндпоинтах не используется — со-организатор права привязки не получает ни по одной
ветке. Соответственно UI уже так и построен: `OrganizerClubManage.tsx:519-523` скрывает таб
«Чат» у со-орга и откатывает его `?tab=chat` на «Статистику» с комментарием «таб с
гарантированным 403 хуже скрытого». Панель по `isManager` дала бы со-организатору кнопку
«Подключить», которая ведёт на таб, которого у него нет, — тупик. Гейт: **`isOwner`**.

**Почему `club.chatLinked`, а не `useChatLinkStatusQuery`.** Поле уже приходит в
`ClubDetailDto` (`club/ClubDto.kt:54`, заполняется в `ClubService.getClub`) и уже используется на
этой же странице для кнопки «Чат клуба» — отдельный запрос был бы вторым источником правды и
лишним раундтрипом на самом посещаемом экране. Важнее другое: `ClubPage` **до** рендера тела
делает ранний выход по `clubQuery.isPending` (спиннер) и по `clubQuery.error || !club`
(`Placeholder`) — `ClubPage.tsx:146-159`. Значит панель структурно не может отрисоваться ни во
время загрузки, ни при ошибке: «мигания» и показа поверх ошибки (урок F5-20) не существует по
построению. Запрос `useChatLinkStatusQuery` такой гарантии не даёт — он загружался бы отдельно
от страницы, и панель либо мигала бы, либо требовала ручной обработки трёх состояний.

Семантика `chatLinked` на бэке — `chatLink?.botStatus?.isInChat == true`
(`club/ClubService.kt:146`), то есть «привязан **и бот в чате живой**». Прямое следствие: если бота
кикнули из группы, `chatLinked` становится `false` и панель может вернуться (при не-скрытом
флаге) с текстом «Подключи чат клуба». Это принято: CTA ведёт в таб «Чат», где владельца
встречает состояние C с точной диагностикой («Бот удалён из чата» + «Привязать бота заново»),
так что тупика нет — только формально неточный заголовок. Зафиксировано в открытых вопросах.

#### Скрытие «Позже в настройках»

- Скрытие **постоянное для этого клуба**, не «до перезагрузки»: организатор, осознанно
  отложивший подключение, не должен видеть один и тот же баннер при каждом заходе на страницу
  своего клуба. Потерять фичу при этом нельзя — постоянный путь остаётся
  «Управление → Чат», и ghost-кнопка прямо на него указывает («Позже **в настройках**»).
- Хранилище: `localStorage`, ключ `clubs:chat-banner-dismissed:<clubId>` — по клубу, а не
  глобально: у организатора несколько клубов, отложив один, он должен получить предложение
  во втором.
- **Чтение и запись обязательно в `try/catch`** — в части клиентов Telegram (приватный режим,
  веб-вью с отключённым storage) доступ к `localStorage` бросает. Паттерн уже применён в
  `store/useThemeStore.ts:26-31,57` и `components/CityPicker.tsx:70,87-90`.
- Значение по ключу — `'1'`, но смысл несёт **само наличие ключа**: чтение проверяет
  `getItem(...) !== null`, а не равенство строке. Любое непустое значение считается «скрыто».
- **Оба слоя скрытия привязаны к `clubId`, а не к монтированию компонента.** `ClubPage`
  отрисовывает один и тот же элемент маршрута `/clubs/:id` — при переходе между клубами
  (A → B) React **не перемонтирует** ни страницу, ни панель, меняется только параметр. Поэтому:
  отметка из хранилища перечитывается на смену `clubId` (`useMemo(..., [clubId])`), а скрытие
  текущей сессии хранит **id скрытого клуба**, а не булево (`dismissedClubId === clubId`).
  Булев флаг «скрыто», выставленный один раз, протёк бы на соседний клуб: организатор нажал
  «Позже» в клубе A, перешёл в клуб B — и панели там бы не увидел (нарушение AC-50).
- **Поведение при недоступном `localStorage`:** чтение в `catch` возвращает «не скрыто» —
  панель показывается; нажатие «Позже» скрывает панель через локальное состояние компонента,
  запись молча не удаётся. Такое скрытие живёт, пока `ClubPage` смонтирована (переходы между
  клубами его не сбрасывают — оно тоже привязано к `clubId`); уход на другой экран и возврат
  на страницу клуба вернут панель. Это осознанная деградация: неработающая кнопка хуже, чем
  кнопка с коротким эффектом; ошибку пользователю не показываем.
- Флаг скрытия **не сбрасывается** ни при привязке, ни при отвязке чата, ни при смене
  владельца — единственный способ сброса — очистка данных приложения. Отдельного UI сброса нет
  (YAGNI: путь «Управление → Чат» доступен всегда).
- Флаг проверяется **последним**: при `chatLinked = true` панель скрыта независимо от флага
  (условие 2 гейта), поэтому «панель исчезает сама после привязки» работает и для тех, кто
  «Позже» не нажимал.

#### Non-functional

- **Доступность:** у арта лиса — осмысленный alt (`Лис-маскот`); у иконки телеграм-самолётика
  alt пустой (`alt=""`) — она декоративна, смысл несёт соседний текст. Обе кнопки — настоящие
  `<button type="button">`, не `div` с `onClick`: иначе теряются фокус, Enter/Space и
  озвучивание роли скринридером.
- **Хаптика** (как во всей волне 3 empty-states): акцентная «Подключить» — `impact('medium')`
  (совпадает с CTA «Привязать чат» в табе «Чат», `components/manage/ClubChatTab.tsx:63`); ghost «Позже» —
  `impact('light')`.
- **Производительность:** нулевая сетевая стоимость — ни запросов, ни новых полей; арт
  импортируется как статический ассет (Vite хеширует и кеширует).
- **Логирование:** событий нет — панель полностью клиентская.
- **Безопасность:** новых поверхностей нет. Панель не показывает никаких данных о чате
  (ни `chatInviteLink`, ни chat_id) — только факт «чат не подключён», и только владельцу.
  Скрытие клиентское, поэтому его подделка ничего не даёт: доступ к привязке по-прежнему
  гейтится серверным `requireOwner`.

#### Интеграции

- **`ClubDetailDto` / `queryKeys.clubs.detail(clubId)`** — источник `chatLinked` и `ownerId`.
  Отдельного query-ключа фича не заводит. После привязки чата в табе «Чат» мутации
  `chatLink.ts` уже инвалидируют `queryKeys.clubs.detail(clubId)`
  (`queries/chatLink.ts:28,54`) — возврат на страницу клуба покажет её без панели
  автоматически, без ручного refetch.
- **Навигация** — `react-router` на `/clubs/{clubId}/manage?tab=chat` (`OrganizerClubManage`,
  `resolveInitialTab`). Ассеты — `assets/mascot/fox-chat.png`, `assets/icons/telegram-plane.png`.
- **Стили** — токены Banco-Plata из `styles/brand-theme.css`, классы карточки — в
  `styles/redesign.css` рядом с прочими `rd-*`.

#### Критерии приёмки (staging)

45. **Показ владельцу.** GIVEN я владелец клуба, у которого чат не привязан, WHEN открываю
    страницу клуба, THEN под хиро и до секции «О клубе» вижу карточку с лисом, заголовком
    «Подключи чат клуба», тремя буллетами и кнопками «Подключить» / «Позже в настройках».
46. **Переход по CTA.** GIVEN панель показана, WHEN жму «Подключить», THEN попадаю на
    «Управление» клубом с **открытым** табом «Чат» (состояние A с кнопкой «Привязать чат»),
    а не на «Статистику».
47. **Со-организатор не видит (негатив).** GIVEN я активный со-организатор клуба без
    привязанного чата, WHEN открываю страницу клуба, THEN панели нет (и таба «Чат» в
    «Управлении» у меня тоже нет).
48. **Участник и гость не видят (негатив).** GIVEN я участник / должник / гость клуба без

49. **Скрытие постоянное.** GIVEN панель показана, WHEN жму «Позже в настройках», THEN панель

    полного перезапуска Mini App она не появляется; AND «Управление → Чат» по-прежнему
    открывается и предлагает привязку.
50. **Скрытие — только для этого клуба.** GIVEN я скрыл панель в клубе A, WHEN открываю
    страницу другого своего клуба B без привязанного чата, THEN в клубе B панель показана.
51. **Исчезновение после привязки.** GIVEN панель показана (или была скрыта), WHEN я привязываю
    чат и возвращаюсь на страницу клуба, THEN панели нет; AND вместо неё доступна кнопка
    «💬 Чат клуба».
52. **Нет мигания и нет показа поверх ошибки.** GIVEN медленная сеть, WHEN открываю страницу
    клуба, THEN до окончания загрузки виден спиннер и панель не «подмигивает»; AND при ошибке
    загрузки клуба виден экран ошибки без панели.
53. **Заголовок одной строкой.** GIVEN панель показана на узком экране (360 px), WHEN смотрю
    на заголовок, THEN иконка и текст «Подключи чат клуба» стоят в одну строку, панель не
    даёт горизонтальной прокрутки, буллеты выровнены по левому краю единым блоком.

Покрытие: AC 45–51 закрыты автотестами `frontend/src/test/pages/ClubPageChatBanner.test.tsx`
(показ владельцу с текстами и кнопками; переход с `?tab=chat`; негативы со-организатор /

#### Риски и открытые вопросы

- **Арт `fox-chat.png` — временный.** Сгенерирован в nano banana (тот же пайплайн, что у
  остальных лис-сцен — `docs/backlog/empty-states-handoff.md` §5). Сейчас в файле кремовая
  («бежевая») версия бумажных самолётиков; PO ждёт голубую. Замена — **подмена файла по тому
  же пути, без правок кода** (имя, размеры и импорт не меняются). Приёмку панели это не
  блокирует.
- **Заголовок при кикнутом боте.** `chatLinked = false` означает и «не привязан», и
  «привязан, но бот вылетел» (см. § «Гейт видимости»). Во втором случае владельцу, который
  панель не скрывал, покажется «Подключи чат клуба» вместо «Верните бота». Принято как
  осознанная неточность: CTA всё равно ведёт в таб «Чат» с точным алертом состояния C.
  Разделить состояния можно только отдельным полем в `ClubDetailDto` — не делаем (YAGNI,
  ситуация редкая).
- **Ключ `localStorage` расходится с существующими по стилю.** В проекте уже два разных
  разделителя: `clubs.cityChoice` (точка) и `clubs-theme-mode` (дефис). Для панели утверждён
  третий вариант — `clubs:chat-banner-dismissed:<clubId>` (двоеточие как разделитель
  пространства имён и параметра). Унификация ключей — отдельная косметическая задача, в scope
  этой фичи не входит.
- **Скрытие не переносится между устройствами и слетает при очистке данных** — следствие
  клиентского хранения. Серверное поле (`clubs.chat_banner_dismissed_at`) было бы честнее, но
  это миграция и эндпоинт ради баннера — отложено; вернуться, если PO увидит жалобы
  «баннер вернулся».
- **Слепая зона измерения:** аналитики показов/кликов нет, поэтому эффект панели на долю
  клубов с привязанным чатом придётся оценивать косвенно — по динамике самой доли.

## Слайс 3 — Живой закреп + итог встречи

> Мокап: `docs/design/club-chat-link/mockups/03-live-pin-and-summary.html`.
> Принцип: у бота ОДНО закреплённое сообщение-статус на событие, которое он редактирует
> (метка «изменено» вместо ленты дублей), и один пост-итог после встречи. Всё остальное — тишина.

### Данные

**Миграция V49**: колонка `club_chat_links.live_pin_enabled BOOLEAN NOT NULL DEFAULT FALSE`
(тумблер) + таблица «какие сообщения бот ведёт в чате по событию»:

```sql
CREATE TABLE event_chat_pins (
    event_id           UUID PRIMARY KEY REFERENCES events(id) ON DELETE CASCADE,
    chat_id            BIGINT NOT NULL,      -- чат, куда постился статус (снимок на момент поста)
    message_id         BIGINT,               -- id сообщения-статуса (NULL = пост не создавался/не удался)
    closed_at          TIMESTAMPTZ,          -- NULL = закреп живой (редактируется); NOT NULL = закрыт
    summary_message_id BIGINT,               -- id поста-итога (NULL = итог ещё не постился — дедуп-гейт)
    created_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Тумблер «Живой закреп» (PATCH livePinEnabled)

`UpdateChatLinkRequest` становится частичным: `{ doorEnabled?: boolean, livePinEnabled?: boolean }`
(меняются только присланные поля; фронт шлёт по одному).

- **Включение** требует бота в чате + `can_pin_messages` (409 с объяснением — зеркалит дверь).
  Сразу **backfill**: для всех БУДУЩИХ неотменённых событий клуба (upcoming/stage_2,
  `event_datetime > now`) без живого закрепа бот постит статус и закрепляет — организатору
  не нужно ждать следующего события, чтобы увидеть фичу.
- **Выключение**: живые закрепы открепляются, их строки удаляются (сообщения остаются в истории
  чата обычными постами — права удалять сообщения у бота нет и не просим). Закрытые строки
  (прошедшие события, итоги) не трогаются. Повторное включение создаёт свежие пины.
- Потеря права закрепа ПОСЛЕ включения: тумблер не сбрасывается, бот продолжает редактировать
  уже созданные закрепы (редактирование своего сообщения права не требует), новые пины не
  закрепятся (пост уйдёт без pin). UI показывает алерт деградации (как у двери).

### Жизненный цикл сообщения-статуса

| Триггер | Действие |
|---|---|
| `EventCreatedEvent` (AFTER_COMMIT) | если чат привязан, бот жив и `live_pin_enabled` → пост-статус Этапа 1 + pin (`disable_notification`, без пуша всем) + строка в `event_chat_pins` |
| Голос Этапа 1 (`castVote`), подтверждение/отказ Этапа 2, промоут из очереди (обе ветки: отказ и выход из клуба) | новое доменное событие `EventRosterChangedEvent(eventId)` / существующий `WaitlistPromotedEvent` → **dirty-флаг** в памяти |
| `Stage2StartedEvent` | dirty-флаг → при ближайшем flush сообщение перерисуется в режим «гонка за места» |
| Flush-планировщик (`chatlink.live-pin-flush-ms`, дефолт 30 000) | раз в период: все dirty-события перерисовываются из БД одним `editMessageText` на событие — дебаунс ≥30 сек гарантирован периодом; ошибка «message is not modified» глотается |
| Событие стартовало (тот же flush, close-проход по `event_datetime <= now`) | финальный edit «Событие началось — подтвердили N из L» + **unpin** + `closed_at` (не «Сбор закрыт»: слово «сбор» занято складчиной, путало — фидбек PO 2026-07-08) |
| `EventCancelledEvent` | немедленный edit «❌ событие отменено (+причина)» + unpin + `closed_at` + **отдельный громкий пост об отмене** (правки беззвучны; после маршрутизации пост — единственный пинг чата, DM об отмене уходит только тем, кого нет в чате) |

Текст сообщения (plain text, как все DM бота; время — МСК, формат `dd.MM.yyyy HH:mm МСК`):

- **Этап 1**: `📅 {title}` / `🗓 {дата} · 📍 {место}` / `✅ Идут — N` / `🤔 Возможно — M` /
  `👥 Мест — {limit}` + url-кнопка **«Проголосовать»**. Открытая встреча (V62):
  вместо строки мест — `👥 Открытая встреча — без репутации и лимита мест`.
- **Этап 2**: `🏁 {title} — подтверждение мест` / `🗓 …` / `✅ Подтвердили — N из {limit}` /
  `📋 В очереди — K` / `⏳ Подтвердить до — {старт события}` + url-кнопка **«Подтвердить участие»**.
  Открытая встреча: заголовок `— подтверждение участия`, счёт `✅ Подтвердили — N` без
  знаменателя, строка очереди не рендерится (waitlist недостижим).
  (Дедлайн = старт события — честная граница окна подтверждения на бэке; отказ от
  подтверждённого места закрывается раньше, T-4ч, но в закрепе показываем именно окно
  подтверждения.)
- ⚠️ Кнопки — **только url** `https://t.me/<bot>?startapp=event_<uuid>` (Main Mini App):
  WebApp-кнопки в группах запрещены Telegram, а формат `t.me/<bot>/app?…` требует отдельной
  регистрации short name через /newapp, которой у ботов нет («Bot App Not Found» на staging).
  **Требование инфраструктуры:** у бота должен быть включён Mini App
  (BotFather → Bot Settings → Configure Mini App → Enable, URL = базовый URL фронта) — и у
  staging-бота (`clubs_v2_test_bot` → staging URL), и у прод-бота (`clubs_v2_bot` → прод URL);
  без этого ссылка молча открывает чат с ботом. `DeepLinkHandler` фронта парсит `event_*`
  из `startParam`.

### Итог после встречи

Триггер: `AttendanceMarkedEvent` (первая отметка явки организатором — тёплый момент «вечером
после встречи», а не финализация через 48ч). Гейты: чат привязан, бот жив, `live_pin_enabled`,
итог ещё не постился (`summary_message_id IS NULL` — повторные отметки явки итог не дублируют).
Обычный **тихий** пост (`disable_notification`, решение PO 2026-07-08: итог — фоновая сводка,
пуш всем участникам чата не нужен), НЕ в закреп:

```
Встреча №{N} прошла ✅          ← N = все прошедшие неотменённые события клуба
👥 Пришли — {X} из {Y}          ← X = отмечены present, Y = подтверждённые Этапа 2
🎉 {Имя} — впервые на встрече клуба   ← участники, у кого это первый present в клубе (если есть)
Следующая — {дата}: {title}     ← ближайшее будущее событие клуба (если есть)
```

- url-кнопка «Иду на следующую» → `startapp=event_<nextId>` — только если следующее событие есть.
- Имена «впервые» — именительный падеж без склонения («Наташа и Марк — впервые…»), максимум 3,
  дальше «и ещё k».
- «Серия клуба N встреч подряд» из мокапа — **НЕ входит** в этот слайс: метрики «серия» нет в
  модели данных (осознанный gap, вернёмся при геймификации качества клуба).

### Известные ограничения (осознанные)

- Dirty-флаги живут в памяти — рестарт бэкенда теряет несброшенные правки; следующее изменение
  ростера перерисует закреп. Close-проход не зависит от dirty (сканирует БД) — закрытие
  закрепов надёжно.
- Выход участника из клуба обновляет закреп только для событий, где у него была подтверждённая
  бронь (через `WaitlistPromotedEvent`/`EventRosterChangedEvent`); его голоса Этапа 1 в других
  событиях исчезают из счётчиков при следующем любом изменении ростера.
- Как и вся чат-механика — best-effort: сбой Telegram логируется и не валит бизнес-операцию.

## Слайс 3.5 — Живой статус сбора (складчины)

> Идея PO 2026-07-08 (при приёмке слайса 3, рамки в notes.md): для складчин чат — гарантированный
> канал (DM доходит только после /start у бота), участников без username можно упоминать через
> `text_mention` (`tg://user?id=…`). Тон нейтральный, НЕ коллектор: максимум 1–2 пинга на сбор,
> публичный текст без упоминания штрафов.

### Принцип

У бота ОДНО сообщение-статус на складчину (как живой закреп у события): прогресс «Скинулись N из M»,
дедлайн и строка «Ждём: @…» с text_mention-упоминаниями ещё не ответивших. Сообщение редактируется
с дебаунсом при каждом изменении прогресса; редактирования НИКОГО не пингуют (механика Telegram —
уведомление создаёт только НОВОЕ сообщение с упоминанием). Пинг-бюджет складчины:

1. **Пинг №1** — сам пост при создании сбора (упоминания pending-участников в «Ждём:»).
2. **Пинг №2** — напоминание за ~24ч до дедлайна (только «важные сборы» — тот же гейт
   `affects_reputation`, что у существующего DM-шедулера), отдельным сообщением.

### Данные

**Миграция V50**: колонка `club_chat_links.skladchina_status_enabled BOOLEAN NOT NULL DEFAULT FALSE`
(тумблер) + таблица «какое сообщение-статус бот ведёт по складчине»:

```sql
CREATE TABLE skladchina_chat_posts (
    skladchina_id UUID PRIMARY KEY REFERENCES skladchinas(id) ON DELETE CASCADE,
    chat_id       BIGINT NOT NULL,      -- чат на момент поста (снимок, как event_chat_pins)
    message_id    BIGINT NOT NULL,      -- id сообщения-статуса (строка создаётся только после успешного поста)
    closed_at     TIMESTAMPTZ,          -- NULL = живой (редактируется); NOT NULL = закрыт
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Поля `summary_message_id` нет — у складчины нет поста-итога, финал = последний edit самого статуса.
Неудачная отправка поста строку не создаёт (повторная попытка — при следующем включении
тумблера/backfill); гонка backfill × создание складчины гасится `INSERT … ON CONFLICT DO NOTHING`.

### Тумблер «Статус сборов в чате» (PATCH skladchinaStatusEnabled)

Отдельный от «Живого закрепа» тумблер: публичные упоминания — социально-чувствительная фича,
организатор включает её осознанно. `UpdateChatLinkRequest` расширяется третьим частичным полем:
`{ doorEnabled?, livePinEnabled?, skladchinaStatusEnabled? }`.

- **Включение** требует только бота в чате (409 «верните бота» иначе) — посты, редактирования и
  упоминания прав администратора не требуют. Pin — best-effort при `can_pin_messages` (как у
  живого закрепа: нет права → пост уходит без закрепа). Сразу **backfill**: статус-пост для всех
  АКТИВНЫХ складчин клуба без живого поста.
- **Выключение**: живые посты открепляются, строки удаляются (сообщения остаются в истории);
  повторное включение создаёт свежие. Освобождение чата (`releaseLink` — отвязка / удаление
  клуба / перехват сироты) делает то же до выхода бота.
- Потеря права закрепа после включения ничего не ломает (редактирование своих сообщений
  права не требует).

### Жизненный цикл сообщения-статуса

| Триггер | Действие |
|---|---|
| `SkladchinaCreatedEvent` (AFTER_COMMIT, существующее) | тумблер включён + бот жив → пост-статус + pin (best-effort) + строка в `skladchina_chat_posts`. DM при создании **маршрутизируется** (решение PO 2026-07-08): пост вышел → личку получают только участники, которых нет в чате; платёжная ссылка для остальных — через кнопку «Открыть сбор» |
| Изменение прогресса: mark-paid / decline / орг-отметка / орг-снятие / одобрение отказа / заявка на отказ (сдвиг дедлайна) | новое доменное событие `SkladchinaProgressChangedEvent(skladchinaId)` → **dirty-флаг** в памяти |
| Выход участника из клуба / кик организатором | каскад `deleteParticipantFromActiveSkladchinasInClub` возвращает id затронутых складчин → то же `SkladchinaProgressChangedEvent` по каждой (ушедший — и особенно кикнутый — не должен висеть в «Ждём:») |
| Flush-планировщик (`chatlink.skladchina-status-flush-ms`, дефолт 30 000) | dirty-складчины перерисовываются из БД одним edit; ошибка «message is not modified» глотается |
| Close-проход (тот же flush) | открытые посты складчин со `status != active` → финальный edit + unpin + `closed_at`. **Обязателен для корректности**, не страховка: каскады `cancelActiveByClub` (удаление клуба) и `cancelActiveByEventId` (отмена события split_bill) минуют `closeInternal` и НЕ публикуют событий |
| `SkladchinaClosedEvent` (AFTER_COMMIT, существующее) | немедленный финальный edit + unpin + `closed_at` (не ждём flush) |

Текст живого статуса (HTML parse_mode — нужен для `text_mention`; заголовок и имена экранируются):

```
💰 {title}
👥 Скинулись — N из M
⏳ До {dd.MM.yyyy HH:mm МСК}

Ждём: {@меншены pending-участников}
```

+ url-кнопка **«Открыть сбор»** → `t.me/<bot>?startapp=skladchina_<uuid>` (DeepLinkHandler фронта
уже парсит `skladchina_*`). Когда pending нет — строка «Ждём» не рендерится.

- Упоминание: `<a href="tg://user?id={telegram_id}">{Имя}</a>` — работает без username; пингует
  только участников ЧАТА (не-участникам чата придёт обычный DM-фоллбек, см. напоминание).
- Максимум **15 упоминаний**, дальше «и ещё k» (стена меншенов хуже недопинга; осознанный компромисс).
  Имя в меншене обрезается до 64 символов — 15 имён по 255 символов пробили бы телеграмный
  лимит 4096 на сообщение, и пост молча бы не создался.
- Денег в посте нет (прогресс в людях — Фаза A складчины); платёжная ссылка остаётся в DM
  и приложении, в группе не светится.
- Публичный текст НЕ упоминает репутацию/−40 ни в статусе, ни в напоминании (северная звезда:
  без карательного тона; полный текст с ценой молчания живёт в личных DM).

Финальный edit при закрытии (нейтральный, БЕЗ списка неоплативших): `💰 {title}` /
`Сбор закрыт · скинулись N из M {✅ при closed_success}`; при `cancelled` — `Сбор отменён`.
Упоминания из финального текста убираются.

### Напоминание за 24ч: чат вместо DM (с фоллбеком)

Существующий `SkladchinaReminderScheduler` (реп-сборы, `reminder_sent_at` — та же дедупликация,
один прогон на складчину) меняет канал доставки:

1. Есть живой пост в чате (тумблер включён, бот жив, `message_id` есть) → **новое сообщение** в чат:
   `⏰ Напоминание: сбор «{title}» закрывается {дата}.` / `Ещё не ответили: {@меншены}` — пинг №2.
2. DM уходит ТОЛЬКО pending-участникам, которых **нет в чате** (`getChatMember` по каждому;
   `IN_CHAT` → покрыт меншеном, `NOT_IN_CHAT`/`BANNED`/`UNKNOWN` → DM как раньше, полный текст
   с ценой молчания). Никто не теряет напоминание, никто не получает его дважды.
3. Живого поста нет → DM всем pending, поведение до слайса.

### Известные ограничения (осознанные)

- Dirty-флаги в памяти — рестарт теряет несброшенные правки; следующее изменение прогресса
  перерисует. Close-проход от флагов не зависит (сканирует БД).
- Миграция group→supergroup не переносит `chat_id` открытых постов (снимок) — как у живого закрепа.
- Упоминания сверх лимита 15 не пингуются (получают «и ещё k»); в НАПОМИНАНИИ участники сверх
  лимита уходят в DM-фоллбек — «упомянутый = пингнутый» соблюдается, напоминание не теряется.
- Статус-пост упоминает всех pending-участников складчины, включая тех, кого нет в чате:
  их `telegram_id` виден участникам чата в entity `tg://user?id=…` (security-ревью: Low, принято —
  аудитория ограничена собственным чатом клуба, участники складчины и так члены этого клуба;
  фильтр по членству в чате стоил бы N getChatMember на каждую перерисовку).
- Гонка backfill × создание в узком окне может оставить в чате дубль-сообщение (не закреплённый,
  без строки в БД) — как у живого закрепа, best-effort.
- Telegram-вызовы backfill/disable выполняются внутри транзакции тумблера (паттерн живого
  закрепа) — при недоступности Telegram PATCH ждёт таймаутов; hardening отложен.
- Как вся чат-механика — best-effort: сбой Telegram логируется и не валит бизнес-операцию.

## Слайс 4 — Теги наград («бейджи»)

> Ступень «титулы» лестницы доверия (стратегия §4). Реализовано на **Member Tags** (Bot API 9.5,
> март 2026, фидбек PO): тег обычному участнику БЕЗ повышения в админы. Решения PO 2026-07-10:
> тег = **последняя выданная награда** (отзыв — откат к предыдущей); названия наград ограничены
> **16 символами** (лимит тега Telegram); «полная синхронность» тег↔награда шедулером.

### Принцип

Награда из клубного профиля (V40, косметика от организатора) видна в чате тегом рядом с именем:
`setChatMemberTag(chat_id, user_id, tag)` — участник остаётся ОБЫЧНЫМ участником. Боту нужно
право «Управление тегами» (`can_manage_tags`); тег ≤16 символов, без эмодзи; пустая строка
снимает тег. У Telegram один тег на человека — «последняя выданная» выбрана PO как актуальное
признание без дополнительного UI.

Со строгим режимом (слайс 5) теги НЕ конфликтуют: тегированный участник — не админ,
mute/ban работают на нём как на обычном. При уходе из клуба тег снимается (регалии клуба).

**Рамка библиотеки** (снята 2026-07-11): telegrambots обновлена до 9.5.0 —
`SetChatMemberTag`, поле `tag` и право `can_manage_tags` доступны типизированно;
прямые HTTP-вызовы (`rawApiCall`) из `ChatTelegramGateway` удалены.

### Данные

- **V54** (историческая, титульная механика — прожила только на staging) + **V55**
  (переименования на честные имена): `club_chat_links.award_tags_enabled` (тумблер),
  `can_manage_tags` (право; для строк до V55 — false до refresh/my_chat_member);
  `chat_award_tags (club_id FK→clubs CASCADE, telegram_id, tag VARCHAR(16), tagged_at;
  PK club_id+telegram_id)` — учёт тегов, выставленных НАМИ (для снятия при выключении/
  отвязке/уходе и дедупа). Теги, поставленные организатором руками или участником
  (при can_edit_tag в правах группы), в учёт не попадают и нами не снимаются.
- Валидация: `awards.label` — новые награды ≤16 символов (DTO @Size + сервис после
  нормализации пробелов); легаси длиннее живут как есть, их тег обрезается по 16
  (в проде легаси нет — 0 наград на момент внедрения).

### События (реактивная часть)

Новые доменные события `AwardGrantedEvent` / `AwardRevokedEvent(clubId, userId)` из
AwardService (AFTER_COMMIT, @Async, best-effort): пересчёт тега = label последней награды;
наград не осталось → тег снимается пустой строкой. Уход из клуба (`MembershipAccessRevokedEvent`)
снимает тег независимо от строгого режима.

### Шедулер полной синхронизации (требование PO: «полная синхронность»)

`TagSyncScheduler` (`chatlink.tag-sync-ms`, дефолт 30 000; fixedDelay — без наложений):
по каждому клубу с включённым тумблером (бот в чате, право есть) читает фактический тег
КАЖДОГО живого участника (`getChatMember`) и применяет правила PO:

| В клубе (последняя награда) | В чате (тег) | Действие |
|---|---|---|
| есть | нет | поставить тег из награды |
| любая | есть, ОТЛИЧАЕТСЯ от последней награды | **ручная правка главнее**: тег импортируется наградой (label = тег, эмодзи 🏷️, от имени владельца); существующая награда с тем же названием «переподнимается» в последние без дубля (уточнение PO со staging 2026-07-10) |
| есть | есть, совпадает | ничего |
| нет | нет | ничего |

Не-участники чата пропускаются (тег ставить некуда — появится тегом при вступлении в чат
на следующем тике). Сбой Telegram по участнику/клубу пропускает его, не валя сверку остальных.

### Тумблер «Теги наград» (PATCH awardTagsEnabled)

- **Включение** требует бота в чате + право «Управление тегами» (409 иначе). Сразу backfill:
  теги всем живым участникам с наградами.
- **Выключение / отвязка чата**: снять все теги из учёта.
- Потеря права ПОСЛЕ включения: тумблер не сбрасывается, действия не срабатывают (warn),
  UI — алерт деградации.
- UI: пятый пилл `✓/✕ теги разрешены`, тумблер между «Статусом сборов» и «Строгим режимом».
- Deep link привязки: `&admin=…+manage_tags` (если клиент Telegram ключ не поймёт — право
  выдаётся руками, refresh подтянет).

### Известные ограничения (осознанные)

- Право `can_manage_tags` не видно старой библиотеке — в `my_chat_member` дотягивается
  отдельным raw-вызовом; у привязок до V55 — false до «Проверить права ещё раз».
- Название награды с эмодзи ВНУТРИ текста Telegram отклонит при выставлении тега — тег
  не встанет (warn), награда в приложении живёт как есть.
- Самоназначенные теги участников (если организатор разрешил can_edit_tag в группе)
  импортируются шедулером как награды «от организатора» — буквальное следствие правила PO
  «тег есть, награды нет → создать награду».
- Импортированная награда получает эмодзи 🏷️ (у тега эмодзи не бывает) и авторство
  владельца клуба.
- Снятие тега пустой строкой предполагает семантику Bot API «пустой тег = снять» —
  проверяется на staging (док Bot API описывает параметр без нюансов).
- Как вся чат-механика — best-effort: сбой Telegram логируется и не валит награды/членство.

### Критерии приёмки (staging)

37. Тумблер «Теги наград» активен при пилле «теги разрешены»; включение при участнике
    с наградами → в чате у него появляется тег = последняя награда (участник НЕ становится
    админом чата).
38. Выдача новой награды → тег меняется на новую в пределах секунд; отзыв последней →
    откат к предыдущей; отзыв единственной → тег исчезает.
39. Форма выдачи награды не принимает название длиннее 16 символов.
40. Синхронизация клуб→чат: участник с наградой, вступивший в чат ПОЗЖЕ выдачи, получает
    тег в пределах ~30 сек (шедулер).
41. Синхронизация чат→клуб: организатор руками ставит тег участнику БЕЗ наград → в пределах
    ~30 сек в приложении появляется награда 🏷️ с текстом тега.
42. Ручной тег у участника с наградами → в пределах ~30 сек импортируется НОВОЙ наградой
    (или переподнимает существующую с тем же названием) — тег остаётся, стороны сходятся.
43. Строгий режим дружит с тегами: тегированный должник мьютится (тег остаётся), кик
    тегированного → тег снят, бан сработал.
44. Выключение тумблера → теги, поставленные ботом, сняты (ручные не тронуты); включение →
    backfill вернул. Освобождение чата (отвязка / удаление клуба) → то же перед выходом бота.

## Слайс 5 — Строгий режим

> Стратегия §4 («Завхоз прав», opt-in): приложение управляет физикой чата — заморозка за
> неуплату → mute, исключение → ban. Решения PO 2026-07-08 (эта сессия): backfill-мьют текущих
> должников при включении; бан за ЛЮБОЙ уход из клуба (кик, отказ должнику, отклонение заявки,
> добровольный выход, истёкшая отменённая подписка) — жёсткая версия «членство в чате =
> членство в клубе»; отдельного DM при мьюте НЕТ (должник уже получает DM «доступ закрыт /
> истёк» от существующих флоу, ограничение в чате Telegram показывает сам).

### Принцип

Два рычага, оба действуют только при включённом тумблере и живом боте:
- **Mute (read-only)** — должник (`frozen` — ждёт первого взноса, `expired` — просрочил
  продление) остаётся в чате, но не может писать (`restrictChatMember`, все permissions=false,
  бессрочно). Оплата (`markDuesPaid` / разморозка / «своя дата») возвращает голос.
- **Ban** — человек, покинувший клуб любым путём, вылетает из чата (`banChatMember`).
  Бан НЕ вечный приговор: повторное вступление в клуб снимает его существующим unban-флоу
  двери (`UnbanChatMember(only_if_banned)` при открытии доступа, слайс 2).

Организатор (owner) под рычаги не попадает никогда: его membership не бывает frozen/expired
(`loadManageableMember` защищает роль organizer), а Telegram в любом случае не даёт боту
ограничивать админов чата (ошибка глотается, warn в лог).

### Данные

**Миграция V52**: две колонки в `club_chat_links` (+ русские `COMMENT ON`):
- `strict_mode_enabled BOOLEAN NOT NULL DEFAULT FALSE` — тумблер;
- `can_restrict_members BOOLEAN NOT NULL DEFAULT FALSE` — право «Блокировка пользователей»
  (зеркалит can_pin/can_invite; уже запрашивается в deep link привязки с багфикса USER_KICKED).
  Для чатов, привязанных ДО V52, значение false до первого «Проверить права ещё раз» или
  любого `my_chat_member`-перехода — организатору достаточно нажать refresh.

**Миграция V53** (фидбек PO со staging 2026-07-08): таблица учёта банов
`chat_strict_bans (club_id FK→clubs CASCADE, telegram_id, banned_at; PK club_id+telegram_id)` —
кого бот забанил. Нужна отвязке: после ухода бота впустить забаненного больше некому, поэтому
освобождение чата (`releaseLink` — любой из трёх входов, см. § «Освобождение чата») снимает ВСЕ
учтённые баны (независимо от текущего состояния тумблера) и чистит учёт.
Записывается только при УСПЕШНОМ бане; чистится при открытии доступа (повторное вступление,
независимо от тумблера) и при отвязке. Ручные баны организатора в учёт не попадают и не снимаются.

Отдельной таблицы «кого замьютили» НЕТ (осознанно): множество замьюченных детерминировано =
«должники клуба» (frozen/expired), снятие при выключении тумблера/отвязке идёт по нему же.

### События

| Событие | Публикуется в | Реакция строгого режима (AFTER_COMMIT, @Async, best-effort) |
|---|---|---|
| `MembershipAccessClosedEvent` (**новое**) | `AccessGateService.freezeAccess`; `MembershipActivator.activate(frozen=true)` (approve платного — новичок-должник); `SubscriptionLifecycleService.processExpiry` (по каждой строке active→expired) | mute |
| `MembershipAccessOpenedEvent` (существующее) | без изменений (activateFree, unfreeze, markDuesPaid, setAccessUntil) | unmute (снятие ограничения только если юзер сейчас `restricted` — обычного member не трогаем) |
| `MembershipAccessRevokedEvent` (существующее + новые публикации) | было: `rejectApplication`, `rejectMember`, `removeMember`; добавляется: `leaveFreeClub`, `leavePaidClub`/`cancelMembership` без живого оплаченного окна (мгновенный уход), шедулер по cancelled-строкам с только что истёкшим окном | ban (декларация двери — decline висящей заявки — сохраняется как раньше и работает независимо от тумблера) |

Шедулерная часть (`SubscriptionScheduler`, тот же ежедневный крон):
- active→expired: `expireOverdueAccess` возвращает затронутые `(clubId, userId)` →
  `MembershipAccessClosedEvent` на каждую (mute должников по просрочке).
- «истёкшая отменённая подписка»: cancelled-строки с `subscription_expires_at` в окне
  `[now − 3 дня, now]` → `MembershipAccessRevokedEvent` (бан ушедшего по окончании
  оплаченного периода). Окно в 3 дня — catch-up на пропуски тика (ре-деплой, простой);
  повторная публикация в пределах окна безвредна (ban идемпотентен). Строки старше окна
  не трогаются — осознанное ограничение (организатор разбирается вручную).

### Тумблер «Строгий режим» (PATCH strictModeEnabled)

- **Включение** требует бота-администратора с правом «Блокировка пользователей»
  (`can_restrict_members`, 409 с объяснением иначе). Сразу **backfill**: mute всех текущих
  должников клуба (frozen/expired) — организатор видит эффект немедленно. Бан-backfill НЕ
  делается (кого организатор не выгнал до включения — форвард-онли).
- **Выключение**: unmute всех текущих должников (только тех, кто сейчас `restricted` в чате).
  Баны НЕ снимаются — путь назад через вступление в клуб (unban-флоу двери).
- **Освобождение чата** (владельческая отвязка / удаление клуба / перехват сироты — единый
  `releaseLink`, см. § «Освобождение чата»): unmute должников + снятие ВСЕХ банов строгого
  режима по учёту `chat_strict_bans` (см. § Данные) до выхода бота из чата — освобождение
  терминально, после него снять бан из приложения уже нечем.
- Потеря права ПОСЛЕ включения: тумблер не сбрасывается, действия перестают срабатывать
  (warn в лог), UI показывает алерт деградации (как у двери/закрепа).

### UI (таб «Чат»)

- Тумблер «Строгий режим» (метка «скоро» снимается): активен при боте в чате +
  `canRestrictMembers`; описание «Должники — только чтение; покинувшие клуб — бан в чате».
- Четвёртый пилл прав: `✓/✕ блокировки разрешены`.
- Алерт деградации: `strictModeEnabled && !canRestrictMembers` → «Бот потерял право
  блокировать участников…».

### Известные ограничения (осознанные)

- `restrictChatMember` работает только в **супергруппах** — в базовой группе mute молча не
  сработает (warn в логе). Реальные клубные чаты с заявками — супергруппы; конверсия
  group→supergroup ловится существующим migrate-хендлером.
- Mute не-участника чата невозможен (USER_NOT_PARTICIPANT глотается): должник, вошедший в
  чат ПОЗЖЕ включения строгого режима, попадает под mute только при следующем событии
  (шедулерный тик / freeze / повторное включение тумблера). Дверь его и так не впустит
  (frozen/expired заявки висят) — окно узкое: только «сидел в чате до привязки».
- Unmute выставляет полные permissions — если организатор сам ограничивал этого участника
  руками, оплата взноса снимет и ручное ограничение (принято: источник правды — приложение).
- Каскад «удаление клуба» (`cancelActiveByClub`) банов не публикует — чат отвязывается
  вместе с клубом, физика чата умирает вместе с привязкой. **Подтверждено кодом
  (`bugfix/chat-link-deadlock`, 2026-07-20):** `deleteClub` зовёт `releaseOnClubDeleted`, которое
  снимает мьюты и баны по учёту `chat_strict_bans` и выводит бота — отдельные ban-события
  каскаду действительно не нужны. До этого фикса строка привязки переживала soft-delete,
  и эта формулировка была планом, а не реальностью.
- Авто-отклонение заявки по таймауту 48ч (`ApplicationScheduler.markAutoRejected`) событий
  не публикует — ни decline chat-заявки (так же было в слайсе 2), ни бана: таймаут — не
  решение организатора, банить за него было бы карательным перегибом (северная звезда).
- Как вся чат-механика — best-effort: сбой Telegram логируется и не валит бизнес-операцию.

## Маршрутизатор клубных рассылок (PO 2026-07-08)

> Проблема: после привязки чата участник получал ДВА пинга об одном событии — пост в группе
> и личный DM. Решение — один механизм для всех клубных broadcast'ов.

**Инвариант:** каждый получатель уведомляется ровно один раз — в чате, если пост туда
фактически вышел и человек в чате; иначе — в личке.

Механизм опирается на **факт доставки поста**, а не на конфигурацию: клуб без чата,
выключенный тумблер, кикнутый бот, сбой Telegram — всё схлопывается в «поста нет» → DM всем,
ровно как до чат-интеграции. Членство в чате проверяется живьём на каждой рассылке
(`getChatMember`); «Telegram не ответил» = «не в чате» → DM (лишний DM лучше потерянного).

Реализация: `ChatAwareBroadcast.dmTargets(chatPostChatId?, telegramIds)`. Чат-пост и DM
идут **последовательно в одном @Async-потоке** (оркестраторы `EventBotNotifier`,
`EventCancelledListener`, `EventRescheduledListener`, `SkladchinaBotNotifier`) — два
независимых листенера гонялись бы и давали двойной пинг.

| Рассылка | Пост в чат (гейт) | DM |
|---|---|---|
| Событие создано | статус живого закрепа (`live_pin_enabled`) | только не-в-чате |
| Событие отменено | громкий пост об отмене (`live_pin_enabled`) + тихая правка закрепа | только не-в-чате |
| Дата перенесена (Этап 1) | громкий пост «было → стало» (`live_pin_enabled`) + dirty-флаг закрепа | только не-в-чате |
| Складчина создана | живой статус сбора (`skladchina_status_enabled`) | только не-в-чате |
| Складчина: напоминание T-24ч | пост с @-меншенами (`skladchina_status_enabled`) | только не-в-чате (и упомянутым сверх капа 15) |

**Мимо маршрутизатора — всегда личка** (персональный адресат / приватные данные / личный CTA):
Этап 2 «подтверди», «место ваше» из очереди, явка («вас отметили отсутствующим», споры,
«отметьте явку» оргу), заявки, отказы по складчине, «−40», итог сбора оргу,
взнос/заморозка/кик, продление подписки, door-DM. (Напоминание «подтверди участие» T-2ч
удалено PO 2026-07-08 — V51.)

Осознанная цена: участник, **замьютивший чат клуба**, больше не получает «пробивающий» DM
о новых событиях/сборах — мьют считаем осознанным выбором (принято PO 2026-07-08).

## Безопасность
- Верификация владельца при привязке — обязательный гейт (см. флоу). Одноразовые коды отвергнуты PO.
- **Перехват чата с осиротевшей привязкой** требует, сверх владения клубом, статуса
  `creator`/`administrator` **в самом чате** (`isChatAdmin`, fail-closed: молчание Telegram =
  отказ). Иначе рядовой участник легаси-группы увёл бы её под свой клуб вместе со всеми правами
  бота, а настоящие админы чата отменить это не смогли бы. На обычную привязку гейт не
  распространяется сознательно (Telegram сам гейтит добавляющего бота; см. § «Перехват…»).
- **Отказ не разрушает чужую живую привязку:** бот выходит из чата только если за чатом нет
  живого клуба. Иначе посторонний одним `/start` с чужим payload'ом выгонял бы бота и ломал
  работающую интеграцию (DoS на клуб-хозяин чата). См. § «Отказ в привязке: уходить или остаться».
- **Анти-спам на отказах:** сообщение об отказе не уходит в чат, если отправитель не
  верифицированный владелец запрашиваемого клуба И чат держит живую привязку — иначе любой
  участник группы гонял бы `/start` с произвольным UUID и превращал бота в спамера.
- **Имя клуба не утекает в отказе не-владельцу:** текст «Привязать чат может только владелец
  клуба…» названия не содержит — иначе обладатель UUID приватного клуба подтверждал бы его
  существование и узнавал имя (enumeration).
- Все REST-эндпоинты — ownership-check (`club.ownerId == caller`), 403 иначе.
- Callback `chatlink:unlink` — проверка, что жмёт владелец (по telegram_id → user → owner).
- `door_invite_link` не светится гостям; никаких секретов в логах (chat_id/club_id — ок).
- Payload deep link — только UUID (парсинг с отказом на мусор), никакого исполнения ввода.
- Fail-open запрещён: при недоступности Telegram API привязка/дверь падают с понятной ошибкой,
  членство в клубе НЕ блокируется (DM/approve — best-effort, ошибки логируются).
- Строгий режим (слайс 5): рычаги (mute/ban) применяются ТОЛЬКО к чату, который организатор
  сам привязал и где сам включил тумблер; все переходы статусов гейтятся существующей
  серверной авторизацией (ownership-check на PATCH, организаторские операции — @RequiresOrganizer).
  Fail-open исключён: сбой Telegram не открывает и не закрывает доступ в приложении, мьют/бан
  просто не срабатывает (warn). Организатор неприкасаем (см. § Принцип).
- Живой статус сбора (слайс 3.5): HTML parse_mode ⇒ заголовок складчины и имена участников
  (пользовательский ввод) **экранируются** (`&`, `<`, `>`) перед вставкой в разметку —
  инъекция тегов/ссылок через название сбора невозможна. В `tg://user?id=` попадает только
  числовой telegram_id. Пост видят только участники группы, которую организатор сам привязал;
  сумм и платёжных ссылок в посте нет.

## Критерии приёмки (staging, ручной тест PO)
Привязка:
1. Таб «Чат» виден владельцу в «Управлении» (4-й таб между «Финансы» и «Настройки» — порядок: Статистика·Финансы·Чат·Настройки).
2. «Привязать чат» открывает пикер Telegram; после добавления бота в группу: сообщение в чате
   «Чат привязан…», DM владельцу с кнопкой «Отвязать», таб показывает состояние B с пиллами прав.
3. Не-владелец добавляет бота по чужой ссылке **в свободный чат** → бот пишет отказ
   («Привязать чат может только владелец клуба…», БЕЗ названия клуба) и выходит из группы.
3a. Не-владелец шлёт `/start <чужой_club_id>` **в чате с живой привязкой** (чат уже обслуживает
   другой клуб) → бот **остаётся** в чате и **ничего не пишет** (анти-спам); интеграция
   клуба-хозяина (дверь, закрепы, теги, баны) продолжает работать. В логах — WARN
   `Chat link refused … verifiedOwner=false botLeftChat=false`.
3b. Владелец клуба Б шлёт `/start <club_id Б>` в чате, привязанном к живому клубу А → бот
   **остаётся**, но пишет отказ «Этот чат уже привязан к другому клубу. Один чат — один клуб:
   сначала отвяжите его в том клубе, «Управление» → «Чат».»
4. Кик бота из группы → таб показывает C-алерт; возврат бота + «Проверить ещё раз» → B.
5. «Отвязать чат» → бот выходит из группы, таб снова A. Кнопка [Отвязать] в DM делает то же.
5a. **Удаление клуба с привязанным чатом** (`Управление` → `Настройки` → «Удалить клуб») → бот
   выходит из группы сам; закрепы событий и статус-посты сборов откреплены, мьюты и баны
   строгого режима сняты (ранее забаненный входит в группу по обычной ссылке), теги наград
   сняты, door-ссылка отозвана (перестаёт пускать), строка `club_chat_links` удалена.
   В логе удаления клуба — `chatUnlinked=true`.
5b. **Тот же чат можно привязать заново** к новому клубу сразу после удаления старого:
   «Привязать чат» → бот добавляется → «✅ Чат привязан…». Отказа «Этот чат уже привязан
   к другому клубу» быть НЕ должно (это и был баг-дедлок).
5c. **Перехват осиротевшего чата (легаси-строки до фикса):** владелец нового клуба, являющийся
   администратором чата в Telegram, шлёт `/start <new_club_id>` → привязка проходит, бот
   остаётся в чате, в логах WARN `Releasing orphan chat link of a deleted club`. Тот же
   сценарий от владельца клуба, который **не** админ этого чата → отказ «Этот чат раньше
   принадлежал другому клубу. Перепривязать его может только администратор чата в Telegram.»
   и бот выходит.
Дверь:
6. Тумблер требует «приглашения разрешены»; при включении появляется door-ссылка.
7. Чужой человек по door-ссылке → заявка висит, ему приходит DM бота с кнопкой на клуб.
8. Организатор одобряет заявку в приложении (бесплатный клуб) → человек автоматически в чате + DM.
9. Платный клуб: одобрение заявки НЕ впускает (frozen); «Взнос получен» → впускает.
10. Участник с доступом видит кнопку «Чат клуба»; гость видит чип; frozen — ничего.
11. Отказ по заявке → висящая chat-заявка отклоняется.
Живой закреп:
12. Тумблер «Живой закреп» активен при `✓ закреп разрешён`; включение при будущем событии →
    в чате появляется закреплённый статус с голосами и кнопкой «Проголосовать».
13. Голос в приложении → закреп обновляется в пределах ~30 сек (метка «изменено», без нового поста).
14. Переход события в Этап 2 → то же сообщение становится «подтверждение мест» (подтверждено/
    очередь/дедлайн), кнопка «Подтвердить участие».
15. Создание нового события при включённом тумблере → бот сам постит и закрепляет статус.
16. Старт события → закреп открепляется с финальным текстом «Событие началось — подтвердили…».
17. Отмена события → закреп открепляется с текстом «событие отменено».
18. Организатор отмечает явку → бот постит итог (№ встречи, «пришли X из Y», «впервые» — если есть,
    следующее событие + кнопка). Повторная отметка явки итог НЕ дублирует.
19. Выключение тумблера → живые закрепы открепляются; включение обратно → свежие пины.
20. Кнопки в группе открывают Mini App на странице события (url t.me/<bot>?startapp=event_…;
    требует включённого Mini App у бота в BotFather — см. § Слайс 3).
Живой статус сбора:
21. Тумблер «Статус сборов в чате» активен при живом боте; включение при активной складчине →
    в чате появляется пост «Скинулись N из M» с дедлайном, упоминаниями «Ждём:» и кнопкой
    «Открыть сбор» (при праве закрепа — закреплённый).
22. Создание нового сбора при включённом тумблере → бот сам постит статус; участники в чате
    получают уведомление-упоминание, DM при создании приходят как раньше.
23. Отметка «Я оплатил» (или орг-отметка) → пост обновляется в пределах ~30 сек: счётчик N растёт,
    имя уходит из «Ждём» (метка «изменено», без нового поста и без пинга).
24. Закрытие сбора (вручную/все ответили/дедлайн) → финальный текст «Сбор закрыт · скинулись N из M»
    без списка неоплативших + unpin.
25. «Важный сбор» с дедлайном <24ч от текущего момента → в чат приходит напоминание с упоминаниями
    только ещё не ответивших, БЕЗ упоминания штрафа; pending-участник, которого нет в чате, получает
    прежний DM (полный текст), и не получает его, если он в чате.
26. Выключение тумблера → живые посты открепляются; включение обратно → свежие посты (backfill).
27. Кнопка «Открыть сбор» в группе открывает Mini App на странице сбора (startapp=skladchina_…).
Привязка (дополнение слайса 5):
27a. Первичная привязка чата → следом за «Чат привязан…» бот постит приглашение с кнопкой
     «Вступить в клуб» (открывает Mini App на странице клуба); повторный `/start` в том же
     чате приглашение НЕ дублирует.
Строгий режим:
28. Тумблер «Строгий режим» активен при `✓ блокировки разрешены` (пилл); без права — disabled,
    после выдачи права и «Проверить права ещё раз» — активен.
29. Включение при сидящем в чате должнике (frozen/expired) → участник переходит в «только
    чтение» (backfill); выключение → голос возвращается.
30. Активный участник просрочил продление (шедулерный тик active→expired) → mute; «Взнос
    получен» → голос возвращается + существующий DM о продлении.
31. Организатор жмёт «Закрыть доступ» (freeze) → mute; «Открыть доступ» (unfreeze) → unmute.
32. Кик участника из клуба → бан в чате (человек вылетает). Повторное вступление в клуб
    (заявка → одобрение → взнос) → бан снят, приглашение в чат приходит (флоу двери).
33. «Отказать и вернуть» должнику → бан. Отклонение заявки на вступление у человека, сидящего
    в чате → бан.
34. Добровольный выход из бесплатного клуба → бан сразу; выход из платного с оплаченным
    периодом → бан на следующем тике шедулера ПОСЛЕ истечения периода (в течение периода
    человек остаётся в чате).
35. При выключенном тумблере ничего из 29–34 не происходит (только существующий decline заявок).
36. Освобождение чата (отвязка ИЛИ удаление клуба) снимает все баны строгого режима: кикнутый ранее участник после этого может
    войти в группу по обычной ссылке (фидбек PO со staging). Мьюты также снимаются (unmute
    должников до выхода бота).

## Рамки/факты Telegram (проверено, notes.md)
- `my_chat_member` и `chat_join_request` приходят при пустом `allowed_updates` (наш стартер).
- `chat_member` (чужие статусы) — НЕ приходит и не нужен в слайсах 1–2.
- Бот не может: добавить юзера в группу, закрыть чат «по заявкам» за организатора.
- `?startgroup=<payload>&admin=…` — payload ≤64 символа [A-Za-z0-9_-], UUID проходит.
