# Профиль-квест: заполни профиль → уровень 2 «Свой»

**Статус:** ✅ в проде (2026-07-23, ветка `feature/profile-quest`; в том же PR —
статистика одной панелью с герой-строкой «Надёжность» и фикс потери недобранного
интереса). Дизайн залочен PO 2026-07-22.
**Мокап-истина:** `docs/design/profile-completion/mockups/02-quest-card.html` (направление
«Карточка-квест»; варианты 01/03 отклонены).

## Цель

Новый пользователь приходит с пустым профилем и не понимает, что заполнять и зачем.
Квест-карточка в профиле показывает **что, для чего и в каком порядке** заполнить,
платит XP за каждый шаг и доводит ровно до уровня 2 «Свой» с поздравительным бейджем.
Заодно уходит лис-экран интересов (его роль забирает квест).

## Ключевое решение PO (2026-07-22): исключение из «XP только участие»

Принцип P1b «XP начисляется ТОЛЬКО за участие» (решение 2026-06-14, reputation-v2.md §H3)
получает **одно осознанное исключение**: профильный XP. Ограничения исключения:
- **одноразовый** — веха достигается один раз и не отзывается (стёр bio → XP остаётся,
  инвариант «XP не убывает» AC-P1b-6 сохранён);
- **кап 50 XP** — ровно порог уровня 2, дальше профиль XP не приносит;
- отдельный канал в коде (`XpPolicy.ProfileQuest`), не смешан со статами участия.

## Шаги квеста (порядок и XP — залочены мокапом)

| # | Шаг | Поле | XP | «Зачем» (текст в карточке) |
|---|-----|------|----|----------------------------|
| 1 | Город | `users.city` (непустой) | +10 | «Найдём клубы рядом с тобой» |
| 2 | Интересы | ≥1 строка в `user_interests` | +25 | «По ним подберём клубы, которые твои» |
| 3 | О себе | `users.bio` (непустой) | +15 | «Организаторы поймут, кто к ним идёт» |

Сумма 50 = `levelThreshold(1)` — заполнивший профиль гарантированно достигает «Свой».

## Backend

### Миграция V66 — вехи в `users`
Три nullable timestamptz-колонки: `quest_city_at`, `quest_interests_at`, `quest_bio_at`
(NULL = веха не достигнута; проставленная метка никогда не сбрасывается). Русские
`COMMENT ON`. **Backfill**: существующим пользователям вехи проставляются по текущей
заполненности (они честно заслужили +до 50 XP; карточка у них не появится — квест
завершён). Частично заполнившие получат карточку с частью галочек.

### Запись вех
`UserService.updateProfile` после апдейта полей и интересов вызывает
`UserRepository.markQuestMilestones(userId)` — один UPDATE: для каждой NULL-вехи, чьё
поле сейчас непусто, ставится `now()` (COALESCE + CASE, гонок нет). В той же транзакции.

### Чтение и расчёт XP
- `XpPolicy.ProfileQuest(city, interests, bio: Boolean)` + константы XP шагов +
  `profileXp(quest)` (сумма достигнутых, максимум 50) — отдельно от `XpStats` (участие).
- `XpPolicy.totalXp(stats, quest)` = участие + профиль. Уровень/пороги без изменений.
- Бейдж **«Визитка»** (`id=profile_card`, новая семья `PROFILE`): все три вехи достигнуты.
- `XpService.getGamification`: тянет вехи через `UserRepository.findQuestFlags`.
- **Консистентность others-проекции:** `ApplicantSignalService` (пилюля уровня заявителя)
  тоже учитывает профильный XP — батч `findQuestFlagsByIds`, иначе self-уровень 2
  расходился бы с уровнем, который видят организаторы.

### API
`GamificationDto` расширен: `quest: ProfileQuestDto { cityDone, interestsDone, bioDone,
completed }`. Отдельного эндпоинта нет — карточка питается существующим
`GET /api/users/me/gamification`.

## Frontend

- **`ProfileQuestCard`** (новый компонент, по мокапу): развёрнутая карточка
  (донат-«бублик» % прогресса, «N / 50 XP до уровня 2 „Свой“», чеклист 3 шагов с
  нумерацией/галочками/«зачем»/+XP, следующий шаг подсвечен с кнопкой «Заполнить» →
  открывает существующий ProfileEditModal) ↔ свёрнутая пилюля (мини-донат, счёт XP,
  «Дальше: …»). Fold-состояние — localStorage.
- Позиция: между identity-блоком и остальным контентом профиля.
- **Видимость:** карточка показана, пока `quest.completed == false`. Уже-завершённый
  квест при загрузке → карточки нет вовсе (старички ничего не видят).
- **Поздравление:** когда `completed` становится true В ТЕКУЩЕЙ СЕССИИ (переход
  ловится на клиенте после мутации профиля), карточка превращается в поздравление:
  бейдж-звезда, «Уровень 2 — „Свой“!», чипы «Бейдж „Визитка“» и «+50 XP», CSS-конфетти,
  кнопка «Отлично» убирает её (state сессии; после перезагрузки не возвращается).
  Бейдж «Визитка» остаётся чипом в панели «Уровень» (рендерится существующим кодом).
- **Лис-экран интересов удалён** из ProfilePage (пустые интересы → секция просто
  скрыта; плашка ошибки загрузки интересов остаётся). Прод-нудж bio под identity
  остаётся (мокап подтверждает).

## Acceptance criteria

1. Новый юзер (пустой профиль): карточка-квест в профиле, 0/50 XP, шаг 1 «Город»
   подсвечен; лис-экрана интересов нет.
2. Заполнил город → в карточке галочка, +10 XP в панели «Уровень», подсвечен шаг 2.
3. Заполнил все три поля → XP = 50 (уровень 2 «Свой»), карточка → поздравление с
   бейджем «Визитка» и конфетти; «Отлично» убирает её навсегда.
4. Стёр bio после получения вехи → XP не уменьшился, уровень остался; карточка
   не возвращается.
5. Существующий юзер с заполненным профилем: после деплоя +50 XP (backfill), бейдж
   «Визитка» в панели, карточки и поздравления НЕ видит.
6. Организатор в карточке заявителя видит уровень С УЧЁТОМ профильного XP
   (others-проекция консистентна с self).
7. Фарм невозможен: повторное редактирование/очистка/перезаполнение полей не даёт
   нового XP (вехи одноразовые).
8. `./gradlew test` и `npm test` зелёные; чистый tsc.

## Не входит

- Прогресс-кольцо вокруг аватара (направление 01) и визард-шторка (03) — отклонены.
- XP за фото/имя (приходят из Telegram, заполнять нечего).
- Серверное хранение «поздравление показано» — клиентская сессия (осознанный MVP).
