# Карточка участника — админ-редактирование (Variant B)

> **Статус:** спека на согласование (design lock). Макет: `docs/design/member-admin-profile/mockups/variants.html` (вариант B).
> **Ветка:** продолжаем на `feature/de-stars-decoupling` — фича расширяет de-Stars-карточку (org-гейт),
> отдельная ветка потребовала бы мёржа de-Stars первым (он ещё в staging-тесте). PR вырастет, но связно («управление участником»).

## 0. Что строим

Карточка участника (`MemberProfileModal`) для **организатора** получает режим правки (✎ в шапке → форма).
Обычный участник видит карточку без него. 4 блока (PO выбрал все):

1. **Подписка** — продлить (`Взнос получен`), заморозить, **своя дата** окончания.
2. **Роль** — Участник / Модератор / Со-организатор.
3. **Награды клуба** — club-local награды (не глобальные earned-бейджи).
4. **Заметка организатора** — приватное поле.

Кто может править: **владелец клуба** (`clubs.owner_id`) и **Со-организатор** (после ввода ролей). Модератор — нет.
Нельзя править карточку владельца и (для Со-орга) других Со-оргов — см. §4.

## 2. Данные / миграции

### V38 — заметка
```sql
ALTER TABLE memberships ADD COLUMN IF NOT EXISTS organizer_note TEXT NULL;
COMMENT ON COLUMN memberships.organizer_note IS
  'Приватная заметка организатора об участнике (NULL = нет). Видна только организаторам клуба, не самому участнику.';
```
«Своя дата» подписки переиспользует существующую `subscription_expires_at` (колонка уже есть) — только новый сервис-метод.

### V39 — награды клуба
```sql
CREATE TABLE IF NOT EXISTS club_awards (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    club_id     UUID NOT NULL REFERENCES clubs(id) ON DELETE CASCADE,
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    emoji       TEXT NOT NULL,            -- одиночный эмодзи-значок награды
    label       TEXT NOT NULL,            -- короткое название, до 40 симв.
    awarded_by  UUID NOT NULL REFERENCES users(id),
    awarded_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (club_id, user_id, label)      -- одна и та же награда не дублируется
);
COMMENT ON TABLE club_awards IS 'Награды уровня клуба, которые организатор вручает участнику (club-local, НЕ глобальные earned-бейджи; косметика, на репутацию/XP не влияют).';
-- + COMMENT ON каждую колонку (на русском).
```

### V40 — роли
```sql
-- ADD VALUE изолированно (txn-safe оговорки как в V37): без UPDATE в той же миграции.
ALTER TYPE membership_role ADD VALUE IF NOT EXISTS 'moderator';
ALTER TYPE membership_role ADD VALUE IF NOT EXISTS 'co_organizer';
```
Текущие значения: `organizer` / `member`. Владелец остаётся `organizer` (единственный). Со-орг/Модер — новые роли участников.
⚠️ После каждой — `./gradlew generateJooq` (codegen на временном PG, не на dev-БД — см. impl-map de-Stars §Codegen).

## 3. API контракты (организатор-only, owner/co-org гейт)

| Метод | Действие | Слайс |
|---|---|---|
| `PATCH /api/clubs/{clubId}/members/{userId}/note` `{ note: string \| null }` | заметка (сохранить/очистить) | S1 |
| `POST /api/clubs/{clubId}/members/{userId}/access-until` `{ until: ISO-date }` | своя дата окончания (status=active) | S1 |
| `POST /api/clubs/{clubId}/members/{userId}/awards` `{ emoji, label }` → AwardDto | выдать награду | S2 |
| `DELETE /api/clubs/{clubId}/members/{userId}/awards/{awardId}` | снять награду | S2 |
| `PUT /api/clubs/{clubId}/members/{userId}/role` `{ role: "member"\|"moderator"\|"co_organizer" }` | сменить роль | S3 |

`GET /api/clubs/{clubId}/members/{userId}` (`MemberProfileDto`) дополняется:
- `awards: AwardDto[]` (видны всем участникам — это публичное признание).
- `organizerNote: string \| null` — **только организаторам** (как `subscriptionExpiresAt`).
- `role` — уже есть; начинает принимать новые значения.

Существующие de-Stars экшены (`dues-paid`/`freeze`) — без изменений, переиспользуются в форме.

## 4. Роли и права (PROPOSED — главный вопрос на согласование)

Иерархия: **Владелец (organizer)** > **Со-организатор** > **Модератор** > **Участник**.

| Возможность | Участник | Модератор | Со-орг | Владелец |
|---|---|---|---|---|
| Видеть контент клуба (feed/события/складчина) | ✅ | ✅ | ✅ | ✅ |
| Заявки: одобрить/отклонить | — | ✅ | ✅ | ✅ |
| События: создать/отметить посещаемость | — | ✅ | ✅ | ✅ |
| Складчина: создать/отметить оплату | — | ✅ | ✅ | ✅ |
| Участники: взнос получен / заморозить / награды / заметка | — | — | ✅ | ✅ |
| Менять роли участников | — | — | ✅¹ | ✅ |
| Настройки клуба (название/цена/правила) | — | — | ✅ | ✅ |
| Финансы клуба | — | — | ✅ | ✅ |
| Удалить клуб / сменить владельца / биллинг-подписка | — | — | — | ✅ |

¹ Со-орг может назначать только Участник↔Модератор; повышать до Со-орга и трогать других Со-оргов — только Владелец.

**Реализация прав:** сейчас гейт `@RequiresOrganizer(clubIdParam)` бинарный (owner-only через `AuthorizationAspect`).
Под роли нужно расширить до `@RequiresClubRole(min = MODERATOR|CO_ORGANIZER)` и проставить на ~десяток эндпоинтов
(applications/events/skladchina/members/settings). Это и есть основной объём S3.
**Если роли пока не нужны полноценно** — можно сделать роль как «ярлык» (меняется и показывается на карточке/в ростере),
но БЕЗ расширения прав — тогда S3 дешёвый, но Модератор/Со-орг ничего не «могут». Решить: **права или только ярлык**.

## 5. Награды клуба (LOCKED — R2)

- **Что это:** косметическое признание внутри клуба. На репутацию/XP/ранг **не влияют** (анти-фарм чист, R4).
- **Создание (стиль «как интересы»):** орг жмёт «＋ Добавить награду» → форма ввода с **автоподсказками уже существующих наград** (название + эмодзи). Если награда есть в БД (по этому клубу / глобально-частые) — выбирает из подсказок. Если нет — вписывает название + эмодзи и жмёт зелёную стрелку (создать). Создаётся новая запись-награда, доступная как подсказка дальше.
  - **MVP:** без нейронки — ручной ввод + автоподсказки из существующих. **Опционально (если дёшево):** нейронка генерит варианты (название+эмодзи) по описанию клуба как стартовые подсказки.
  - Каталог подсказок = ранее созданные награды (источник: distinct по `club_awards`, можно расширить на топ-частые по всем клубам).
- **Показ:** чипы на карточке участника, видны **всем** участникам клуба (R3 — публичная похвала). На странице клуба/в ростере — нет (только в карточке).
- **Лимит:** до 6 наград на участника в клубе.
- **Кто выдаёт:** Со-орг + Владелец. Снять может тот же уровень.

## 6. Заметка организатора

- Одно текстовое поле (≤500), `memberships.organizer_note`. Видят только организаторы (owner/co-org/мод? — **предложение: owner + co-org**, модератор не видит чужие заметки).
- В форме правки — `textarea`. На read-only карточке у участника — не показывается вообще.

## 7. Критерии приёмки (по слайсам)

**S1:** организатор открывает карточку → ✎ → видит поле заметки + «своя дата»; сохраняет; заметка не видна обычному участнику; «своя дата» меняет `subscription_expires_at`, доступ открыт до неё; не-организатор не может дёрнуть эндпоинты (403).
**S2:** организатор выдаёт награду (эмодзи+название) → чип появляется на карточке у всех; снимает → исчезает; лимит 6; дубль по названию отклоняется.
**S3:** смена роли работает по матрице §4; Модератор реально может одобрять заявки/события, но не управлять участниками; Со-орг может всё кроме владельческого; владельца нельзя разжаловать; codegen/enum не сломали существующее (frozen/access-гейт целы).
