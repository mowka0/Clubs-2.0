# Связка клуб ↔ телеграм-чат — рабочие заметки к спеке

> Зафиксированные решения PO и проверенные технические факты (2026-07-05).
> Мокапы — в `mockups/`. Спека написана: `docs/modules/club-chat-link.md` (слайсы 1–2 реализованы в ветке feature/club-chat-link).

## Решения PO

- **Привязка — вариант А, сразу просто**: payload deep link'а = `club_id`, без
  одноразовых кодов, без хранения состояния (ни Redis, ни Postgres-таблицы).
  Гейт безопасности — обязательная проверка «добавивший бота = владелец клуба»
  (telegram_id из `my_chat_member.from` сверяется с owner'ом клуба).
- **Петля подтверждения — в v1**: после привязки бот пишет в чат «Чат привязан
  к клубу X» и DM владельцу «Чат „Y“ привязан к клубу X. Это были вы? [Отвязать]» —
  фишинг-привязка мгновенно видна и обратима.
- Одноразовые коды как hardening — отвергнуто (не дыра при проверке владельца;
  от фишинга владельца код всё равно не спасает).

## Решение PO: размещение в UI + перестройка «Управления» (2026-07-05)

Сейчас участники дублируются: таб «Участники» на странице клуба (плоский список)
и таб «Участники» в «Управлении» (`managementView` — с attention-бакетами
«Скоро закончится» / «Оплата вступления»). Админ-действия (карточка участника,
кик, подтверждение оплаты) доступны одинаково в обоих местах.

Решение:
- **Таб «Участники» из «Управления» убираем.** Участники живут только на странице
  клуба. Организаторские attention-бакеты переезжают туда же: для владельца таб
  рендерится в `managementView`-режиме (бакеты видит только он; прочим — плоский
  список, как сейчас). Комментарий в `ClubMembersTab.tsx` про «бакеты живут ТОЛЬКО
  в Управлении» при этом инвертируется — поправить.
- **«Чат» становится полноценным табом «Управления».**
  Итоговые табы: `Статистика · Финансы · Чат · Настройки`.
- Дефолтный таб «Управления» после удаления members — `stats`
  (проверить `resolveInitialTab`).
- Навигация: deep-link'ов на `?tab=members` в коде нет (проверено grep'ом);
  входы `/clubs/:id/manage` без параметров — не ломаются.
- Контент таба «Чат»: состояние привязки (мокапы A/B/C из `mockups/01`),
  тумблеры фич (закреп / дверь / строгий режим-«скоро»), health-алерты, отвязка.

## Правка PO (2026-07-05): убрать ложный «Новичок» при скрытом Trust

Контекст: асимметричная видимость Trust (#94) — бэкенд отдаёт `trust=null` всем,
кроме организатора и самого участника о себе, причём null «неотличим от новичка»
**по дизайну** (`MembershipMapper.toMemberListItemDto`, комментарий там же).
Фронт при null пишет «Новичок» — для зрителя-неорганизатора это враньё: у человека
может быть богатая история, просто зрителю её не положено видеть.

Решение — фронт-онли, бэкенд не трогаем (неоднозначность null сохраняем):
- **Список участников** (`ClubMembersTab`, строка ~223): фолбэк «Новичок» в колонке
  скора показывать ТОЛЬКО организатору (`isOrganizer`). Не-организатору при
  `trust=null` — ничего (пустая колонка). «Орг» у владельца — как сейчас.
- **Карточка участника** (`MemberProfileModal`, ~717–733): секцию «Репутация в этом
  клубе» при `trust=null` показывать только если зритель — организатор ИЛИ смотрит
  сам себя (честный «Новичок — пока недостаточно данных»). Чужим — секцию не
  рендерить вовсе. Исключение: для `role === 'organizer'` explainer «Здесь репутация
  начисляется за организаторские качества» остаётся виден всем (скоров не палит).
- `MyClubsPage` «Новичок» не трогаем — там зритель всегда смотрит сам себя.

## Требования PO к «двери» и интеграции (2026-07-06)

1. **Заявка через бота → страница клуба.** Подтверждено: DM бота при chat_join_request
   ведёт deep link'ом на страницу клуба, дальше существующий флоу «Хочу вступить».
2. **Обратная интеграция (вступил в клуб → в чат).** Ограничение Telegram: бот НЕ может
   принудительно добавить человека в группу (нет такого API). Реализация: при вступлении
   в клуб бот шлёт DM со ссылкой-приглашением; тап → chat_join_request → бот мгновенно
   авто-одобряет (человек уже участник клуба). UX ≈ «нажал — и в чате».
3. **Кнопка «Чат клуба» на странице клуба** — участникам с доступом (не frozen).
   Та же бот-ссылка: уже в чате → откроется чат; ещё нет → авто-впуск (п.2).
4. **Награды → титулы в чате** (= ступень «титулы» лестницы доверия). Механика:
   промоут участника в админа с НУЛЕВЫМИ правами + `setChatAdministratorCustomTitle`.
   Рамки Telegram: ≤50 админов на группу; титул ≤16 символов, БЕЗ эмодзи; боту нужно
   `can_promote_members` (бот может менять титулы только тем, кого сам повысил);
   титул один → при нескольких наградах выбираем приоритетную (решить на спеке);
   ⚠️ конфликт: «нулевой админ» неподвластен restrictChatMember → строгий режим
   должен сначала разжаловать, потом мьютить (или не титуловать должников).

## Проверенный факт: allowed_updates (telegrambots 7.10.0)

Цепочка (дизассемблировано из jar'ов нашей версии):
спринг-стартер `TelegramBotInitializer` → `registerBot(token, consumer)` →
`new DefaultGetUpdatesGenerator()` → **пустой список `allowed_updates`
отправляется в каждом getUpdates** → по Bot API пустой список = «все типы,
кроме `chat_member`, `message_reaction`, `message_reaction_count`».

- ✅ `my_chat_member` (статус самого бота: добавлен/кикнут/права) и
  `chat_join_request` приходят из коробки — привязка и «дверь» работают без
  дополнительной конфигурации.
- Пустой список шлётся явно при каждом поллинге ⇒ липкое «previous setting
  will be used» нас не касается.

## На подумать: `chat_member` (статусы других участников)

По умолчанию НЕ приходит (см. выше). Прежде чем тянуться за ним (ручная
регистрация бота через 4-арг `registerBot` с кастомным генератором вместо
авторегистрации стартера) — проверить, что он вообще нужен:

- **Строгий режим, скорее всего, обходится без него**: все действия бота
  (restrict должника, ban исключённого) триггерятся событиями ПРИЛОЖЕНИЯ
  (frozen/кик в клубе), а не событиями чата.
- `chat_member` понадобился бы только для обратного зеркалирования («человек
  сам вышел из чата → что-то сделать в клубе»). Можно осознанно НЕ делать:
  выход из чата ≠ выход из клуба, организатор видит сам.
- Альтернатива без подписки: точечный `getChatMember` в момент, когда статус
  реально важен (перед restrict/ban, при одобрении заявки).
