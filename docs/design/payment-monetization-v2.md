# Оплата и монетизация — версия 2 (стратегия и решения)

**Статус:** черновик-решение (design/decision record), не реализация.
**Дата фиксации:** 2026-06-20.
**Заменяет:** PRD §4.7 (оплата только через Telegram Stars) и `docs/modules/payment.md` v1 (Stars-инвойсы + 20/80 сплит) — как **целевую** монетизацию. v1 остаётся описанием текущего кода до миграции.
**Регион:** РФ. Юрлицо платформы — пока не определено (на старте достаточно ИП).

> Документ фиксирует выводы длинного разбора монетизации. Часть выводов — **РЕШЕНО** (конвергенция),
> часть — **ОТКРЫТО** (требует продуктового/юридического решения). Метки проставлены явно.

## 1. TL;DR

- **Telegram Stars как денежный поток — отвергнуты** для РФ: Stars фактически нельзя купить в РФ без турецкого App Store + гифт-карт, вывод только в TON через Fragment (крипта — не средство платежа в РФ), эффективная потеря ~30%+, нет произвольного маркетплейс-сплита. Оставить максимум как опциональный канал для чисто цифровых внутри-Telegram фич, не для членских взносов.
- **Платформа выходит ИЗ потока денег клуба.** Взносы участник→организатор идут **мимо платформы** (СБП/нал, на стороне организатора). Это снимает разом: статус платёжного агента (103/161-ФЗ), удержание чужих денег, принудительную самозанятость организаторов.
- **Платформа монетизирует СВОЙ продукт** (приложение) — собственным сервисным сбором, рекуррентно. Это чистая продажа цифровой услуги: УСН 6%, свои чеки, никакого сплита.
- **Тарифная модель — РЕШЕНО (каркас):** выбираемые планы-ёмкости («1 клуб / до 3 / безлимит»), цена строится как **скидка за объём** (выше тир = дешевле за клуб), честный «сервисный сбор» строкой, а не скрытая наценка.
- **Главный детерминант успеха — НЕ платёжная схема, а активность клубов** (частота встреч). Стратегия: **сначала доказать цикл на бесплатной базе, монетизацию включать после** — и, вероятно, привязанную к активности и/или с организатора, а не плоским сбором с участника. **(РЕШЕНО 2026-06-25: оба плательщика обязательны — v1 organizer-pays, фаза 2 member-pays; три тарифа + monotone volume-discount; месячная подписка. Цены и timing — отложены. Залоченная спека: `docs/modules/payment-v2.md`.)**

## 2. Что отвергли и почему

| Вариант | Вердикт | Причина |
|---|---|---|
| **Telegram Stars (как основной поток)** | ❌ Отвергнут для РФ | Stars не купить в РФ без турецкого App Store + гифт-карт; вывод только в TON (259-ФЗ: крипта не средство платежа); ~30%+ потерь (Apple/Google + спред Fragment); нет произвольного сплита (только реферальный affiliate). |
| **Авто-сплит у эквайера (ЮKassa Сплит и т.п.) с комиссией 20%** | ⚠️ Не для массы | Юридически чисто, НО **требует, чтобы каждый организатор стал самозанятым/ИП**. Для обычного человека с клубом «20×200₽ = 4000₽» это отсекает ~95% аудитории. Оставить как **опцию** для серьёзных организаторов с оборотом. |
| **Tribute-подобная модель (платформа = иностранный агент, выплаты физлицам без регистрации)** | ❌ Не реализуема российской платформой | «Бесшовность» Tribute возможна только потому, что он **иностранец** и перекладывает налог на получателя. Российское юрлицо, платя физлицу, становится **налоговым агентом → 13% НДФЛ + ~30% страховых взносов** (экономическая смерть), либо уходит в серую зону. Годится максимум как временный костыль для MVP-валидации спроса (воткнуть сам Tribute), не как модель. |
| **Сбор «зашить 30₽ в цену 230₽», не называя сбором** | ❌ Отвергнут | В мире 2 платежей рекламируешь одно число, а просишь два платежа → запутывает; и юридически выглядит как скрытая комиссия с продажи организатора (риск переквалификации). **Честный сервисный сбор строкой — лучше** (честность + чистота). |
| **Платёж в 2 шага (взнос P2P + отдельный сбор) для ЕЖЕМЕСЯЧНОГО режима** | ⚠️ Плохой UX для подписки | Авто-списание P2P на личную карту физлица **не существует**; взнос пришлось бы платить вручную каждый месяц → churn, забывчивость, слабое подтверждение. Приемлемо для разового входа, плохо для подписки. |

## 3. Ключевые юридические/налоговые факты РФ (грунт под решениями)

1. **103-ФЗ к платформе НЕ применяется** — он про приём агентами **наличных**, прямо исключает безналичные расчёты и кредитные организации. Безналичные оплаты идут через НКО. Реальная рамка при работе с деньгами — **161-ФЗ** + **агентский договор (гл. 52 ГК)** + с 01.10.2026 **289-ФЗ «О платформенной экономике»**.
2. **Самозанятый НЕ может быть агентом/платформой** (ст. 4 ч.2 п.5 422-ФЗ — запрет деятельности в интересах другого лица по агентским/комиссии/поручения). Платформе нужен **ИП или ООО на УСН 6%**.
3. **API ФНС для самозанятых** существует (60+ партнёров, ~40% самозанятых зарегистрированы через сторонние приложения): партнёр может **зарегистрировать** человека самозанятым и **авто-выпускать чеки** за него. Это снимает ~90% трения регистрации (если выбран путь со сплитом). Готовые партнёры-агрегаторы: Qugo (от 1,5%, платит НПД за исполнителя), Solar Staff, Консоль.Про; NPD-API у ЮKassa/Robokassa.
4. **НДС 22% на эквайринг с 01.01.2026** — считается **с комиссии банка**, не с суммы платежа (на 1000₽ по карте ~7,7₽; по СБП — нет). Твой собственный НДС возникает только если доход платформы > 20 млн₽/год (2026; далее 15 в 2027, 10 в 2028) и считается с **твоей выручки за услугу** (комиссии/сбора), не с оборота.
5. **Рекуррент (авто-списание) требует мерчант-привязки** — сохранённый токен карты или СБП-подписка. И то, и другое возможно только у **зарегистрированного мерчанта**. Авто-списания «по подписке» на личную карту обычного физлица **не существует**.
6. **СБП P2P не даёт авто-подтверждения** оплаты (нет callback). Надёжное авто-подтверждение взноса = только C2B/зарегистрированный получатель (вебхук). Чистый P2P подтверждается только вручную (организатор отмечает).

## 5. Выбранное направление — РЕШЕНО

### 5.1 Платформа ВНЕ потока денег клуба
- Взносы за клуб идут **напрямую участник→организатор** (СБП P2P / нал / своя онлайн-ссылка организатора). Платформа их **не касается**.
- Платформа **не платёжный агент**, не держит чужих денег, не тащит организаторов в самозанятость.
- **Контроль доступа к клубу — у организатора** (заморозка/разморозка, приём взноса как удобно). Бесплатные клубы становятся тривиальными (нет платёжного события).
- Аналог уже есть в продукте — **модуль складчины** (`docs/modules/skladchina.md`): «деньги идут мимо платформы по ссылке организатора, платформа ведёт только учёт». P2P-сбор взносов ложится на ту же механику.

### 5.2 Платформа монетизирует СВОЙ продукт
- Платформа продаёт **свою цифровую услугу** (приложение/доступ к платным клубам/фичи) — это **её собственный доход**.
- Юридически просто: ИП/ООО на УСН 6%, своя облачная касса (54-ФЗ) **только на свой сбор**, обычный эквайринг на свой счёт (сплит не нужен).
- Рекуррентное авто-списание **своего** сбора — без проблем (платформа = мерчант, токен карты, «подключил один раз»). Это **минус один платёж** из системы — платформа обрабатывает ровно одну транзакцию (свою).

## 6. Тарифная модель — РЕШЕНО (каркас), цифры ОТКРЫТЫ

### 6.1 Честный «сервисный сбор», не скрытая наценка
Показывать строкой: «взнос клубу X · сервисный сбор Clubs Y». Честность принята как продуктовая ценность и как юридическая защита (явно **своя** услуга, а не кусок чужой цены).

### 6.2 Выбираемые планы-ёмкости (а не авто-пересчёт)
Юзер сам выбирает план = потолок числа платных клубов. **НЕ** делать авто-пересчёт цены «сколько клубов прямо сейчас» (ад прорации + непонятные счета).

**Залоченная launch-сетка (PO 2026-06-25, детали → `docs/modules/payment-v2.md` §3):**

| План | Платных клубов | Цена/мес |
|---|---|---|
| `FREE` | 1 | 0₽ |
| `TRIO` | до 3 | 200₽ |
| `UNLIMITED` | ∞ | 400₽ |

**Единица ёмкости = платный клуб** (с взносом участников); бесплатные клубы безлимитны и не считаются. **Первый платный клуб — бесплатно**, сбор со 2-го.

**Правило ценообразования (free-floor + no-cliff):** `FREE` — giveaway-пол вне «цены за клуб»; `UNLIMITED ≤ 2× TRIO` (400≤400 — безлимит открывается дёшево); цена-за-клуб на потолке платных тиров невозрастающая. Анти-паттерн (отвергнут): сетка-штраф, где переход дороже входа в платность — бьёт по discovery. Известный артефакт (не нарушение): 4-й клуб на границе TRIO→UNLIMITED временно дороже за клуб, затем дешевеет.

**Триггер пэйвола:** модалка подписки — на **последнем шаге создания 2-го+ платного клуба** (data введены, осталось «Создать»).

### 6.3 РЕШЕНО (2026-06-25) — детали в `docs/modules/payment-v2.md`
- **Кто платит:** оба плательщика обязательны. **v1 = organizer-pays** (строим первым: один плательщик на клуб, очевидный ROI, обходит непроверенную гипотезу «платить за бывшее бесплатным» на ценочувствительной стороне). **Фаза 2 = member-pays** (committed, gated на §10 + юридику; проектируется в модель данных сразу, но не строится в v1).
- **Структура планов:** три тарифа `SINGLE`/`TRIO`/`UNLIMITED` + инвариант **monotone volume-discount** (выше тир = дешевле за клуб, запрет штрафа за многоклубность).
- **Триггер:** ежемесячная рекуррентная подписка.
- **Конкретные цены и тиры — ОТКРЫТО** (tuning-knob, зависит от НДС-пересчёта эквайринга — §11.6).

## 7. Рекуррент и правила биллинга — РЕШЕНО (упрощено PO 2026-06-25)

> **PO-override 2026-06-25:** подписка = обычная месячная рекуррентная подписка. Работает весь оплаченный период **независимо от наличия активных клубов**, **не паузится** по активности, выключается **только в конце периода** (после отмены или несостоявшегося продления). Это **отменяет** freeze-by-hours (старый §7.3) и aggregate-flag pause (старый §7.5). Залоченная стейт-машина — `docs/modules/payment-v2.md` §4.

### 7.1 Золотое правило — сохранено
**Нет доступа — нет списания.** Сбор и доступ **обязаны быть связаны**. В organizer-pays «доступ» = доступ к премиум-ёмкости платформы; отмена → потеря доступа в конце периода → списания прекращаются. Правило выполняется тривиально.

### 7.2 Машина состояний подписки (плоская месячная)

| Состояние | Доступ к премиум-ёмкости | Сбор |
|---|---|---|
| `ACTIVE` | Есть | Авто-продление раз в месяц |
| `CANCELLED_PENDING_END` | Есть до конца периода | Не продлевается; в конце периода → `ENDED` |
| `PAST_DUE` (dunning) | Есть в течение grace | Ретраи продления; успех → `ACTIVE`, исчерпание → `ENDED` |
| `ENDED` | Нет | Отменён |

**Нет** `Frozen`-состояния на уровне подписки, **нет** банка часов, **нет** aggregate-flag. Переходы forward-only + `@Transactional` с грантом/ревоком доступа.

### 7.3 Кто может завершить
Организатор отменяет свою подписку сам (`ACTIVE → CANCELLED_PENDING_END`, доигрывает до конца оплаченного периода). Завершение = отмена авто-продления.

### 7.4 Заморозка организатором — ортогональна биллингу
«Заморозка доступа участнику» (организатор не получил P2P-взнос) — это **гейт доступа к клубу** (организатор↔участник, honor-system, decoupling-слайс), **не состояние подписки**. В organizer-pays не влечёт платформенного заряда с участника.

### 7.5 Клубы при `ENDED`
Ёмкость падает до базовой (free). Платные клубы сверх новой ёмкости теряют премиум-инструменты, **данные сохраняются** (restore при ре-подписке). Точный over-capacity UX — отложенный tuning-detail.

## 8. Две архитектуры платежа и когда какая

| | **Форма P2P-OUT (база)** | **Форма SPLIT (опция)** |
|---|---|---|
| Взнос | Участник→организатор P2P/нал, мимо платформы | Один платёж эквайеру, сплит 200/30 |
| Фискализация взноса | Нет (не твои деньги) | Да (эквайер пробивает) |
| Требует от организатора | Ничего | **Самозанятость/ИП** (1 тап через API ФНС) |
| Авто-подтверждение взноса | Нет (только ручное) | Да (вебхук) |
| Рекуррент взноса | Нет (вручную) | Да |
| Когда применять | Массовый сегмент, бесплатные/нал-клубы | Серьёзные организаторы, кому нужна белая авто-подписка взноса |

**Важно:** «один платёж + взнос мимо фискализации» **не существует** в чистом виде. Регулируемый сплит фискализирует всё → требует регистрации организатора. Чистый P2P → отдельный платёж, без авто-подтверждения. Выбираешь ось: zero-friction (P2P-out) или гладкая авто-подписка взноса (split + самозанятость).

Обе реализуются за одним интерфейсом `PaymentProvider` — базовая масса на P2P-out, split включается опцией.

## 9. Влияние на стек (high-level)

- **`PaymentProvider`** — абстракция над провайдерами (свой сервисный сбор: обычный эквайринг ЮKassa/Т-Банк; опц. Stars для digital-фич; опц. split-эквайер). Сумму всегда считать на сервере, не доверять клиенту.
- **Эндпоинты:** create-subscription (свой сбор), webhook (верификация подписи + идемпотентность по `provider_event_id UNIQUE`), cancel/leave, статус.
- **Состояния подписки** — машина из §7.2; смена статуса + доступ к клубу атомарно (`@Transactional`), переходы только вперёд (защита от out-of-order вебхуков).
- **Идемпотентность** входящая (UNIQUE по event id) и исходящая (`Idempotency-Key`).
- **Самое важное в реализации — не сплит, а:** вебхук-подтверждение (никогда не клиент), идемпотентность, машина состояний + восстановление частичных сбоев + дуннинг. P2P-out минимизирует это (одна транзакция под контролем).
- Существующий код (`docs/modules/payment.md` v1, модуль `payment/`) — Stars-инвойсы; v2 переводит на эквайринг своего сбора. **Перед реализацией — grep по `payment/` + сверка миграций.**

## 10. Стратегический вывод (надстоящий над платёжной схемой)

Прочитан PRD. Retention-движок реальный: **двухэтапное подтверждение + репутация + гонка за места** решают настоящую боль («15 сказали да, пришли 4»), которую Telegram-чат не решает. Архитектура умная — чат остаётся в Telegram, Mini App делает то, чего чат не умеет.

Но платёжеспособность **условна**:
- **Удержание — заложник частоты встреч**, которую платформа не контролирует. Живой клуб → 30₽ платят без раздумий; спящий → не спасёт никакая модель.
- **Эпизодическое использование** (дом юзера — TG-чат), холодный старт (репутация пуста на старте), ценность достаётся больше **организатору**, культурный барьер «платить за бывшее бесплатным» (это **непроверенная гипотеза** PRD §1.4), риск переусложнения для казуального сегмента.

**Рекомендация:**
1. **Не брать деньги с участников рано.** Сначала доказать цикл (живые клубы, работающая двухэтапка, наполняющаяся репутация) на **бесплатной** базе.
2. **Монетизацию привязывать к активности**, а не к голому членству.
3. **Серьёзно рассмотреть сбор с организатора** (сторона с явным ROI), не плоский сбор с участника.
4. 30₽ — нормальная сумма; рычаги в **timing и target**, не в цене.
5. Задача №1 продукта — **чтобы клубы регулярно встречались** (метрика «встреч на клуб», PRD §1.5). Платежи — следствие.
